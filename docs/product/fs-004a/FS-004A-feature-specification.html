FS-004A — Projects Navigation Experience

Adiciona "Projetos" à navegação principal e uma rota real de listagem — construída inteiramente por reuso de dado e componente já existentes, sem backend novo, sem entidade Project.

Release 0.3 Origem Autorização do Product Owner, entre a FS-004 e a FS-005 Data 2026-07-12 Status Aguardando aprovação — TIP-004A ainda não iniciado. Sem código escrito.
01

Contexto e Objetivo

Na FS-002, NAV_ITEMS ficou limitado a "Dashboard" porque nenhuma outra rota real existia. Com a FS-004 concluída, /workspace/[projectName] é uma experiência real — mas só é alcançável clicando em uma linha da tabela do Dashboard, nunca por navegação direta. Esta feature fecha essa lacuna com o menor incremento possível: um item de menu e uma rota, sem inventar nada que não exista.

Regra permanente registrada pelo Product Owner: um item só entra no menu principal quando tem rota real, experiência utilizável, dados/capacidade reais, estados de loading/vazio/erro, testes, e aderência à Baseline Visual. A Seção 8 (Critérios de Aceite) desta FS é, literalmente, essa regra aplicada a "Projetos".
02

Architecture Review

NAV_ITEMSweb/components/shell/navigation.ts ganha uma 2ª entrada: {"{"}label: "Projetos", href: "/projects", icon: Folder{"}"}. Sidebar já mapeia NAV_ITEMS genericamente (desktop e bottom-bar) — zero mudança em sidebar.tsx. Ícone Folder (lucide-react, já a biblioteca em uso) reaproveita a linguagem visual do item "Projetos" no Figma aprovado, sem replicar o badge numérico fictício ("12") que lá aparece — contagem fictícia, fora de escopo.
Rota novaweb/app/projects/page.tsx + web/app/projects/layout.tsx (envolve <AppShell>, mesmo padrão de dashboard/layout.tsx e workspace/[projectName]/layout.tsx — nenhum layout compartilhado novo, mesma duplicação já aceita nesses dois precedentes).
Dado — zero novoA página reaproveita usePortfolioSummary() (já existe, já testado, já usado pelo Dashboard) e o componente ProjectHealthGrid (já existe, já responsivo, já linka para /workspace/[projectName] com encodeURIComponent desde o TIP-004). Nenhuma rota de backend nova, nenhum BFF novo — a mesma GET /api/bff/dashboard → GET /api/portfolio/summary já em produção.
Gate de sessãoweb/proxy.ts — matcher estendido para incluir "/projects", "/projects/:path*", mesmo padrão já usado para adicionar /workspace no TIP-004.
Estados de telaLoading (Skeleton), vazio (Card, mesmo componente/texto-padrão do EmptyState do Dashboard) e erro (error.tsx, mesma estrutura de dashboard/error.tsx) — todos componentes já existentes, sem padrão novo.
Sobreposição com o Dashboard — decisão registradaO Dashboard já mostra a tabela de projetos como uma de suas seções. "/projects" reaproveita o mesmo componente (ProjectHealthGrid) como conteúdo único e primário da tela — não é um recurso novo, é o mesmo dado com um propósito de navegação diferente: o Dashboard é visão executiva agregada (KPIs + distribuição + tabela como uma seção entre várias); "/projects" é o diretório dedicado para encontrar e abrir um projeto. A duplicação de apresentação é intencional e mínima — nenhum dado ou lógica é duplicado, apenas a mesma tabela é montada em dois contextos de página.
Contrato de API / backendZero alteração — confirma a restrição "não alterar backend se os dados atuais forem suficientes": são.

UX Review

Responsividade reaproveita a Decisão D6 (RFC-001) já em produção — nenhum comportamento novo, apenas mais um item passando pelo mesmo mecanismo.

BreakpointComportamento (já existente, D6)
<768px (mobile)Barra fixa inferior — passa a mostrar 2 destinos (Dashboard, Projetos) em vez de 1
768–1023px (md)Rail somente-ícone — 2 ícones (LayoutDashboard, Folder)
≥1024px (lg)Sidebar completa — 2 itens com ícone+label

Página "/projects": cabeçalho com o título "Projetos" (mesmo tratamento tipográfico do cabeçalho de seção já estabelecido no TIP-003), seguido do ProjectHealthGrid — tabela em md/lg, cards em mobile, idêntico ao já usado no Dashboard. Clicar em um projeto navega para /workspace/{"{"}encodeURIComponent(project_name){"}"}, mesmo link já implementado no componente.

Decisão deliberada: nenhum outro item do menu do Figma (Chat IA, Central de Riscos, Timeline, Documentos, Aprovações, Analytics, Configurações, seção Workspaces) é adicionado nesta feature — cada um deles ainda não atende à Regra Permanente (Seção 1). Ficam para a fase "Navigation & Module Expansion".
04

Escopo

  • Item "Projetos" em NAV_ITEMS, ícone Folder, sem badge numérico
  • Rota /projects, protegida pelo mesmo gate de sessão do Dashboard/Workspace
  • Listagem via reuso de usePortfolioSummary() + ProjectHealthGrid — zero dado novo
  • Navegação de um projeto para /workspace/[projectName], com encodeURIComponent
  • Estados de loading, vazio e erro, reaproveitando os componentes já existentes
  • Responsividade nos 3 breakpoints, via o mecanismo já existente do Sidebar
05

Fora de Escopo (explícito)

  • Qualquer entidade, tabela, repositório ou modelo Project
  • Cadastro, edição ou exclusão de projetos
  • Qualquer dado novo — nenhuma coluna, contagem ou capacidade além do que ProjectSummary já tem
  • Qualquer outro item do menu do Figma (Chat IA, Central de Riscos, Timeline, Documentos, Aprovações, Analytics, Configurações, Workspaces)
  • Badge numérico no item "Projetos" (o "12" do Figma é contagem fictícia)
  • Busca, filtro ou ordenação na listagem — não autorizado nesta feature
  • Qualquer alteração de backend, contrato de API, ou nos 3 agentes de IA
06

User Stories

Como PMO executivo, acesso a lista de projetos diretamente pela navegação principal, sem precisar passar pelo Dashboard primeiro.

Critério: "Projetos" aparece na sidebar/rail/bottom-bar nos 3 breakpoints e navega para /projects.

Como usuário em "/projects", escolho um projeto e vou direto para seu Workspace.

Critério: clicar em qualquer projeto navega para /workspace/{"{"}project_name{"}"} corretamente codificado, inclusive para nomes com "/".

Como usuário sem nenhum projeto analisado ainda, vejo um estado vazio claro em vez de uma tela quebrada.

Critério: "/projects" reaproveita o mesmo EmptyState do Dashboard quando o portfólio está vazio.

09

Riscos Específicos

  • Baixo risco geral — reuso quase total de componentes e dados já testados em produção.
  • Testes E2E existentes que assumem exatamente 1 item de navegação (shell.spec.ts — "renders exactly one nav item, active, pointing at Dashboard") precisam de atualização explícita para 2 itens; risco de regressão silenciosa se esquecido.
  • Extensão do matcher de proxy.ts — mesmo padrão de risco já mitigado no TIP-004 (teste E2E de redirect não-autenticado obrigatório para /projects também).
