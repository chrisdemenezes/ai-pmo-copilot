User Journey — Meeting Intelligence

O projeto é o objeto principal, do início ao fim. O Diretor nunca "abre uma reunião" — ele está sempre dentro do Workspace de um projeto, e a reunião é só mais uma fonte de informação sobre ele, no mesmo padrão já usado por Status Executivo e Avaliação de Riscos.

01

Persona e Ponto de Entrada

Persona: Diretor de PMO, o mesmo já usado em toda a plataforma (User Workflow Discovery, FS-005/FS-006). Ponto de entrada: ele já está dentro do Workspace de um projeto — chegou até ali pelo Dashboard ou por Projetos, exatamente como já acontece hoje. Não existe uma tela "Reuniões" nem uma lista de reuniões separada do projeto: reforça a decisão do Founder de que o projeto continua sendo o objeto principal.

04

Pontos de Decisão Explícitos

  • Passo 4: o Diretor decide o que colar como contexto — a plataforma não julga a qualidade da entrada além do limite de tamanho já real (10–20.000 caracteres).
  • Passo 9: ao ver pontos de atenção reais pela primeira vez, o Diretor decide se algo exige ação imediata fora da plataforma (ex.: uma ligação, um e-mail) — a plataforma não tenta prever isso.
  • Passo 16: o único ponto de decisão que a plataforma ativamente propõe — aceitar o próximo passo sugerido (rodar Status ou Risco) ou não. Nunca uma ação silenciosa; sempre um clique explícito do Diretor.
05

Fora de Escopo (confirmado)

  • Nenhuma tela ou rota nova dedicada a "reuniões" — o projeto continua sendo o objeto principal (restrição do Founder, Rev. 2).
  • Nenhuma reclassificação de issues em riscos estruturados (probabilidade/impacto) sem a Avaliação de Riscos rodar de fato (Discovery §05).
  • Nenhuma sugestão de próximo passo para Feature que não existe ainda (Plano de Ação, Steering Committee) — Discovery §00.
  • Nenhum nome técnico (transcript, summary, issues) como rótulo de interface.
  • Nenhuma comparação entre reuniões, nenhum histórico agregado "por pessoa" — dados que o schema atual não sustenta (Discovery §03).
