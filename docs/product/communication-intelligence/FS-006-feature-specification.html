01

Contexto e Objetivo

Esta FS formaliza 3 documentos já aprovados pelo Founder: User Decision Discovery (achado central: summary e issues são reais no backend desde sempre, nunca exibidos), User Journey (16 passos, projeto como objeto principal, hierarquia executiva de 7 níveis) e UX Flow (linguagem orientada a objetivo — o usuário nunca escolhe um agente, escolhe o que quer entender sobre o projeto). Nenhuma decisão arquitetural nova é tomada aqui além de formalizar essas três.

Princípio final do Founder, normativo a partir desta FS: Pergunta → Capability → Executor, nunca o contrário. O catálogo interno passa a expressar essa hierarquia explicitamente (Seção 2).
02

Architecture Review

2.1 — Catálogo: Pergunta → Capability → Executor

Pergunta (goalLabel, UI)→ Capability (capability, interno)→ Executor (kind → hook → BFF → agente) 
export type Capability = "project-intelligence" | "risk-intelligence" | "communication-intelligence";

export interface AnalysisCatalogEntry {
  kind: AnalysisListItem["kind"];   // Executor -- nunca exibido
  capability: Capability;            // arquitetura interna -- nunca exibido
  goalLabel: string;                 // a pergunta -- único texto que o usuário vê
  description: string;
}

export const ANALYSIS_CATALOG: readonly AnalysisCatalogEntry[] = [
  { kind: "status",  capability: "project-intelligence",       goalLabel: "Como está o projeto?",              description: "..." },
  { kind: "risk",    capability: "risk-intelligence",          goalLabel: "Quais riscos exigem atenção?",       description: "..." },
  { kind: "meeting", capability: "communication-intelligence", goalLabel: "O que mudou na última reunião?",     description: "..." },
];

capability é metadado puro — nenhum componente lê esse campo para decidir comportamento visível nesta Feature; ele existe para documentar a hierarquia e preparar a fusão futura das 3 Capabilities (UX Flow §04), sem forçar nenhuma decisão de UI ainda não aprovada. Os 2 rótulos já reais (status, risk) são renomeados para pergunta nesta mesma Feature — UX Flow §02 já formalizou isso como escopo desta FS, não uma mudança separada.

2.2 — Dialog

AnalyzeProjectDialog ganha a 3ª TabsTrigger automaticamente (já itera sobre ANALYSIS_CATALOG). A DialogDescription do caso length === 1 fica morta (catálogo nunca mais tem 1 entrada) — removida nesta Feature por limpeza, não por necessidade funcional. Nova descrição fixa: "O que você quer entender sobre este projeto?" (UX Flow §02), substituindo "Escolha o tipo de análise".

A Textarea ganha rótulo/placeholder condicionais por kind ativo — achado da Discovery §Reuso: o backend chama o campo de transcript para reunião, não project_context. Rótulo passa a ser "Contexto" para Status/Risco (inalterado) e "Cole a ata, notas ou transcrição da reunião" para Comunicação — nunca a palavra transcript na interface.

2.3 — BFF, Hook

Mesmo padrão de sempre: web/app/api/bff/workspace/[projectName]/analyze/meeting/route.ts (proxy fino para POST /api/meetings/analyze, corpo {"{"}transcript, project_name{"}"} — único BFF que usa transcript como chave, não project_context), 60s, mesmo formato de erro. Novo hook useSubmitMeetingIntelligence, mesmo template useMutation.

2.4 — Comunicação (nova seção, substitui Ações + Decisões)

Seções 5 (Ações) e 6 (Decisões) fundem-se em uma seção "Comunicação" — mesmo princípio do TIP-005A (Status: 2 seções → 1 Brief) e do TIP-006 (Risco: painel técnico → Brief). web/lib/workspace/meeting-momentum.ts espelha decision-momentum.ts/risk-momentum.ts: nenhum campo de severidade — meeting_intelligence não tem equivalente a health_status, então "Impacto da reunião" (Nível 1 da hierarquia) é uma contagem real, nunca uma nota inventada.

export function impactHeadline(decisionsCount: number, issuesCount: number, actionItemsCount: number): string {
  return `${decisionsCount} decisão(ões) · ${issuesCount} ponto(s) de atenção · ${actionItemsCount} responsabilidade(s)`;
}

export function suggestedNextStep(
  issuesCount: number,
  decisionsCount: number,
): { label: string } | null {
  if (issuesCount > 0) return { label: "Executar Avaliação de Riscos" };
  if (decisionsCount > 0) return { label: "Atualizar Status Executivo" };
  return null; // honesto -- nenhuma sugestão fabricada
}

export const NEXT_STEP_FALLBACK_MEETING =
  "Nenhum próximo passo adicional sugerido a partir desta reunião.";
"Próximo passo" permanece texto, não botão. Confirmado contra o código real: nos Briefs de Status e Risco já implementados (TIP-005A/TIP-006), "Próximo passo" é sempre <p>, nunca um CTA clicável que abre outra aba do mesmo Dialog. Para Meeting seguir exatamente o padrão existente (Founder: "reutilizar integralmente"), a sugestão aqui também é texto — abrir o Dialog na aba sugerida exigiria elevar o estado do Dialog para fora do componente (arquitetura nova, não autorizada nesta Feature). Registrado como possível evolução futura, não construído agora.

2.5 — Robustez de schema

Mesma disciplina do achado real do TIP-006: hasMeetingShape(modelOutput) adicionado a web/lib/workspace/types.ts, mesmo formato de hasStatusShape/hasRiskShape — verifica summary string, e os 4 arrays (decisions, action_items, issues, dependencies). Aplicado em toda leitura de model_output desta Feature, incluindo o drill-down de Histórico.

07

User Stories

Como Diretor de PMO, escolho "O que mudou na última reunião?" sem nunca ver o nome de um agente de IA.

Critério: os 3 rótulos das abas são perguntas; nenhum nome técnico aparece em nenhum ponto da interface.

Como Diretor de PMO, entendo o impacto de uma reunião de 90 minutos em menos de 2 minutos de leitura.

Critério: os 7 níveis da hierarquia executiva aparecem em um único bloco "Comunicação", sem rolar entre seções distantes.

Como Diretor de PMO, quando não há pontos de atenção nem decisões, não recebo uma sugestão de próximo passo fabricada.

Critério: suggestedNextStep() retorna null nesse caso, e a interface mostra o fallback honesto.
