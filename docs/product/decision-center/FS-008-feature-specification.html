01

Contexto e Objetivo

Esta FS formaliza os 3 documentos já aprovados (Discovery, Journey, UX Flow) e as diretrizes que os acompanharam. A Executive Decision Queue organiza, por prioridade executiva, as decisões que já existem hoje dispersas em 2 sinais reais (Status e Risco) -- nunca lista tudo com o mesmo peso, nunca decide ou age no lugar do executivo. Fecha o mesmo Executive Loop já formalizado por Action Intelligence, mas a partir de um ponto diferente do ciclo:

Pergunta→ Compreensão→ Decisão→ Ação→ Retorno

Diferente de Action Intelligence (que organiza o elo "Agir"), Decision Center organiza o elo "Decidir" -- o momento exato em que o executivo, de posse do que a plataforma já compreendeu, decide o que fazer. Ver Seção 2 para a formalização deste comportamento.

02

Comportamento Executivo

Nova seção obrigatória em toda Feature Specification, por instrução do Founder: qual comportamento executivo esta Capability pretende incentivar?

CapabilityComportamento incentivado
Project Intelligence (Status)Acompanhar
Risk IntelligenceMitigar
Communication IntelligenceAlinhar
Action IntelligenceExecutar
Decision CenterDecidir

Coerência com o Executive Questions Framework: "Decidir" responde diretamente a pergunta 2 do framework ("qual decisão ela permite tomar?") -- mas, diferente das Capabilities anteriores, essa é a própria razão de existir do módulo, não um efeito colateral. A pergunta 3 ("qual ação ela desencadeia?") permanece consistente com o princípio já estabelecido: nenhuma ação acontece dentro do próprio Decision Center -- ela acontece fora (uma ligação, uma escalação) ou em outra Capability já existente (rodar Status/Risco de novo), nunca dentro da fila.

Coerência com o Executive Decision Performance Framework: incentivar "decidir" é o próprio alvo do Time To Decision (TTD, princípio oficial) -- Decision Center existe para encurtar o tempo entre "a plataforma compreendeu algo" e "o executivo está diante da decisão". O clique de retorno ao Workspace, em cada Executive Decision Card, é o mecanismo desenhado para favorecer o Return Rate (RR, conceitual) -- o mesmo tratamento sem instrumentação já aplicado em Action Intelligence (TIP-008 §02/§08).

03

Architecture Review

3.1 -- Nova leitura: ProjectSummaryService.list_latest_risks()

Escopo completo aprovado pelo Founder na User Journey: a leitura agregada de riscos em zona de atenção entra desde este primeiro incremento, como investimento arquitetural reutilizável. Mesma classe, mesmo padrão de list_action_items() -- não um serviço paralelo. Diferença importante: mantém só o risco mais recente por projeto (mesmo princípio de latest_health_status em _aggregate), não o histórico inteiro -- a zona de atenção é sempre sobre a análise mais atual, igual ao que o Brief de Riscos já mostra hoje no Workspace.

class ProjectSummaryService:
    def list_latest_risks(self, project_name: str | None = None) -> list[dict]:
        # Mesma chamada já usada por list_action_items() -- zero query nova.
        records = self._repository.list_analyses(
            project_name=project_name, kind="risk", limit=None,
        )
        seen_projects: set[str | None] = set()
        items = []
        for record in records:  # já newest-first
            if record.project_name in seen_projects:
                continue
            model_output = (record.payload or {}).get("model_output")
            if not isinstance(model_output, dict) or not model_output.get("structured"):
                continue  # não marca como visto -- uma análise mais antiga
                          # e estruturada ainda pode contar como "mais recente"
            seen_projects.add(record.project_name)
            escalation_recommendation = model_output.get("escalation_recommendation")
            for risk in model_output.get("risks") or []:
                items.append({
                    "project_name": record.project_name,
                    "description": risk.get("description"),
                    "probability": risk.get("probability"),
                    "impact": risk.get("impact"),
                    "mitigation": risk.get("mitigation"),
                    "escalation_recommendation": escalation_recommendation,
                    "source_analysis_id": record.id,
                    "source_created_at": record.created_at,
                })
        return items

Mesma disciplina de robustez de schema já estabelecida em list_action_items() -- structured e formato verificados antes de ler o array; uma resposta mal-formada de um projeto específico não derruba o rollup, só é excluída dele.

04

Princípio de Atenção — novo princípio permanente

Registrado oficialmente nesta etapa, por instrução do Founder: "A ausência de informação também comunica. A plataforma não deve preencher a interface -- ela deve mostrar apenas aquilo que realmente merece a atenção do executivo." Princípio permanente da STRATECH, mesma categoria de Incrementos de Valor, Executive Loop e Executive Decision Performance (TIP-008 §08) -- orienta toda decisão de UX futura, não só desta Capability.

Decision Center é a primeira Capability desenhada inteiramente sob este princípio desde a concepção: um projeto sem decisão pendente não gera nenhum Executive Decision Card (Journey §2), e o estado vazio é desenhado para comunicar tranquilidade, não ausência técnica de dado (UX Flow §5).

O princípio também valida, retroativamente, decisões de UX já tomadas antes de existir como registro formal: "nunca listar tudo com o mesmo peso" (Action Intelligence, FS-007 §2.7), a linha de contexto que só aparece quando a contagem é maior que zero (FS-007 §2.7), e os estados vazios já honestos de todo o Workspace ("Nenhuma análise registrada ainda"). Nenhuma dessas implementações muda -- o princípio as reconhece como já corretas, não exige retrabalho.

User Stories

Como executivo, abro "Decisões" e vejo, em segundos, quais projetos exigem uma decisão minha agora, sem abrir cada Workspace.

Critério: a fila já chega ordenada (Hoje antes de Esta semana), filtrada (projetos no curso esperado não aparecem), sem nenhuma configuração prévia.

Como executivo, cada decisão me diz claramente qual projeto, por que ela depende de mim, o que acontece se eu não agir, e qual o próximo passo.

Critério: as 5 perguntas do Executive Decision Card (§3.5) estão sempre presentes, nunca uma delas ausente ou vaga.

Como executivo, quando não há nenhuma decisão pendente, sinto confiança de que o portfólio está sob controle, não a sensação de uma tela quebrada ou vazia.

Critério: o estado vazio (§4, §6) é afirmativo e só aparece quando é literalmente verdadeiro.

Critérios de Aceite

  • A Executive Decision Queue mostra só projetos com Status crítico/em atenção e/ou risco em zona de atenção -- nenhum projeto "no curso esperado" aparece
  • Ordenação fixa: "Hoje" antes de "Esta semana", nunca reordenável pelo usuário
  • Cada Executive Decision Card responde as 5 perguntas da Seção 3.5, sempre
  • Estado vazio exibe a mensagem afirmativa definida na Seção 4/6, nunca um "vazio" genérico
  • Zero escrita -- nenhum botão de criar/editar/resolver em nenhuma tela desta Feature
  • Clicar num card leva ao Workspace do projeto correspondente
  • Testes: serviço (list_latest_risks), rota, BFF, hook, decision-queue.ts, componente, E2E, backend como regressão
  • Verificação real contra Demo Mode, mesma disciplina das Features anteriores
