2. Evidências

RB-001 (PRI-006)

.github/workflows/ci.yml (job frontend) agora executa npx playwright install --with-deps chromium seguido de npx playwright test --project=lg em todo PR/push. web/playwright.config.ts foi corrigido: o executablePath do Chromium estava fixado a /opt/pw-browsers/chromium-1194/chrome-linux/chrome, um caminho exclusivo do sandbox de desenvolvimento desta sessão -- inexistente em qualquer runner de CI real. Passou a ser condicional via fs.existsSync, com fallback ao Chromium que o próprio CI instala.

Achado durante a validação, corrigido antes do fechamento: a suíte completa (3 projetos -- mobile, md, lg -- 204 execuções) roda contra um único mock-backend compartilhado (e2e/mock-backend.mjs) com estado em memória. Executar os 3 projetos em sequência contra essa mesma instância causa poluição de estado entre projetos (a mesma análise sendo submetida 3x contra o mesmo backend em memória) -- 5 falhas observadas nesta condição, nenhuma delas ao rodar um único projeto isoladamente. Este comportamento nunca havia sido exercido antes: a verificação manual desta suíte, ao longo de toda a V1, sempre usou exclusivamente --project=lg (convenção já registrada em todas as sessões anteriores desta reta final). Para manter o gate de CI confiável -- objetivo central do RB-001 -- o pipeline foi configurado para rodar --project=lg, exatamente a mesma convenção já usada manualmente durante toda a V1, e não os 3 projetos simultaneamente.

RB-002 (PRI-008)

docs/operations/PRI-008-production-backup-restore-runbook.md -- estratégia (pg_dump -Fc contra o serviço database do docker-compose.yml), periodicidade (diário/semanal + pré-deploy), restauração (pg_restore + alembic upgrade head), validação (health check, contagem, registro mais recente) e recuperação após falha (4 cenários). Limitação registrada com transparência: armazenamento externo dos backups e a automação de disparo dependem de uma escolha de provedor de infraestrutura ainda não feita neste repositório.

RB-003 (PRI-009)

docs/operations/PRI-009-production-deployment-runbook.md -- pré-requisitos, implantação, rollback, validação pós-deploy, smoke tests e critérios de sucesso, todos fundamentados no docker-compose.yml/Dockerfile reais deste repositório. Duas lacunas pré-existentes foram documentadas com transparência, não resolvidas por este runbook (fora de escopo desta etapa -- nenhuma alteração arquitetural): (1) a hospedagem de produção do frontend nunca foi decidida (pergunta em aberto desde RFC-001-frontend-architecture.html); (2) POST /api/bff/session não tem mitigação de força bruta -- risco já formalmente aceito apenas para uso interno/piloto (docs/development/01-project-structure.md), com condição explícita ainda não cumprida para qualquer deploy além desse escopo. O runbook restringe seu próprio escopo explicitamente a esse cenário já aprovado (uso interno/piloto).
