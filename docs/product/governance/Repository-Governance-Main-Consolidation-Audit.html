7. Estratégia de consolidação e governança futura

  • Merge: merge commit (preferência do Founder, confirmada tecnicamente): preserva os 55 commits e o histórico decisório; revert atômico via git revert -m 1. Squash destruiria a rastreabilidade; rebase merge reescreveria SHAs citados nos próprios documentos de governança.
  • Sem squash do histórico existente — cada commit é uma etapa formal do processo STRATECH; preservar vale mais que estética.
  • Branch protection na main (após autorização): PR obrigatório (inclusive agentes), checks validate + frontend obrigatórios, sem force push, sem exclusão, branch atualizada e conversas resolvidas antes do merge.
  • Política de branches: main protegida; feature/, fix/, chore/, docs/, release/, hotfix/; branches de agente seguem o mesmo processo de PR; vida curta; exclusão após merge; fim do padrão de branch única longa.
  • Commits novos: Conventional Commits com escopo (feat(actions):, docs(rc1):…). Os 55 existentes não serão reescritos.
  • Versionamento: v1.0.0-rc.1 → v1.0.0 (pós-pilot) → V2 só ganha v2.0.0-alpha.1 quando houver implementação real; documentação nunca recebe tag de software.
  • CI mínimo confiável: pipeline atual + passo de migrations no job backend; gitleaks/npm-audit como melhorias futuras não bloqueantes. Nenhuma ampliação indiscriminada.
  • Documentação: nenhuma movimentação em massa na consolidação; migração incremental por PRs pequenos com verificação de links. CODEOWNERS: criar (1 linha, Founder) somente junto com a branch protection — antes disso não agrega valor (conclusão registrada).
  • Rollback: pré-merge = fechar o PR (main intocada); pós-merge = git revert -m 1 sem reescrita; tag criada só após validação pós-merge; branch excluída só após merge + validação + tag + autorização explícita.
