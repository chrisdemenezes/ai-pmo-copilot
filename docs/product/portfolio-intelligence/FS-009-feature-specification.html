01

Contexto e Objetivo

Esta FS formaliza os 3 documentos já aprovados (Discovery, Journey, UX Flow) e o ajuste conceitual do Founder na aprovação da User Journey (camada de risco nunca infere consequência) mais a diretriz desta etapa (Progressive Context). A Executive Portfolio View organiza o portfólio inteiro em 4 camadas transparentes de atenção -- nunca lista tudo com o mesmo peso, nunca recalcula uma decisão, nunca decide ou age no lugar do executivo.

Pergunta→ Compreensão→ Decisão→ Ação→ Retorno

Diferente de Decision Center (que organiza o elo "Decidir" para quem já sabe qual projeto olhar), Portfolio Intelligence organiza a etapa anterior do mesmo Loop -- "Compreensão": onde o executivo deve concentrar sua atenção antes mesmo de chegar a uma decisão específica. Ver Seção 4 para como isso se encaixa na hierarquia de Progressive Context.

02

Comportamento Executivo

Seção obrigatória em toda Feature Specification, por instrução do Founder: qual comportamento executivo esta Capability pretende incentivar?

CapabilityComportamento incentivado
Project Intelligence (Status)Acompanhar
Risk IntelligenceMitigar
Communication IntelligenceAlinhar
Action IntelligenceExecutar
Decision CenterDecidir
Portfolio IntelligencePriorizar

Fixado nesta FS, resolvendo o fork registrado na Discovery §00 (achado de nomenclatura): "Priorizar" é o próprio termo que o Founder usou no diagrama de Progressive Context ("Portfolio Intelligence ↓ Priorização") -- nenhuma invenção nova, apenas a formalização do vocabulário já dado. Não colide com "Decidir" (Decision Center, que já possui a decisão pronta) nem com "Acompanhar" (Project Intelligence, que lê o estado de um projeto por vez).

Coerência com o Executive Questions Framework: "Priorizar" responde à pergunta 1 do framework ("por que essa informação importa agora?") de forma mais ampla que qualquer Capability anterior -- estabelece a ordem de importância entre projetos, não apenas dentro de um projeto só.

Coerência com o Executive Decision Performance Framework: ao encurtar o caminho até "onde olhar", Portfolio Intelligence reduz o tempo antes mesmo do relógio de TTD começar a contar para uma decisão específica -- efeito qualitativo, sem instrumentação nesta fase, mesma disciplina de TIP-008/TIP-009.

03

Architecture Review (preliminar)

3.1 -- Nenhuma leitura nova, nenhuma rota nova

Escopo enxuto aprovado pelo Founder (Discovery §04, Opção 1): a Executive Portfolio View é inteiramente uma composição de funções puras no cliente sobre dado já em cache -- usePortfolioSummary() (já existente) e useLatestRisks() (já existente, Decision Center) alimentam tanto buildExecutiveDecisionQueue() quanto o novo agrupador desta Feature. Nenhum método novo em ProjectSummaryService, nenhuma rota nova em src/api/routes/intelligence.py, nenhum BFF novo.

3.2 -- portfolio-view.ts (novo, mesma família de decision-queue.ts)

Combina 2 sinais já reais em 4 camadas -- nunca recalcula o que buildExecutiveDecisionQueue()/rankByRisk() já decidem:

export type PortfolioLayer =
  | "decision_today"
  | "decision_this_week"
  | "risk_to_monitor"
  | "no_signal";

export interface PortfolioIntelligenceItem {
  project_name: string;
  layer: PortfolioLayer;
  whyAttention: string;   // rótulo institucional fixo da camada -- Seção 3.5
  realSignal: string;     // dado real verbatim -- ExecutiveDecision.context (camadas 1-2),
                            // contagem de open_risks (camada 3), ou ausência real (camada 4)
  nextMove: { label: string; href: string } | null; // navegação real -- null só na camada 4
}

export function buildExecutivePortfolioView(
  portfolio: ProjectSummary[],
  decisions: ExecutiveDecision[], // saída de buildExecutiveDecisionQueue() -- nunca recalculada
): PortfolioIntelligenceItem[] {
  // 1. Projetos com decisao(oes) janela "hoje" -> camada decision_today
  // 2. Projetos com decisao(oes) janela "esta_semana" (sem entrada em 1) -> decision_this_week
  // 3. Projetos restantes com open_risks > 0, ordenados por rankByRisk() -> risk_to_monitor
  // 4. Projetos restantes -> no_signal
  // Ordenacao: camada (1-4), depois project_name -- fixa, nunca inferida pela IA.
}

Um projeto com múltiplas entradas na Executive Decision Queue (ex.: Status crítico e risco em atenção) ocupa uma única linha na Executive Portfolio View, na camada mais alta que se aplica -- diferente da própria Executive Decision Queue (onde as 2 entradas continuam distintas). É uma visão de priorização por projeto, não uma lista de decisões -- distinção que preserva Single Decision Source (a contagem/o detalhe de cada decisão continua vivendo só na Executive Decision Queue) e evita um novo tipo de fusão de dado.

3.3 -- Single Decision Source: fronteira explícita

buildExecutivePortfolioView() recebe a saída já computada de buildExecutiveDecisionQueue() como parâmetro -- nunca chama ProjectSummaryService/useLatestRisks para recalcular por conta própria o que significa "decisão pendente". Se, no futuro, a Executive Decision Queue mudar sua regra de janela, a Executive Portfolio View herda a mudança automaticamente, sem nenhum código próprio a atualizar.

04

Progressive Context — novo princípio permanente

Registrado oficialmente nesta etapa, por instrução do Founder: "A STRATECH não navega entre telas. Ela aprofunda contexto. Cada Capability deve entregar um nível maior de compreensão do mesmo objeto." Princípio permanente da STRATECH, mesma categoria de Single Decision Source, Princípio da Atenção e Executive Trust -- orienta toda decisão de arquitetura de informação futura, não só desta Capability.
DashboardResumo Executivo -- números agregados do portfólio inteiro
↓
Portfolio IntelligencePriorização -- quais projetos, em que ordem, por quê
↓
WorkspaceContexto do Projeto -- todos os sinais de um único projeto
↓
Executive BriefAnálise -- o texto real da IA por trás do sinal

4.1 -- O Executive Portfolio Card respeita Progressive Context?

Avaliação explícita, campo a campo (Seção 3.5):

  • "Por que este projeto merece minha atenção?" -- é a camada, um rótulo de priorização que não existe em nenhuma tela hoje. Nível novo de compreensão, não uma repetição.
  • "Quais sinais reais justificam essa posição?" -- reaproveita, verbatim, o mesmo texto que já justifica uma entrada na Executive Decision Queue (camadas 1-2) ou a mesma contagem de open_risks (camada 3). Aqui está o ponto de atenção real -- ver 4.2.
  • "Qual o próximo movimento recomendado?" -- é navegação, nunca dado -- nível novo por natureza (o Dashboard de hoje não linka projeto a projeto, nenhuma tela anterior faz isso).

4.2 -- Achado honesto: sobreposição real com widgets já existentes do Dashboard

Para a Architecture Review resolver explicitamente, não decidido nesta FS: o Dashboard já possui hoje dois widgets que parcialmente antecipam a função da Executive Portfolio View -- ProjectHealthGrid (tabela de todos os projetos, sem ordenação, com open_risks já visível por linha) e RiskConcentrationRanking (top 5, já ordenado por open_risks, mesmo cálculo que alimentaria a camada 3). Se a Executive Portfolio View entrar em produção mostrando o mesmo número (open_risks) já visível nesses 2 widgets, isso é, por definição, repetição de informação -- exatamente o que Progressive Context proíbe.

Duas saídas possíveis, nenhuma decidida aqui:
  1. Recomendada: a Architecture Review avalia retirar ou reduzir RiskConcentrationRanking do Dashboard (sua função de "ranking ordenado" é herdada e superada pela camada 3 da Executive Portfolio View) e simplificar ProjectHealthGrid para um inventário bruto sem pretensão de priorização -- Dashboard volta a ser puramente "Resumo Executivo" (números agregados, Seção acima), Portfolio Intelligence assume integralmente a "Priorização".
  2. Alternativa: manter os 2 widgets como estão, diferenciando por escopo (Dashboard: top 5, sem link de navegação, sem camada 1-2; Portfolio Intelligence: portfólio inteiro, 4 camadas, navegável) -- aceita menor coerência com Progressive Context em troca de zero mudança em telas já aprovadas.
Esta FS não decide entre as duas -- ambas preservam Single Decision Source e não exigem código nesta etapa. A escolha pertence à Architecture Review (próxima etapa), por envolver uma tela já implementada e aprovada em TIP anterior (TIP-004/TIP-008), não uma decisão de escopo desta Feature.

4.3 -- Nenhum contexto perdido durante a navegação

Todo "próximo movimento recomendado" (Seção 3.5) leva a uma superfície que já contém, de forma mais profunda, exatamente o mesmo objeto que motivou o clique -- nunca um destino genérico. Camadas 1-2 levam à Executive Decision Queue (mesmo projeto, mesma decisão, agora com as 5 perguntas completas do Executive Decision Card); camada 3 leva ao painel de Riscos do Workspace daquele projeto (mesmo projeto, agora com a lista completa de riscos, não só a contagem). Nenhum clique exige que o executivo "comece de novo" -- ele sempre chega mais fundo no mesmo objeto que já estava olhando.

4.4 -- Cada clique aprofunda o entendimento

Confirmado pela própria hierarquia de 4 níveis: Dashboard (números) → Portfolio Intelligence (quais projetos, em que ordem, por quê) → Workspace (todos os sinais daquele projeto) → Executive Brief (o texto real da análise). Em nenhum ponto um nível repete o nível anterior por completo -- cada um acrescenta uma camada de compreensão que o nível anterior não tinha.

09

User Stories

Como executivo, abro a Executive Portfolio View e vejo, em segundos, onde devo concentrar meu tempo agora, entre todos os projetos do portfólio.

Critério: as 4 camadas já chegam ordenadas e justificadas, sem nenhuma configuração prévia.

Como executivo, cada projeto me diz claramente por que merece minha atenção, quais sinais reais o justificam, e qual o próximo movimento dentro da plataforma.

Critério: as 3 perguntas do Executive Portfolio Card (§3.5) estão sempre presentes, nunca uma consequência inferida ou dado fabricado.

Como executivo, ao clicar num projeto, chego mais fundo no mesmo contexto que já estava olhando -- nunca perco o fio nem preciso recomeçar.

Critério: navegação leva à Executive Decision Queue ou ao Workspace do mesmo projeto, nunca a um destino genérico (§4.3).
