1. O que pertence ao STRATECH Core

Critério de corte: pertence ao Core tudo que reutiliza a arquitetura já existente (agentes, BFF, Design System, Executive Brief) e não depende de nenhuma decisão de backend ainda não tomada. O que exige uma entidade de domínio nova, uma API de armazenamento nova, ou um RFC de backend específico fica fora do Core por definição — não porque seja menos importante, mas porque abri-lo agora contrariaria "reutilizar arquitetura existente" e multiplicaria o tempo até a conclusão.

4. Caminho crítico

Nada do que falta para o STRATECH Core bloqueia tecnicamente outra coisa de forma rígida -- a maior parte pode correr em paralelo (Seção 5). O único encadeamento real é Executive Brief Contract → Decision Catalog/Domain/Context, porque os dois tocam exatamente a mesma superfície de código (ANALYSIS_CATALOG e os 3 componentes de Brief) -- fazer os dois ao mesmo tempo, ou fora de ordem, arrisca retrabalho. Esse par de itens é o caminho crítico do Core: é o único trabalho de Capability ainda pendente, todo o resto é hardening de produção que não toca nessa mesma superfície.

7. Oportunidades de desenvolvimento paralelo

  • Toda a Trilha B (E2E no CI, versionamento de API, model_version, backup/restore) não compartilha superfície de código com Briefs/Dialog/Catálogo -- pode rodar 100% em paralelo com a Trilha A sem risco de conflito.
  • Dentro da Trilha B, os 4 itens são independentes entre si -- podem ser sequenciados por conveniência, não por dependência técnica.
  • As 5 sessões reais de validação (Frente B) continuam em paralelo às duas trilhas -- nenhuma delas interrompe a Frente A, por decisão já registrada do Founder.

8. Decisões que aguardam o Founder (não decididas aqui)

Escopo de autenticação multi-usuário. É o maior item de esforço/risco de todo este Board e o único que genuinamente expande o Domain Model. Duas leituras válidas: (a) fica fora do STRATECH Core -- o produto continua demonstrável e pilotável com senha única de workspace, como já validado no RC-1; ou (b) entra no Core porque "pronto para produção" (critério do próprio Founder para Capability Completion) pressupõe login individual antes de qualquer cliente pago real. Recomendo (a) para não atrasar o caminho crítico real (Trilha A) por um item que não bloqueia nenhuma Capability existente -- mas é uma recomendação, não uma decisão tomada.
Uma 4ª Capability de IA. Este Board não propõe nenhuma nova Capability de IA -- todo o trabalho identificado é hardening e consistência do que já existe. Se o Founder quiser abrir uma 4ª Capability agora, ela segue o processo de 8 etapas já estabelecido (Discovery em primeiro lugar), em paralelo ou depois da Trilha A -- decisão do Founder, não uma lacuna deste documento.
