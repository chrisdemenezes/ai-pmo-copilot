FS-003 — Enterprise Dashboard Experience

Eleva o Dashboard Executivo ao nível visual do Figma aprovado, usando exclusivamente dados e funcionalidades já existentes. Substitui a prioridade originalmente prevista no Release Plan 0.3.

Release 0.3 Baseline Visual Figma aprovado + FS-002 Rev.3 + TIP-002 + Visual Fidelity Report Data 2026-07-11 Status Aguardando aprovação — TIP-003 ainda não iniciado
01

Contexto e Objetivo

A Sprint de Fidelidade Visual já fechou os itens crítica do Dashboard — eyebrow+heading, raio/sombra dos cards, tabela densa, badge em pill (Visual Fidelity Report, commit 321c745). O que resta não foi tocado naquela Sprint porque estava fora do seu recorte (pure-visual, sem reorganização estrutural): cabeçalhos de seção, estados de carregamento/erro/vazio, e a hierarquia relativa entre os widgets que já existem. A FS-003 fecha esse restante — não reabre o que já foi decidido, e não introduz nenhum widget, dado ou rota novos.

A Baseline Visual da Release 0.3 (Figma aprovado + FS-002 Rev.3 + TIP-002 + Visual Fidelity Report) é a referência obrigatória desta feature, por decisão do Product Owner — nenhum novo documento de tokens é produzido aqui.
02

Architecture Review

Componentes tocadosapp/dashboard/page.tsx (ordem das seções, cabeçalhos, skeleton), components/dashboard/portfolio-summary-strip.tsx, components/dashboard/health-status-distribution.tsx, components/dashboard/risk-concentration-ranking.tsx, app/dashboard/error.tsx. Nenhum arquivo novo — apenas refinamento dos 5 já existentes.
DadosZero mudança de contrato. Todos os componentes já recebem ProjectSummary[] via usePortfolioSummary() (TanStack Query, BFF /api/bff/dashboard) — nenhum campo novo é lido, nenhuma chamada de rede nova é introduzida.
Skeleton desatualizado — achado realO DashboardSkeleton em page.tsx ainda renderiza um grid de 3 blocos h-32 em formato de card para a seção "Projetos" — isso representava o layout pré-Visual-Fidelity-Sprint. Hoje "Projetos" é uma tabela densa em md/lg e uma lista de cards só em mobile (project-health-grid.tsx). O skeleton ficou inconsistente com o layout real que ele deveria antecipar. Corrigir isso é implementável nesta Feature — não é mudança de dado, é o esqueleto de carregamento acompanhar a UI real.
Ordem das seçõesReordenar <section> dentro de page.tsx é troca de posição de JSX já existente — sem novo componente, sem nova query. Detalhado na Seção 3 (UX Review).
Section headersHoje "Distribuição de saúde", "Maior concentração de risco" e "Projetos" usam o mesmo <h2 className="text-sm font-medium text-ink-muted"> sem hierarquia entre si. Elevar o peso tipográfico e adicionar um subtítulo com contagem real (projects.length, já calculado) é troca de className + interpolação de um número já disponível — não é dado novo.
Contrato de API / backendNenhuma alteração. Nenhuma rota nova, nenhum agente de IA tocado, Demo Mode inalterado.
03

UX Review

O Figma dá ênfase clara à tabela de projetos como o conteúdo dominante da tela (maior área, primeiro no fluxo visual depois dos KPIs). A implementação real hoje intercala dois widgets menores (Distribuição de saúde, Maior concentração de risco) antes da tabela, invertendo essa ênfase.

MudançaRacional
Reordenar: KPIs → Projetos (tabela) → Distribuição de saúde → Maior concentração de riscoAlinha a ênfase da IA real à do Figma (conteúdo tabular como âncora principal), sem remover nenhum widget existente
Cabeçalho de seção "Projetos" ganha peso tipográfico maior + subtítulo "N projetos"Mesmo tratamento do Figma para o cabeçalho da tabela — usa projects.length, dado já real
Skeleton de carregamento passa a refletir o layout atual (tabela em md/lg, cards em mobile) em vez do grid de cards antigoConsistência entre estado de loading e estado carregado — o usuário não deve ver uma forma diferente da tela real "piscando" durante o carregamento
Estado de erro (error.tsx) e estado vazio (EmptyState em page.tsx) recebem o mesmo raio/sombra de card já estabelecido pela Visual Fidelity SprintHoje usam border-dashed genérico sem o token --radius atualizado — o Figma não define esses estados (é um mockup estático), então a extrapolação aqui é aplicar o mesmo sistema de tokens já real ao produto, não copiar algo que não existe na referência
Limite explícito: o Figma não modela estados de vazio/erro/carregamento — é uma tela estática com dados preenchidos. Qualquer refinamento nesses estados nesta Feature usa o sistema de tokens já estabelecido (raio, sombra, tipografia), nunca invenção de novo conteúdo visual não ancorado no que já existe.
04

Escopo

  • KPIs: nenhuma mudança visual adicional (já fechado na Visual Fidelity Sprint) — apenas manter no reposicionamento da Seção 3
  • Cards: nenhuma mudança adicional (raio/sombra já fechados) — aplicar o mesmo tratamento aos cards de estado vazio/erro
  • Grid: manter a estrutura responsiva já existente (sm:grid-cols-3 para KPIs) — sem novas colunas
  • Table: nenhuma mudança estrutural (já fechada) — apenas garantir que o skeleton de loading a antecipe corretamente
  • Section Headers: peso tipográfico + subtítulo com contagem real para "Projetos"; avaliação do mesmo tratamento para os demais cabeçalhos de seção
  • Estados visuais: harmonizar loading/erro/vazio com os tokens de raio/sombra já estabelecidos
  • Hierarquia da informação: reordenar as seções da página para dar à tabela de projetos a mesma ênfase que tem no Figma
  • Refinamento dos componentes existentes: portfolio-summary-strip.tsx, health-status-distribution.tsx, risk-concentration-ranking.tsx, error.tsx — apenas estilo/organização, mesma lógica e dados
05

Fora de Escopo (explícito)

  • Qualquer novo widget, KPI, coluna de tabela ou dado — inclusive os que existem apenas no Figma (Budget Utilizado, Sprint Velocity, Capacidade Time, colunas Progresso/Cronograma/PM)
  • Sparklines e indicadores de tendência nos KPIs — exigem série histórica que o produto não persiste
  • Painel "Risk Radar" — exige nova rota de API (Release Plan 0.3, Épico 2), não é mudança visual
  • Busca global, notificações, "Ask Copilot", identidade de usuário nomeada — nenhuma dessas capacidades existe
  • Link "Ver todos" nos cabeçalhos de seção — não existe uma segunda view para linkar
  • Qualquer alteração de backend, contrato de API, agente de IA ou Demo Mode
  • Painel de marca do Login (já classificado "importante"/adiado na Visual Fidelity Sprint) — fora do recorte desta feature, que é exclusivamente Dashboard
06

User Stories

Como PMO executivo, vejo a tabela de projetos como o conteúdo principal da tela, sem precisar rolar além dos KPIs para chegar nela.

Critério: ordem real das seções passa a ser KPIs → Projetos → Distribuição de saúde → Maior concentração de risco.

Como usuário aguardando o carregamento, vejo um esqueleto que já antecipa a forma real da tela (tabela em md/lg, cards em mobile).

Critério: DashboardSkeleton não renderiza mais o grid de cards antigo para a seção de projetos.

Como usuário sem projetos analisados ou diante de uma falha de carregamento, vejo um estado visualmente consistente com o resto do produto.

Critério: EmptyState e error.tsx usam o mesmo raio/sombra de card já estabelecido, sem novo conteúdo textual além do já existente.

08

Critérios de Aceite

  • Ordem real das seções no DOM: KPIs, Projetos (tabela), Distribuição de saúde, Maior concentração de risco
  • Cabeçalho da seção "Projetos" mostra contagem real (projects.length) — nunca um valor fixo
  • DashboardSkeleton não contém mais o grid de 3 blocos h-32 em formato de card
  • EmptyState e error.tsx usam rounded-lg/shadow-md consistentes com Card
  • Nenhum novo campo, rota, KPI ou coluna aparece em nenhum estado da tela
  • Suíte completa (backend + frontend + E2E) passa sem alteração de contrato
09

Riscos Específicos

  • Baixo risco geral — reorganização de apresentação sobre dados já reais, sem mudança de contrato.
  • Reordenação de seções pode exigir ajuste nos testes E2E existentes que talvez assumam a ordem atual (shell.spec.ts, testes de page.tsx) — precisa de verificação explícita, não suposição.
  • Ambiguidade do limite Figma-não-modela-estados (Seção 3) — se qualquer refinamento de estado vazio/erro parecer inventar conteúdo, interromper e levar para decisão do Product Owner em vez de resolver unilateralmente.
