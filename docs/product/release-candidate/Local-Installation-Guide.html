STRATECH V1 RC-1 — Local Installation Guide

Guia completo para instalar e executar a STRATECH V1 RC-1 localmente, do zero, em macOS/Linux ou Windows. Validado nesta revisão com uma instalação limpa real, a partir de um clone isolado sem qualquer ambiente pré-existente (sem .venv, sem node_modules, sem demo/.env) -- ver Installation-Report.html para as evidências.

Instalação

macOS/Linux:

git clone <url-do-repositorio> ai-pmo-copilot
cd ai-pmo-copilot
bash scripts/rc1-local-start.sh

Este único comando: verifica Python/Node, cria o ambiente virtual Python (.venv), instala requirements.txt, instala as dependências do frontend (web/node_modules), cria demo/.env com uma SESSION_SECRET gerada, e inicia backend + frontend.

Windows:

git clone <url-do-repositorio> ai-pmo-copilot
cd ai-pmo-copilot
setup.bat
start.bat

Equivalente ao script acima, em duas etapas (setup, depois start) por causa de como o Windows gerencia janelas/processos em segundo plano. Versões PowerShell equivalentes: setup.ps1 / start.ps1.

Configuração e variáveis de ambiente

Nenhuma variável precisa ser definida manualmente para uso local em Demo Mode -- o primeiro start cria demo/.env a partir de demo/.env.example automaticamente. Variáveis relevantes (já preenchidas pelo script):

VariávelValor em Demo ModePropósito
LLM_PROVIDERmockUsa MockLLMProvider -- nenhuma chave de API real necessária
API_KEYgerado em demo/.env.exampleAutenticação do backend (X-API-Key)
WORKSPACE_PASSWORDdemo-local-passwordSenha de login do workspace (sessão compartilhada)
SESSION_SECRETgerado automaticamenteAssina a sessão HMAC do workspace

Para usar um provider de LLM real em vez do Demo Mode, defina LLM_PROVIDER=anthropic e ANTHROPIC_API_KEY em demo/.env antes de iniciar -- ver demo/.env.example para a lista completa comentada.

Banco de dados

Uso local (este guia): SQLite, arquivo único (ai_pmo_copilot.db), criado e migrado automaticamente pelo próprio backend (Base.metadata.create_all() em src/database/repository.py) -- nenhum comando de migração manual é necessário para uso local.

Uso com Postgres real (opcional, requer Docker): ver docs/operations/PRI-009-production-deployment-runbook.md, que já documenta docker compose up + alembic upgrade head para esse cenário -- fora do escopo deste guia de instalação local.

Migrations

Só relevante para o cenário Postgres/Docker acima. Para SQLite local, o schema é criado automaticamente na primeira execução -- não há passo de migração a executar.

Execução — backend

Já coberto pelo script de start. Para rodar manualmente, sem o script:

source .venv/bin/activate   # Windows: .venv\Scripts\activate.bat
uvicorn src.main:app --host 0.0.0.0 --port 8000

Execução — frontend

cd web
npx next dev -p 3000

Troubleshooting

SintomaCausa provávelSolução
ERROR: python not foundPython não está no PATHInstalar Python 3.11+ e reabrir o terminal; no Windows, marcar "Add python.exe to PATH" no instalador
ERROR: node not foundNode.js não está no PATHInstalar Node.js 22+ (nodejs.org) e reabrir o terminal
Backend não sobe / curl http://localhost:8000/health falhaPorta 8000 já em uso, ou dependências não instaladasVerificar demo/logs/backend.log (macOS/Linux) ou a janela do backend (Windows); confirmar que setup rodou sem erro
Frontend não sobePorta 3000 já em uso, ou npm install incompletoVerificar demo/logs/frontend.log ou a janela do frontend; rodar npm install novamente em web/
Login falha com "Senha do workspace incorreta"demo/.env tem uma WORKSPACE_PASSWORD diferente da usadaConferir o valor real em demo/.env -- não assumir o valor padrão do exemplo
Dashboard abre vazio (sem projetos)O portfólio de demonstração ainda não foi populadoRodar python3 demo/seed_demo_data.py com o backend já no ar
Erros de permissão executando .sh no macOS/LinuxScript sem permissão de execuçãochmod +x scripts/rc1-local-start.sh (já vem com essa permissão no repositório, mas alguns clientes Git/OS a removem)
Windows: PowerShell recusa executar .ps1Política de execução padrão do Windows bloqueia scripts não assinadosUsar os arquivos .bat em vez dos .ps1 (não exigem alteração de política), ou rodar Set-ExecutionPolicy -Scope Process Bypass antes, só para a sessão atual
