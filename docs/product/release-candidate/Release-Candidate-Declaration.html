Objetivo do RC-1

Preservar a estabilidade da plataforma, garantir rastreabilidade de qualquer alteração, impedir evolução funcional na branch RC, e preparar a STRATECH V1 para pilotos controlados com clientes reais. O RC-1 não é uma nova fase de desenvolvimento -- é um regime de estabilização sobre uma plataforma já funcionalmente completa e auditada.

Escopo funcional congelado

As 8 Capabilities da STRATECH V1, cada uma respondendo exatamente 1 pergunta executiva, congeladas no estado aprovado pela Executive Review de Organizational Intelligence e consolidadas na Product Constitution:

Nenhuma nova Capability, integração, alteração arquitetural ou de UX é permitida enquanto o RC-1 estiver em vigor. Alterações aceitas: correções de defeito, ajustes de segurança, documentação, observabilidade e hardening -- ver docs/product/release-candidate/Baseline-Configuration.html para o controle formal de versão que passa a reger qualquer mudança.

Critérios para promoção à V1.0

O RC-1 é promovido a V1.0 (release estável, não mais "candidata") quando, cumulativamente:

  1. Ao menos um piloto controlado (ver Pilot-Readiness-Checklist.html) for concluído com os critérios de aceite do piloto satisfeitos.
  2. Nenhum defeito classificado como bloqueante permanecer aberto contra a branch RC.
  3. Todo item aceito durante o RC (correção/segurança/documentação/observabilidade/hardening) estiver registrado na Baseline Configuration com controle de versão formal.
  4. O Founder/CTO aprovar formalmente a promoção -- esta decisão nunca é automática nem inferida por este ou qualquer outro documento.

Critérios para abertura da V2

A V2 só pode ser aberta formalmente quando:

  1. A V1.0 estiver promovida (Seção "Critérios para promoção à V1.0").
  2. O Founder/CTO autorizar explicitamente o início do desenvolvimento da V2 -- nenhum item do V2-Candidate-Backlog.html é implementado antes dessa autorização, mesmo que já esteja catalogado.
  3. Um novo ciclo completo de governança (Discovery → User Journey → UX Flow → FS → Architecture Review → TIP) for aberto para cada item priorizado da V2 -- o backlog é candidato, não um compromisso de escopo.
Nenhuma linha de código deve ser criada apenas para "melhorar" a plataforma durante o RC-1. A missão desta fase é preservar a estabilidade do RC-1. Toda proposta funcional futura é registrada exclusivamente no V2-Candidate-Backlog.html, nunca implementada nesta branch.
