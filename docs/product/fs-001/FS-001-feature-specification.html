PR

Product Review (Portão 1, AI-PEF v1.0)

Reexecutado sob ADR-010: FS-001 é a primeira Feature do primeiro produto integrado do AI PMO Copilot — nenhuma métrica histórica de negócio pode existir ainda. As 2 perguntas sem baseline são respondidas como Product Hypothesis, não como afirmação vaga.

  1. Problema de negócio: PMOs agregam manualmente o status de portfólio (Product Vision, Blueprint §1) — respondível com evidência qualitativa real.
  2. Melhora a tomada de decisão: visibilidade agregada + priorização por risco (W5) para as personas que precisam de agregação antes de detalhe — respondível, grounded em FS-001 §2.
  3. Reduz esforço operacional: uma tela agregada substitui abrir N telas — respondível funcionalmente.

Product Hypothesis 1 — Métrica de negócio

Problema observado→PMOs/gestores gastam tempo desproporcional agregando manualmente status de portfólio (Blueprint §1), sem medição hoje
Hipótese→O Dashboard reduz o tempo que um PMO Analyst/Executive Sponsor gasta obtendo visão de portfólio, frente ao processo manual atual
Benefício esperado→Menos tempo em preparação de relatório/reunião de status de portfólio
Métrica esperada→Tempo relatado para montar visão de portfólio (pesquisa) + acessos únicos semanais ao Dashboard (proxy de adoção, já em FS-001 §19)
Método de validação→Pesquisa qualitativa com 3–5 usuários piloto, mesma pergunta antes e depois do release
Critério de sucesso→Redução de tempo relatada por ≥70% dos usuários piloto, ou adoção semanal recorrente nas primeiras 4 semanas
Horizonte de validação→4 semanas após o Release

Product Hypothesis 2 — Valor de mercado

Problema observado→O diferencial "rollup de portfólio sem agregação manual" (Blueprint §1) nunca foi demonstrado com usuário/comprador real — produto ainda sem nenhuma tela integrada a dado real
Hipótese→Um Dashboard funcional viabiliza a primeira demonstração real do produto, materializando o diferencial já declarado na visão de produto
Benefício esperado→Destrava validação de mercado hoje impossível (nada demonstrável integrado a dado real)
Métrica esperada→Nº de demonstrações/pilotos reais viabilizados pelo Dashboard
Método de validação→Registro de sessões de demonstração que usaram o Dashboard como tela principal + feedback qualitativo coletado nessas sessões
Critério de sucesso→≥1 demonstração real (interna ou externa) usando o Dashboard dentro de 60 dias do Release
Horizonte de validação→60 dias após o Release
Veredito do Product Review: as 5 perguntas estão respondidas — 3 com evidência qualitativa direta, 2 com Product Hypothesis mensurável e falsificável (ADR-010). Portão 1 aprovado. As 2 hipóteses ficam registradas como dívida de validação, com checagem obrigatória no Portão 11 (Knowledge Base Update) dentro dos horizontes acima — não é uma aprovação permanente sem essa checagem.
1

Objetivo

Dar ao PMO Lead / gestor de portfólio uma visão agregada da saúde de todos os projetos em uma única tela, sem precisar abrir cada projeto individualmente — a dor central descrita no Product Vision do Product Blueprint ("consolidar em minutos o que hoje leva horas de reuniões e planilhas").

É a primeira tela funcional do produto a consumir dado real do backend (Sprint 1 entregou apenas o Design System, sem integração).

Valor para o Negócio→Primeira tela que materializa o diferencial de rollup automático já declarado no Product Vision (Blueprint §1) — viabiliza demonstração real do produto (Product Hypothesis 2, acima)
Valor para o Usuário→Elimina a agregação manual de status entre projetos para o Executive Sponsor e o PMO Analyst (Product Hypothesis 1, acima)

Perguntas que o Dashboard deve responder em menos de 30 segundos

Derivadas exclusivamente dos 5 campos reais de ProjectSummaryResponse — nenhuma pergunta abaixo depende de dado que não existe hoje:

  • Quantos projetos têm alguma análise registrada no portfólio?
  • Quais projetos têm o maior número de riscos extraídos (open_risks)?
  • Quais projetos têm ações pendentes acumuladas (pending_action_items)?
  • Qual o status de saúde mais recente de cada projeto (latest_health_status), e quantos estão fora do estado saudável?
  • Qual projeto tem mais atividade de análise registrada (total_analyses), como sinal indireto de quanto monitoramento aquele projeto vem recebendo?
Pergunta que o Dashboard não pode responder hoje, e não deve prometer: "Qual projeto está mais atrasado?", "Qual o custo/ROI do portfólio?", "O que mudou desde ontem?". Nenhum desses dados existe no schema real (sem cronograma, sem custo, sem snapshot histórico — ver Data Sources §10 e Product Blueprint §12, risco "domain model real muito distante do alvo").
Nuance semântica importante sobre open_risks: o nome vem do schema da API, mas o valor é a contagem cumulativa de todos os riscos já extraídos por análises risk com saída estruturada — não existe estado de risco "resolvido/fechado" hoje (Risk não tem máquina de estados no Product Blueprint). O rótulo "aberto" não deve ser lido pelo usuário como "ainda pendente de tratamento"; é matéria de UX Review decidir como comunicar isso na tela sem induzir a essa leitura incorreta.

  • Estado de erro (falha ao contatar o backend — usa error.tsx do App Router).

4

Fora de escopo (explícito)

  • Drill-down por projeto — pertence à Release 0.3 (Gestão de Projetos), tela própria com rota /projetos/[nome].
  • Gráfico de tendência/série temporal — não existe dado histórico agregável hoje; AnalysisRecord não versiona snapshots de portfólio ao longo do tempo. Construir isso exigiria uma mudança de schema fora do escopo desta feature.
  • Filtro por período no Dashboard — os parâmetros created_from/created_to existem em GET /api/analyses, mas essa é uma tela diferente; o Dashboard mostra sempre o estado atual.
  • Exportação (PDF/Excel/CSV) — não mencionado em nenhum artefato aprovado anteriormente; não inventado aqui.
  • Atualização em tempo real via push — RFC-001 decidiu Polling como estratégia (D3), detalhado na Data Freshness Matrix (§10).

Information Architecture

Fluxo completo e real, da origem do dado até a apresentação na tela — nenhuma etapa abaixo é hipotética, todas correspondem a código existente.

  1. Origem (fora desta feature): um PM/PMO envia texto livre via POST /api/meetings/analyze, /api/risks/analyze ou /api/projects/analyze, em outra tela do produto.
  2. O agente de IA correspondente (Meeting Intelligence, Risk Review ou Project Status) processa o texto e produz um payload estruturado (ou fallback, se a saída do modelo não seguir o schema pedido).
  3. AnalysisRepository persiste o resultado como um AnalysisRecord — tabela única, imutável, sem PUT/PATCH/DELETE.
  4. Ao receber uma requisição em GET /api/portfolio/summary, ProjectSummaryService.summarize_portfolio() busca todas as análises em uma única query e agrupa em memória por project_name — não existe agregação pré-computada ou materializada; o número é sempre recalculado do zero a cada chamada.
  5. O backend FastAPI serializa a lista como list[ProjectSummaryResponse], protegida por X-API-Key (Nível 2) e pelo rate limiter.
  6. A Route Handler do BFF (/api/bff/dashboard, a ser criada nesta feature) recebe a requisição do browser, injeta o X-API-Key (nunca exposto ao cliente) e repassa a chamada ao backend.
  7. O browser (TanStack Query) busca via fetch, aplica staleTime e cacheia — ver Data Freshness Matrix (§10).
  8. React renderiza W1/W2 (e W3–W5, se aprovados) a partir do mesmo objeto em cache — nenhum widget refaz a chamada de rede.
  9. Enquanto a aba está visível, o polling condicional de 60s (RFC-001, decisão D3) repete os passos 4–8 automaticamente; o usuário também pode forçar via botão de atualização manual.

Data Freshness Matrix

Política de atualização por conjunto de informação — grounded na Estratégia de Estado (RFC-001 §6) e na decisão de Tempo Real (RFC-001 §7, D3). O backend não mantém nenhum cache próprio: summarize_portfolio() recalcula a cada chamada (§8) — toda a "obsolescência" descrita abaixo é inteiramente client-side.

Conjunto de dadosstaleTimePollingAtualização manual
W1 Portfolio Summary Strip30s60s (refetchInterval, só enquanto a aba está visível — RFC-001 D3)Botão sempre disponível, mecanismo primário — o poll é complemento
W2 Project Health Grid30s
W3 Health Status Distribution30s
W5 Risk Concentration Ranking30s
W4 Recent Analyses Feed candidato30s (família analyses-list)Se aprovado: mesmo ciclo de 60s de W1/W2 — não introduzir um segundo timerHerdado do botão da tela

W1, W2, W3 e W5 compartilham a mesma chamada de rede — são reapresentações do mesmo payload, portanto têm exatamente a mesma frescura por construção; não é possível um estar "mais atualizado" que outro.

Restrição de orçamento a respeitar: rate limit de 60 requisições/60s por chave, compartilhado entre todos os operadores do workspace via BFF (RFC-001). Cada widget adicional com fonte de dados própria (como W4) consome desse mesmo orçamento — argumento a favor de manter W3/W5 como reapresentação client-side em vez de novas chamadas, e de sincronizar o timer de W4 ao de W1/W2 se aprovado.

Nenhum indicador de "atualizado há Xmin" está no escopo desta revisão — RFC-001 já registrou isso como decisão em aberto para o Sprint 2 (Pergunta em Aberto).

Mapeamento de dados → UI

Todo campo abaixo vem de ProjectSummaryResponse (schema real, src/api/routes/intelligence.py:64), com semântica confirmada em docs/technical/04-api-design.md. Nenhum campo é inventado.

project_name→Título do card
total_analyses→Contador "Análises registradas" — inclui análises fallback (não-estruturadas)
open_risks→Contador "Riscos identificados" (renomeado no UX Review — evita a leitura incorreta de "aberto" como "pendente de tratamento") + tooltip explicando que é contagem cumulativa; soma de risks[] entre análises risk estruturadas; destaque visual quando > 0
pending_action_items→Contador "Ações pendentes" — soma de action_items[] entre análises meeting estruturadas
latest_health_status→Badge via healthStatusVariant() — vem da análise status estruturada mais recente; null se nenhuma existir
Notas de layout do UX Review: project_name não tem limite de tamanho no schema (str | None, sem max_length) — truncar com ellipsis + atributo title no card. Contadores sem cap superior conhecido — usar font-variant-numeric: tabular-nums, largura não fixa. Grid via auto-fill/minmax, não colunas fixas, para funcionar igualmente bem com poucos ou muitos projetos. Detalhe completo em UX-REVIEW-FS001.html §2.
