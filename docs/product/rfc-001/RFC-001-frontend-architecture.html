Decisões Arquiteturais Registradas

Formato fixo: Decisão, Justificativa, Alternativas rejeitadas, Consequências. As decisões operacionais menores ficam registradas dentro de cada seção temática.

D1 — Padrão Backend-for-Frontend (BFF)
Decisão
O navegador nunca fala diretamente com o FastAPI. Toda chamada passa por Route Handlers do Next.js em app/api/*.
Justificativa
O backend usa uma API_KEY única compartilhada — se o navegador a guardasse, qualquer DevTools a exporia.
Alternativas rejeitadas
Guardar a chave em localStorage/env pública (inseguro); pedir ao usuário sua própria chave por sessão (o backend não suporta múltiplas chaves).
Consequências
Toda rota de produto precisa de um Route Handler correspondente; latência adicional de um hop; mas é o único ponto de troca de credencial, o que habilita a seção 9 (evolução de auth) sem quebra.
D2 — Radix UI vendorizado à mão, sem CLI do shadcn/ui
Decisão
Primitivos escritos em components/ui/ a partir de @radix-ui/react-* instalados via npm.
Justificativa
ui.shadcn.com está bloqueado pela política de rede deste ambiente (403 do proxy).
Alternativas rejeitadas
MUI/Mantine/Chakra (linguagem visual própria, bundle maior); esperar a liberação de rede (sem prazo definido).
Consequências
Atualizações de acessibilidade do Radix exigem merge manual; ganho é controle total do código, sem dependência de runtime de terceiros para estilo.
D3 — Sem push em tempo real (SSE/WebSocket) no MVP
Decisão
Nenhum mecanismo de push. Ver comparação completa na seção 7.
Justificativa
Nenhum dos 7 endpoints reais emite eventos ou faz streaming — construir push sobre uma API request/response seria arquitetura paralela.
Alternativas rejeitadas
SSE simulado no BFF sem fonte real de eventos; WebSocket "para o futuro" sem consumidor hoje.
Consequências
Dashboard usa polling condicional (seção 7); revisitar apenas se o backend ganhar um endpoint assíncrono real.
D4 — Dois níveis de autenticação, não um
Decisão
Nível 1 (sessão de workspace, navegador↔BFF) separado de Nível 2 (credencial de backend, BFF↔FastAPI). Ver seção 9.
Justificativa
Um 401 do FastAPI hoje só pode significar "o operador configurou a API_KEY errada" — nunca "sessão de um usuário expirou", porque não há usuários.
Alternativas rejeitadas
Tratar os dois como um único conceito de "login" (a Revisão 1 cometeu esse erro).
Consequências
É essa separação que torna a migração para JWT/OIDC (seção 9) uma mudança contida ao BFF, não uma reescrita.
D5 — Nenhuma store global de estado
Decisão
TanStack Query é o estado de servidor; não há Redux/Zustand/Context global de dados.
Justificativa
A API já é a fonte da verdade; duplicar esse dado numa store separada cria duas fontes para sincronizar.
Alternativas rejeitadas
Redux Toolkit (overhead de boilerplate sem necessidade real); Zustand para tudo (mistura estado de servidor com estado de UI).
Consequências
Estado de UI puro continua em useState/URL — ver seção 6.
D6 — Navegação mobile: bottom tab bar, não drawer
Decisão
Abaixo de 768px, a sidebar vira uma barra fixa inferior de 3 itens (Portfólio · Análises · Nova análise).
Justificativa
Só existem 2 destinos de navegação + 1 ação primária — um drawer (menu hambúrguer) adicionaria um toque extra para algo que cabe sempre visível.
Alternativas rejeitadas
Drawer lateral (padrão para >5 itens, over-engineering aqui); esconder "Nova análise" atrás de um FAB (menos descobrível).
Consequências
Se a navegação crescer além de 3-4 destinos, este padrão precisa ser revisitado — documentado como gatilho, não ignorado.
D7 — "Central de Agentes" v1 sem novo endpoint
Decisão
Reenquadrar o Sprint 5 como uma vista dos 3 tipos de análise já existentes (meeting/risk/status), agregados via GET /api/analyses?kind=, em vez de "gestão de agentes" (que exigiria endpoint novo).
Justificativa
Os três "agentes" já têm identidade real no backend como valores de kind — uma vista de contagem/atalho por tipo é honesta e buildável hoje; "gerenciar" um agente (editar prompt, ver config) não é.
Alternativas rejeitadas
Esperar um endpoint de gestão de agentes antes de tocar o Sprint 5 inteiro (desperdiça o que já é possível); construir uma tela de configuração fictícia (violaria "não inventar endpoints").
Consequências
Sprint 5 passa a ter escopo v1 real (seção 10) — só a parte de configuração/gestão continua bloqueada.

Dark theme

Implementado desde o Sprint 1 via @media (prefers-color-scheme: dark) (segue o SO por padrão) com override manual possível via data-theme="dark"/data-theme="light" no elemento raiz — mesmo padrão usado em todo artefato de engenharia desta sessão. Nenhum token tem só uma leitura: todo par claro/escuro foi definido junto, não invertido automaticamente (inversão ingênua quebra contraste em cores saturadas como o accent).

Acessibilidade e contraste

Razões de contraste calculadas via a fórmula de luminância relativa do WCAG 2.1 ((L1+0.05)/(L2+0.05)), não estimadas:

ParTemaRazãoNível

Honestidade sobre cobertura: os 5 pares acima foram calculados manualmente para validar o método e os pares de maior risco (texto secundário, botão primário). Os demais pares (badges de status, dark mode completo) precisam do mesmo cálculo — proposto como um passo de CI no Sprint 2 (axe-core ou Lighthouse CI rodando contra /style-guide), não verificação manual repetida a cada novo token. Nenhuma alegação de conformidade total até esse gate existir.

Convenções adicionais: todo elemento interativo tem estado focus-visible distinto (nunca removido sem substituto); toda cor semântica (ok/warn/danger) é acompanhada de texto ou ícone, nunca só cor (daltonismo); prefers-reduced-motion respeitado nas transições do Dialog/Toast.

Estratégia de Estado

Estado de servidor

Vive exclusivamente no cache do TanStack Query. Nenhum dado vindo da API é copiado para useState/store — o componente lê direto do hook de query (usePortfolio(), useProjectSummary(name) etc).

Estado local

useState por componente, para UI pura sem correspondência no servidor: aba ativa do Composer, diálogo aberto/fechado, item de tabela em hover. Nunca elevado a um contexto global — se dois componentes distantes precisarem do mesmo estado de UI, isso é sinal de que deveria estar na URL, não numa store.

Estado de formulário

React Hook Form, isolado por formulário — não compartilha estado com o restante da árvore. Validação via schema Zod, espelhando literalmente as constraints reais do backend (min_length=10, max_length=20000). Erros de validação vivem no estado interno do React Hook Form, nunca duplicados em useState paralelo.

Cache

QuerystaleTimeJustificativa
portfolio-summary30sRollup que muda com uso de outros operadores do workspace
project-summary(name)30sMesma razão, escopo menor
analyses-list(filters)30sComplementado por invalidação sob demanda
analysis-detail(id)InfinityBackend não tem PUT/PATCH/DELETE para análises — imutável por design da API
agent-type-counts30sMesma família de analyses-list, só reagrupada

Invalidação

Um POST de análise bem-sucedido invalida: analyses-list (qualquer filtro), project-summary do projeto envolvido, portfolio-summary, e agent-type-counts. Nunca invalida analysis-detail de registros existentes.

Persistência

Decisão: nenhuma persistência do cache de servidor entre sessões (sem localStorage/IndexedDB para o TanStack Query). Justificativa: os dados são baratos de re-buscar (a API já responde rápido para leitura), e persistir cache aumentaria o risco de mostrar dado obsoleto sem que o usuário perceba. O que persiste: o cookie de sessão de workspace (nível 1) e os filtros de /analyses (na URL, não em storage local).

Tratamento de dados obsoletos

Padrão stale-while-revalidate do próprio TanStack Query: dado em cache é mostrado imediatamente (mesmo se "stale"), uma revalidação em background acontece conforme a política acima, e a UI atualiza sem piscar quando o novo dado chega. Nenhum indicador de "desatualizado" é mostrado ao usuário hoje — avaliar no Sprint 2 se um rótulo discreto ("atualizado há Xmin", já usado no Design System da seção 1) é necessário no Dashboard, dado o poll de 60s.

Integração Next.js (BFF) × FastAPI

Fluxo da requisição

  1. Browser chama app/api/* (mesma origem, cookie de sessão nível 1 enviado automaticamente)
  2. Route Handler verifica o cookie — sem sessão válida, responde 401 (do BFF, não do FastAPI) antes de qualquer chamada externa
  3. Com sessão válida, monta a requisição ao FastAPI: anexa X-API-Key (nível 2) do env server-only
  4. Chama PMO_API_BASE_URL com AbortController (timeout ~65s)
  5. Repassa a resposta ao browser — status e corpo preservados, exceto a reescrita de 401 (ver Mapa de erros)

Propagação de headers

HeaderDireçãoTratamento
X-API-KeyBFF → FastAPIAnexado pelo Route Handler a partir do env server-only — nunca vem do browser
Cookie (sessão)Browser → BFFConsumido e verificado; nunca repassado ao FastAPI (não significa nada lá)
X-Request-IDFastAPI → BFF → BrowserRepassado sem alteração — ver Correlation ID abaixo
Content-TypeAmbas direçõesPreservado (application/json)

Proteção da API Key

Ver Decisão D1 e D4. PMO_API_KEY é uma variável de ambiente lida apenas em código de servidor (Route Handlers, nunca em Client Components) — o Next.js só expõe ao bundle do cliente variáveis prefixadas com NEXT_PUBLIC_, e esta nunca é prefixada assim. Verificação de que a chave nunca vaza: inspecionar o output de next build (o bundle client não deve conter a string da chave) como parte do critério de aceite do Sprint 2 (seção 10).

Tratamento de erros

StatusCausa realLeitura na UI
401 (do FastAPI)API_KEY do BFF está errada — erro de configuração do operadorReescrito para 503 pelo BFF — "workspace indisponível, contate o operador"
503API_KEY não configurada, ou provider de IA sem config"Serviço temporariamente indisponível"
429Rate limit por chave excedido (compartilhado)"Muitas requisições — tente novamente em instantes" (sem Retry-After hoje, ver Perguntas)
502Chamada real à Anthropic falhou"O motor de IA não respondeu — tente novamente"
422Validação Pydantic (replicada no Zod client-side)Erro inline — raro, já barrado antes do envio
404Análise não encontrada por idEstado vazio dedicado

Timeout

AbortController no fetch do Route Handler, ~65s — acima de qualquer SLA conhecido (o backend não documenta um, ver Riscos), abaixo do timeout padrão de infraestrutura.

Retry

GETs: retry automático do TanStack Query (3x, backoff exponencial) — seguro, idempotente. POSTs de análise: nunca retry automático — uma chamada já pode ter custado uma chamada real à Anthropic; reenvio é sempre ação explícita do usuário.

Correlation ID

O X-Request-ID gerado pelo backend (US-OBS-01, já em produção) é repassado sem alteração até o browser, e exibido em todo ErrorBanner. Adicionalmente, o próprio BFF gera seu próprio X-Request-ID para erros que nunca chegam a sair do Next.js (ex: sessão inválida, timeout antes de qualquer resposta do FastAPI) — os dois IDs são distintos e ambos logados, para que um erro seja rastreável independente de em qual camada ele se originou.

Logging

Route Handlers logam (stdout, capturável pelo mesmo docker-compose.yml se o frontend ganhar um serviço lá — ver Perguntas): método, rota, status de resposta, latência, e o X-Request-ID relevante. Nunca logam o corpo da requisição (pode conter texto de transcrição/contexto de projeto) nem a API_KEY.

Evolução da Autenticação

Só é possível responder isto sem quebra de arquitetura por causa da Decisão D4 (dois níveis). A migração troca o mecanismo do nível 1 — nunca o contrato entre componentes/rotas/hooks e o BFF, porque eles nunca conheceram o mecanismo, só um cookie de sessão.

FaseNível 1 (sessão)Nível 2 (credencial de backend)Impacto em rotas/componentes
0 · API Key compartilhada (hoje)N/A — não existe sessão de usuário, só a chave direta se alguém chamasse a API sem BFFAPI_KEY únicaN/A
1 · Sessão no BFF (Sprint 2)Senha compartilhada → cookie assinadoAPI_KEY única, inalteradaNenhum — é a arquitetura já descrita neste RFC
2 · Usuários individuaisLogin por usuário (email/senha) → cookie com user_idAPI_KEY ainda única — o backend precisa primeiro de uma tabela User (RFC de backend separado, fora deste documento)Nenhum em web/ além do próprio módulo de sessão — lib/api-client.ts continua falando só com o BFF
3 · JWT/OIDCRedireciona a um provider (Auth0/Okta/Azure AD), troca código por token, guardado na mesma forma de cookie assinadoRoute Handler anexa Authorization: Bearer <jwt> em vez de/junto com X-API-Key — backend precisa validar JWT (RFC de backend)Um arquivo (lib/session.ts) muda o mecanismo; zero mudança em rotas/componentes/hooks
4 · RBACClaims de papel (role: pmo_manager | executive | admin) no token/sessãoBackend passa a checar permissão por rota — RFC de backendComponentes de nav/ação passam a checar useSession().role para mostrar/esconder itens — aditivo, não uma reestruturação
5 · Multi-tenantClaim de tenant_id; possivelmente subdomínio por tenantproject_name hoje é texto livre global — precisaria de escopo por tenant no backend (mudança de modelo de dados, RFC de backend)Fora do horizonte deste documento — mencionado porque foi pedido, não porque está próximo de acontecer

Compatibilidade durante a transição

Cada fase é aditiva sobre a anterior no nível 1 (o formato de cookie assinado não muda de forma incompatível — só ganha claims novos, com valores default para sessões antigas). No nível 2, a Fase 3 pode conviver com a Fase 1 durante uma migração gradual: o Route Handler tenta Authorization: Bearer se disponível, cai para X-API-Key caso contrário — mas isso exige que o backend aceite ambos simultaneamente, decisão de um RFC de backend, não deste documento.

Por que isso não quebra nada: nenhum componente, rota ou hook de web/ jamais fala diretamente com o FastAPI ou lê o cookie de sessão — todos passam por lib/api-client.ts. A Fase 3 é uma mudança contida a esse arquivo e ao módulo de sessão.

Riscos Globais

  • Latência de análise sem SLA conhecido — o backend não documenta um teto de tempo de resposta do provider Anthropic. Mitigação: estado "demorando mais que o normal" no Composer (Sprint 3), nunca uma barra de progresso que promete um tempo que ninguém garante.
  • Rate limit compartilhado por todo o workspace — 60 req/60s (default) dividido entre todos os usuários simultâneos via BFF. Mitigação: polling conservador (seção 7), sem retry automático em POST (seção 8).
  • Radix vendorizado, não versionado como dependência — atualizações de acessibilidade/bugfix do Radix upstream exigem merge manual dos wrappers em components/ui/. Aceito conscientemente (Decisão D2).
  • Senha de workspace única não audita quem fez o quê — aceito até a Fase 2 da seção 9 (usuários individuais), que depende de backend fora do escopo deste RFC.
  • Cobertura de contraste WCAG parcial — 5 pares verificados manualmente; os demais dependem de um gate de CI ainda não construído (seção 1).

Perguntas em Aberto

  • Vale a pena adicionar Retry-After ao 429 do backend?

    Pequena mudança de backend, fora do escopo deste RFC (frontend-only). Melhoraria a UI de rate limit de "tente de novo mais tarde" para um contador real.

  • Deploy: nova entrada no docker-compose.yml existente, ou pipeline separado?

    O Next.js com Route Handlers precisa rodar como servidor Node — compatível com um terceiro serviço no docker-compose.yml já consertado (TASK-INF-02), mas não desenhado neste RFC.

  • A senha de workspace é rotacionável sem invalidar todas as sessões?

    Hoje, trocar WORKSPACE_PASSPHRASE não invalida cookies já emitidos (assinados com SESSION_SECRET, um segredo diferente) — decisão consciente para a Fase 1 da seção 9, vale confirmar se é aceitável operacionalmente antes do Sprint 2.

  • O gate de contraste WCAG entra no CI do Sprint 2 ou fica para depois?

    Proposto como parte do Sprint 2 (mesmo PR que cria o pipeline de CI de frontend, já existente desde o Sprint 1) — mas é uma adição de escopo que vale confirmação explícita antes de assumida.
