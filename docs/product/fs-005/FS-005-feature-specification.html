Feature Specification · Release 0.3

FS-005 — AI Experience

Melhora como o usuário escolhe, aciona e acompanha os 3 agentes de IA já reais (project_status, risk_review, meeting_intelligence), e como distingue conteúdo gerado por IA de dado agregado pelo sistema. Nenhum agente novo, nenhum chat genérico, nenhuma síntese entre agentes, nenhuma mudança de schema.

Release 0.3 Origem Autorização do Product Owner, pós-encerramento da FS-004 Data 2026-07-12 Status Aguardando aprovação — TIP-005 ainda não iniciado. Sem código escrito.
01

Contexto e Objetivo

Os 3 agentes de IA do produto são reais, testados e já persistem resultado estruturado em analysis_records — mas hoje não existe nenhuma forma de acioná-los pela interface. O Workspace (FS-004) já lê e exibe o resultado dessas análises com excelência, mas nunca as produz. O objetivo desta feature é fechar exatamente essa lacuna: a experiência de acionar um agente, não a de ler seu resultado (já resolvida).

02

Fluxo Atual (Mapeamento completo)

Levantamento direto do código, antes de qualquer proposta de tela.

Usuário→ Endpoint→ Agente→ Persistência→ Dashboard/Workspace
UsuárioAchado central Não existe nenhum ponto de acionamento na UI. Os únicos invocadores reais dos 3 endpoints POST /api/{"{"}meetings,risks,projects{"}"}/analyze são chamada direta de API (curl) e demo/seed_demo_data.py. web/app/style-guide/page.tsx tem um Textarea com contador de caracteres e uma Tabs listando os 3 endpoints como texto estático — decorativo, sem onSubmit real, sem fetch para nenhum dos 3 endpoints.
EndpointPOST /api/meetings/analyze (transcript, 10–20000 chars), POST /api/risks/analyze e POST /api/projects/analyze (ambos project_context, mesma faixa) — todos já reais, já validados (Field(min_length=10, max_length=20000) + validator contra string só de espaços), já protegidos por X-API-Key e rate limit (60 req/60s por chave, src/api/rate_limiter.py).
AgenteCada agente chama o LLM real (claude-3-5-sonnet-20241022, max_tokens=1200) sem timeout configurado no lado do backend — só o SDK padrão. Retorna {"{"}agent, project_name, model_output{"}"}, com model_output.structured=false como fallback caso o parse JSON falhe.
Persistênciarepository.save_analysis(kind=..., payload=result, project_name=...) grava em analysis_records — inalterado, já é exatamente a mesma tabela que a FS-004 lê.
Dashboard/WorkspaceO Workspace já lê tudo isso (Riscos/Ações/Decisões/Recomendações/Timeline/Histórico). Achado Nenhuma seção distingue visualmente texto gerado pela IA de dado agregado pelo sistema — o "Achados-chave" (IA) e as contagens (sistema) do Executive Summary usam exatamente o mesmo Card, sem nenhum marcador. A única identificação de proveniência em qualquer tela é o badge genérico de kind ("Reunião"/"Risco"/"Status") — nunca o nome do agente, mesmo esse campo (payload.agent) já existindo no dado real.
03

Problemas Reais Identificados

  1. Nenhum ponto de acionamento. O usuário não tem como rodar uma análise nova pela interface — a única forma é uma chamada de API manual.
  2. Nenhuma captura de contexto real. O Textarea do style-guide é decorativo; não existe formulário funcional para transcript/project_context.
  3. Nenhum padrão de loading para submissão. Todo o loading já construído (Workspace, Dashboard) é para leitura (GET); não existe padrão para "enviei algo, aguardando o agente responder" — e, sem timeout no backend, uma UI ingênua poderia esperar indefinidamente sem feedback.
  4. Nenhuma distinção entre IA e agregação. Achados-chave, riscos, ações, decisões e recomendações (todos texto gerado pelo LLM) aparecem com o mesmo tratamento visual que contagens calculadas pelo próprio sistema.
  5. Proveniência fraca. Só o kind (Reunião/Risco/Status) identifica uma entrada — nunca qual agente, mesmo esse dado já existindo no payload.
  6. Sem confirmação de sucesso. Mesmo com o Toaster (sonner) já montado globalmente (web/app/layout.tsx), nenhum fluxo real dispara um toast — não existe feedback de "análise salva com sucesso" em lugar nenhum do produto hoje.
  7. Risco de duplicidade futura. Sem um princípio explícito de onde a experiência de acionamento vive, ela poderia nascer espalhada entre Dashboard e Workspace, duplicando a leitura de resultado que a FS-004 já resolveu.

Architecture Review

Onde vive o formulárioDentro do Workspace (/workspace/[projectName]) — nunca uma tela própria. O project_name já está no contexto da URL, resolvendo "como fornece o contexto necessário" para project_name automaticamente; só falta o texto (transcript/project_context).
3 rotas BFF novas, finas (proxy puro)web/app/api/bff/workspace/[projectName]/analyze/meeting/route.ts, .../analyze/risk/route.ts, .../analyze/status/route.ts — uma por agente, corpo tipado (evita um parser de union discriminada), mesmo padrão de X-API-Key/cache:"no-store" das rotas BFF já existentes. Nenhum endpoint novo no backend — os 3 endpoints reais já existem.
Timeout do BFF para submissãoOs proxies de leitura usam 8s (adequado para uma query rápida ao banco). Uma chamada real ao LLM é mais lenta e sem timeout de backend — o BFF de submissão usa 60s, o mesmo valor já real e em uso em demo/seed_demo_data.py para o cliente httpx, não um número novo inventado.
Atualização do Workspace pós-submissãoqueryClient.invalidateQueries (TanStack Query, já em uso) nas chaves workspace-timeline/workspace-latest do projeto atual após sucesso — a nova análise aparece nas seções já existentes sem novo componente de exibição.
Marcador de proveniência de IAComponente de apresentação reaproveitando o token de Badge já existente — não um novo sistema, apenas uma variante visual consistente aplicada às seções cujo conteúdo vem de payload.model_output (Riscos, Ações, Decisões, Recomendações, Achados-chave), nunca às contagens agregadas pelo sistema.
Feedback de sucesso/errosonner (já montado globalmente, hoje ocioso) passa a disparar de verdade em sucesso ("Análise salva") e erro (mensagem amigável, incluindo 429 de rate limit).
Contrato de backendZero alteração. Os 3 endpoints, seus schemas de request/response, e o schema de analysis_records permanecem exatamente como estão.

Escopo

  • Formulário de acionamento dos 3 agentes reais, dentro do Workspace
  • 3 rotas BFF finas de submissão (proxy puro para os 3 endpoints já existentes)
  • Estados de loading/sucesso/erro para a submissão, incluindo rate limit (429)
  • Validação client-side espelhando os limites reais (10–20000 caracteres)
  • Marcador visual de proveniência de IA nas seções alimentadas por model_output
  • Exibição do agente (payload.agent) nas seções que já mostram kind
  • Atualização automática do Workspace (Timeline/Histórico/painéis "latest") após submissão bem-sucedida
08

Fora de Escopo (explícito)

  • Qualquer agente novo, chat genérico, ou síntese entre agentes (princípios 1-3)
  • Qualquer alteração de schema de saída ou de contrato de análise (princípios 4-5)
  • Confiança, previsão, ou explicação de raciocínio da IA que o backend não fornece (princípio 6)
  • Streaming de resposta, fila assíncrona, ou qualquer mudança na forma como o LLM é chamado
  • Editar ou corrigir um resultado de IA já salvo
  • Notificações, sistema de identidade de usuário, ou qualquer capacidade não já existente
  • Qualquer ponto de acionamento fora do Workspace (ex.: um "botão de análise" no Dashboard)
  • Qualquer alteração em src/agents/, nos 3 endpoints POST, ou em analysis_records

User Stories

Como PMO executivo dentro do Workspace de um projeto, escolho qual dos 3 agentes rodar e forneço o texto necessário, sem sair da tela.

Critério: os 3 agentes reais aparecem como opções; o campo de texto muda conforme o agente (transcript vs. project_context).

Como usuário aguardando uma análise, sei que ela está em andamento e recebo confirmação clara quando termina (ou uma mensagem honesta se falhar).

Critério: botão de envio mostra estado de carregamento; toast de sucesso ou erro ao concluir; erro de rate limit tem mensagem específica, não genérica.

Como usuário lendo o Workspace, distingo imediatamente o que a IA escreveu do que o sistema apenas contou/agregou.

Critério: todo conteúdo vindo de model_output tem o mesmo marcador visual consistente; contagens agregadas nunca o têm.

Como usuário que acabou de rodar uma análise, vejo o resultado aparecer no Workspace sem precisar recarregar a página manualmente.

Critério: Intelligence Timeline, Histórico completo, e o painel "latest" do tipo correspondente refletem a nova análise imediatamente após o sucesso da submissão.

NFRs Aplicáveis

  • Nenhuma regressão nos 92 testes de backend — nenhum arquivo de backend é tocado
  • Nenhuma regressão nos testes de frontend já existentes
  • Timeout de 60s no BFF de submissão, com mensagem honesta se estourado (não um estado de carregamento infinito)
  • Mensagem específica e amigável para rate limit (429), distinta de um erro genérico de backend
  • Contraste WCAG AA no marcador de proveniência de IA, em ambos os temas
  • Responsividade nos 3 breakpoints já estabelecidos
11

Critérios de Aceite

  • É possível acionar qualquer um dos 3 agentes reais a partir do Workspace de um projeto, e ver o resultado aparecer nas seções já existentes sem reload manual
  • O formulário valida o texto (10–20000 caracteres) antes de habilitar o envio, espelhando exatamente os limites do backend
  • Um erro de rate limit (429) mostra uma mensagem distinta de um erro genérico de backend
  • Toda seção alimentada por model_output (Riscos, Ações, Decisões, Recomendações, Achados-chave) mostra o marcador de proveniência de IA; nenhuma seção de contagem agregada o mostra
  • O agente que produziu cada entrada da Timeline/Histórico é identificável além do badge genérico de kind
  • Nenhum ponto de acionamento existe fora do Workspace
  • Zero alteração em src/agents/, nos 3 endpoints POST, ou no schema de analysis_records
  • Suíte completa (backend + frontend + E2E) passa sem alteração de contrato de backend

13

Riscos Específicos

  • Chamada ao LLM sem timeout no backend — o BFF precisa de um timeout próprio (60s, proposto) para nunca deixar o usuário esperando indefinidamente; se 60s for insuficiente na prática, é um ajuste de configuração, não uma mudança de arquitetura
  • Cache desatualizado após submissão — se a invalidação de query falhar silenciosamente, o usuário pode não ver o resultado sem um refresh manual; precisa de teste explícito, não suposição
  • Rate limit real durante uso legítimo — 60 req/60s é generoso para um usuário humano, mas o teste deve confirmar que a mensagem de erro é honesta, não um erro genérico assustador
14

Checklist de Segurança (ADR-011)

Aplica-se parcialmente As 3 novas rotas BFF de submissão seguem o mesmo padrão de credencial das rotas de leitura já auditadas (TIP-004): API_KEY só no servidor, nunca ecoado ao navegador. Nenhuma rota nova de sessão/autenticação é criada — reaproveita o gate já existente do Workspace. Nenhum dado sensível novo é introduzido (o texto submetido é o mesmo tipo de conteúdo — transcript/contexto de projeto — que já trafega via API hoje). Checklist completo a ser aplicado de fato na escrita do TIP-005.

Rollback

Reverter o commit desta feature remove o formulário de acionamento, as 3 rotas BFF de submissão, e o marcador de proveniência de IA. O Workspace volta a ser somente-leitura, exatamente como está hoje. Nenhuma migração de dado, nenhuma mudança de schema — rollback é reversão de código pura.
