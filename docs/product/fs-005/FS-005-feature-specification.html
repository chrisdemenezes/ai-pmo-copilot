FS-005 — Nova Análise: Project Status

Dentro do Workspace de um projeto, o usuário aciona "Nova Análise", fornece contexto, executa o agente project_status real, e vê o resultado refletido no Workspace e no Dashboard sem reload manual. Recorte deliberado: só Project Status — risk_review e meeting_intelligence ficam para incrementos seguintes, mesmo padrão, escopo menor.

Release 0.3 Origem User Workflow Discovery + decisão do Founder Data 2026-07-12 Status Aguardando aprovação — TIP-005 ainda não iniciado. Sem código escrito.
01

Contexto e Objetivo

O User Workflow Discovery mapeou os 10 passos da jornada de um Diretor de PMO usando IA para decidir e confirmou: nenhum passo depende de capacidade que não existe hoje. O menor incremento que prova a jornada inteira, ponta a ponta, é um agente só — project_status, o de schema mais simples (health_status, key_findings, recommendations, sem arrays aninhados). Esta FS entrega exatamente isso: o loop completo, uma vez, bem feito.

Substitui integralmente a versão anterior desta FS (que cobria os 3 agentes como "apresentação"). Aquela versão nunca teve TIP autorizado — nenhum código depende dela.

Restrição arquitetural para o que vem depois (decisão do Founder): após este incremento, risk_review e meeting_intelligence entram como extensões do mesmo padrão estabelecido aqui — não como experiências reinventadas do zero. Concretamente, isso significa: mesmo gatilho ("Nova Análise" no WorkspaceHeader, mesmo Dialog), mesma convenção de nomeação de rota BFF (.../analyze/{kind}/route.ts), mesmo formato de hook de submissão (useSubmitX via useMutation), e a mesma forma de tabela de invalidação de query keys da Seção 3 — adaptada ao schema de cada agente, nunca redesenhada. A Seção 2 deste documento é, portanto, o template arquitetural dos dois próximos incrementos, não uma decisão isolada para Project Status.

02

Architecture Review

Onde vive a açãoBotão "Nova Análise" no WorkspaceHeader, ao lado de "Atualizar". Abre um Dialog — não uma rota nova, não um formulário inline empurrando as 8 seções existentes para baixo.
Por que Dialog (justificativa de UX)components/ui/dialog.tsx já existe e já é usado neste mesmo Workspace (drill-down do Histórico completo, TIP-004) — mesma linguagem visual, mesmo componente, zero padrão de interação novo. Um modal contém a ação sem perturbar o layout de leitura já estabelecido, e é naturalmente dispensável — apropriado para uma ação pontual, não uma navegação.
"Selecionar Project Status" (Passo 3)Sem seletor de agente — só há 1 opção real. O título do modal já comunica isso ("Nova Análise — Status do Projeto"), rotulado para o usuário, nunca o nome técnico project_status. Um seletor com 1 opção seria ruído; a Central de Agentes (múltiplos tipos) fica para uma Feature futura, quando houver mais de 1 opção real a mostrar.
BFF novo — proxy finoweb/app/api/bff/workspace/[projectName]/analyze/status/route.ts. Recebe {"{"}project_context{"}"} do cliente, adiciona project_name a partir do segmento de rota (já decodificado, mesmo padrão do TIP-004/TIP-004A), encaminha para POST /api/projects/analyze. Nenhum endpoint novo no backend, nenhuma mudança de schema.
Timeout do BFF60s — mesmo valor já real, usado pelo cliente httpx em demo/seed_demo_data.py, não um número novo inventado. A chamada ao LLM não tem timeout próprio no backend (SDK padrão); o BFF precisa do seu.
Erro — reuso, não novo tipoparseWorkspaceResponse/WorkspaceFetchError (já existentes, lib/hooks/workspace-fetch-error.ts) reaproveitados sem alteração — a nova rota BFF retorna o mesmo formato {"{"}error, detail{"}"} das rotas já existentes.
Submissão — mutation, não queryNovo hook useSubmitProjectStatus(projectName), usando useMutation do TanStack Query — mesma biblioteca já em uso, API já disponível, nenhuma dependência nova.
PersistênciaJá automática — POST /api/projects/analyze já chama repository.save_analysis(...) antes de responder. Nenhuma mudança de backend.
Este é o padrão de referência, não um caso único. Cada linha desta tabela foi escolhida para generalizar: o BFF novo segue .../analyze/{kind}/route.ts (aqui, status; os próximos incrementos usam risk e meeting no mesmo lugar), o hook segue useSubmit{Kind}, e o corpo do Dialog troca apenas o campo de contexto e o schema de resposta renderizado — o mecanismo (abertura, loading, sucesso/erro, preservação de texto, invalidação) não muda. risk_review e meeting_intelligence reaproveitam esta seção por referência quando forem especificados; não deverão reabrir estas decisões.
03

Query Keys e Estratégia de Invalidação

Mapeamento exato de quais painéis um resultado de Project Status realmente afeta — nem mais, nem menos.

Query keyPainel(is) afetado(s)Por quê
["workspace-summary", projectName]Cabeçalho (badge de saúde), Executive Summary (contagens)total_analyses incrementa, latest_health_status pode mudar
["workspace-latest", projectName, "status"]Executive Summary (achados-chave), Recomendaçõesé literalmente a análise que acabou de rodar
["workspace-timeline", projectName, ...]Intelligence Timeline, Histórico completonova entrada na lista; invalidação por prefixo cobre qualquer limit/offset já em cache
["portfolio-summary"]Dashboard inteiro (KPIs, tabela, distribuição de saúde, concentração de risco)os 4 widgets do Dashboard leem essa única query

Deliberadamente não invalidado: ["workspace-latest", projectName, "risk"] e ["workspace-latest", projectName, "meeting"] — Project Status não produz riscos nem ações; invalidar esses painéis seria refetch sem necessidade.

onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: ["workspace-summary", projectName] });
  queryClient.invalidateQueries({ queryKey: ["workspace-latest", projectName, "status"] });
  queryClient.invalidateQueries({ queryKey: ["workspace-timeline", projectName] }); // prefixo
  queryClient.invalidateQueries({ queryKey: ["portfolio-summary"] });
}

Por que isso satisfaz "imediatamente ou comprovadamente" para o Dashboard

Verificado em código: web/app/providers.tsx cria um único QueryClient, montado no RootLayout — compartilhado por Dashboard, Projetos e Workspace durante toda a sessão do navegador (mesma aba). invalidateQueries marca ["portfolio-summary"] como stale mesmo com o Dashboard desmontado; a próxima navegação client-side até lá já dispara o refetch automaticamente, sem o usuário precisar clicar "Atualizar".

Limite honesto: isso é "sem intervenção do usuário dentro da mesma sessão/aba", não sincronização em tempo real entre abas ou dispositivos diferentes — o que exigiria WebSocket ou polling, explicitamente proibido nesta Feature. Nenhuma alegação além do que foi verificado.

05

Escopo

  • Botão "Nova Análise" no WorkspaceHeader
  • Modal (Dialog) com Textarea de contexto, validação de 10–20.000 caracteres
  • 1 rota BFF nova, proxy fino, para POST /api/projects/analyze
  • Estados de loading, sucesso (toast) e erro (inline, sem perder o texto digitado)
  • Invalidação das 4 query keys mapeadas na Seção 3
  • Timeout de 60s no BFF, com mensagem honesta em caso de estouro
06

Fora de Escopo (explícito)

  • risk_review e meeting_intelligence — extensões do mesmo padrão desta FS (Seção 2), incrementos futuros, não reinvenções
  • Central de Agentes ou qualquer seletor de tipo de análise
  • Novo item no menu principal ou nova rota de navegação
  • Chat, IA conversacional, síntese entre agentes
  • Qualquer mudança de schema, contrato, ou endpoint de backend
  • WebSocket, polling, ou qualquer arquitetura de tempo real
  • Score de confiança, previsão, ou qualquer interpretação que o backend não forneça
  • Editar ou corrigir uma análise já enviada
07

User Stories

Como Diretor de PMO dentro do Workspace de um projeto, aciono "Nova Análise" e rodo uma análise de status real, sem sair da tela.

Critério: modal abre sobre o Workspace, título "Nova Análise — Status do Projeto".

Como usuário enviando um contexto muito curto ou muito longo, sou impedido de enviar antes mesmo de tentar.

Critério: botão "Executar Análise" desabilitado fora de 10–20.000 caracteres, com contador visível.

Como usuário cuja submissão falha, não perco o texto que escrevi.

Critério: em erro, o modal permanece aberto com o texto intacto; usuário pode tentar de novo sem redigitar.

Como usuário que acabou de rodar uma análise, vejo o resultado no Workspace imediatamente, e no Dashboard na próxima vez que o abro — sem clicar em nada manualmente.

Critério: as 4 invalidações da Seção 3 disparam automaticamente ao sucesso.

NFRs Aplicáveis

  • Nenhuma regressão nos 92 testes de backend — nenhum arquivo de backend é tocado
  • Nenhuma regressão nos testes de frontend já existentes
  • Timeout de 60s no BFF, nunca uma espera indefinida sem feedback
  • Contraste WCAG AA no modal e nos novos estados, em ambos os temas
  • Responsividade nos 3 breakpoints já estabelecidos
  • Rate limit (429) do backend propagado com mensagem específica, não genérica
09

Critérios de Aceite

  • "Nova Análise" abre um modal sobre o Workspace atual — nenhuma navegação, nenhuma rota nova
  • Envio bloqueado fora de 10–20.000 caracteres, com feedback visível antes do clique
  • Estado de carregamento visível durante a chamada; duplo envio impossível
  • Sucesso: modal fecha, toast confirma, nenhum dado inventado é exibido
  • Erro: modal permanece aberto, mensagem específica por tipo (validação/rate limit/timeout/erro genérico), texto do usuário preservado
  • health_status, key_findings, recommendations aparecem nos painéis já existentes do Workspace sem reload manual
  • Dashboard reflete o novo resultado na próxima navegação client-side até ele, sem clique em "Atualizar"
  • Zero linha nova em src/database/, src/api/, src/services/, nos 3 agentes, ou no schema de analysis_records
  • Suíte completa (backend + frontend + E2E) passa sem alteração de contrato de backend

Estratégia de Testes

CamadaO que cobrir
BackendNenhum teste novo necessário — nenhum arquivo de backend muda. Suíte existente (92 testes) roda como regressão.
BFF (unit)Rota nova: injeta project_name da URL no corpo, propaga erro (incluindo 429), timeout de 60s, X-API-Key nunca vazado — mesmo padrão das rotas BFF já existentes.
Hook (unit)useSubmitProjectStatus: chama a rota BFF correta, dispara as 4 invalidações exatas da Seção 3 em sucesso, nenhuma invalidação em erro.
Componentes (unit)Modal: validação de tamanho, estado de loading, sucesso fecha + toast, erro mantém aberto com texto preservado, mensagem de erro específica por tipo.
E2E (Playwright, 3 breakpoints)Fluxo completo: abrir projeto → "Nova Análise" → preencher → executar → ver toast → ver painéis do Workspace atualizados sem reload → navegar ao Dashboard → ver refletido. Cenário de erro (rate limit simulado no mock). Mock de backend estendido com POST /api/projects/analyze.
Backend real, sem mockPelo menos uma submissão real contra o backend real (Demo Mode), confirmando persistência e reflexo no produto — mesma disciplina já aplicada no TIP-004/TIP-004A.
11

Riscos Específicos

  • Timeout de 60s pode ser insuficiente em picos de latência do LLM — ajuste de configuração, não de arquitetura, se necessário no futuro
  • Invalidação incompleta ou excessiva — a Seção 3 já enumera exatamente o necessário; teste E2E cobrindo Workspace e Dashboard juntos evita regressão silenciosa
  • Perda de contexto digitado em erro — requisito explícito de UX (Seção 4); testar especificamente que o texto sobrevive a um erro simulado
12

Checklist de Segurança (ADR-011)

Aplica-se parcialmente — a nova rota BFF segue o mesmo padrão de credencial já auditado (TIP-004): API_KEY só no servidor, nunca ecoado ao navegador. Nenhuma rota de sessão/autenticação nova. O texto submetido pelo usuário (project_context) é o mesmo tipo de conteúdo que já trafega via API hoje — nenhum dado sensível novo introduzido. Checklist completo aplicado de fato na escrita do TIP-005.

13

Rollback

Reverter o commit desta feature remove o botão "Nova Análise", o modal, a rota BFF, e o hook de submissão. O Workspace volta a ser somente-leitura para Project Status, exatamente como está hoje. Nenhuma migração de dado, nenhuma mudança de schema.
