O Método — reutilizável para toda Feature futura

5 perguntas, nesta ordem, antes de qualquer tela, componente ou arquitetura:

  1. Quem inicia a jornada?
  2. Qual problema está tentando resolver?
  3. Quais passos executa?
  4. Quais informações precisa?
  5. Qual decisão consegue tomar no final?

Aplicação à FS-005

Quem iniciaUm Diretor de PMO já dentro do Workspace de um projeto real (chegou lá via Dashboard ou Projetos, TIP-004A)
Qual problemaPrecisa de uma leitura de IA sobre o projeto (status, riscos, ou uma reunião) para decidir o próximo passo — e hoje não tem como pedir essa leitura pela interface, só por chamada de API
Quais passosOs 10 passos da Seção 3
Quais informações precisaJá tem o projeto selecionado (contexto automático); precisa fornecer o texto de entrada (transcript de reunião ou contexto do projeto) — nada além do que os 3 endpoints já pedem hoje
Qual decisão consegue tomar no finalEscalar um risco, agir sobre uma recomendação, ou confirmar que o projeto está saudável — decisão real, apoiada em achados, riscos e recomendações gerados pela IA e já persistidos, sem depender de outra ferramenta
02

Nível de Maturidade

A FS-005 evolui exatamente um nível — Nível 2, Analisar — sem tocar nos demais.

Nível 1 — Observar

O usuário entende seu portfólio

  • Dashboard real
  • Projetos real
  • Workspace real

Nível 2 — Analisar FS-005

O usuário usa IA para compreender seus projetos

  • Project Status backend real, sem UI
  • Risk Review backend real, sem UI
  • Meeting Intelligence backend real, sem UI

Nível 3 — Decidir

O usuário transforma análises em decisões

  • Plano de Ação futuro
  • Recomendações futuro
  • Central de Riscos futuro
  • Timeline futuro

Nível 4 — Governar

O usuário governa seu portfólio

  • Analytics futuro
  • Integrações futuro
  • Aprovações futuro
  • Configurações futuro
A FS-005 fecha o Nível 2 inteiro com os 3 agentes já reais — não abre o Nível 3. "Recomendações" e "Central de Riscos" já aparecem como painéis de leitura dentro do Workspace (TIP-004); virarem um módulo próprio do Nível 3 é decisão de uma Feature futura, não desta.
03

Os 10 Passos — verificados contra o código real

JÁ REAL só falta ligar PARCIAL parte existe, parte falta NÃO EXISTE a construir nesta Feature
1 · Escolha um projeto

Entregue no TIP-004A. project_name já é contexto automático dentro do Workspace.

2 · Escolha um tipo de análise

Nenhum seletor existe. Candidato: Tabs (já no design system, hoje decorativo em style-guide/page.tsx), com os 3 agentes reais como opções.

3 · Forneça o contexto

Endpoints já validam (10–20.000 caracteres, rejeita string vazia). Candidato: Textarea já existente, mesmo limite espelhado no cliente.

4 · Execute a IA

Os 3 POST /api/{"{"}meetings,risks,projects{"}"}/analyze já chamam o LLM real e retornam o resultado completo. Zero mudança de contrato.

5 · Acompanhe a execução

Nenhum padrão de loading para submissão existe (só para leitura). Sem timeout no backend para a chamada ao LLM — a UI precisa do seu próprio limite. Toaster (sonner) já montado, hoje ocioso.

6 · Visualize claramente o resultado

A resposta do POST já contém o resultado inteiro — pode ser mostrado no instante em que chega, sem esperar uma segunda leitura.

7 · Compreenda os riscos

O painel Riscos do Workspace (TIP-004) já existe e já inclui a Matriz de Riscos derivada (probabilidade × impacto) — real, testado, em produção. Falta só ele refletir uma análise recém-executada sem reload manual (mesma invalidação do Passo 9).

8 · Compreenda as recomendações

O painel Recomendações do Workspace já existe e já lê recommendations[] real. Mesmo gap: falta a atualização automática pós-submissão.

Achado adicional real: meeting_intelligence.summary e meeting_intelligence.issues existem no payload persistido mas nenhum painel os exibe hoje — gap real a fechar nesta Feature, não um novo dado, apenas um campo já real sem lugar na tela.

9 · Impacto imediato no Workspace

queryClient.invalidateQueries (já em uso) sobre as chaves já existentes (workspace-latest, workspace-timeline, workspace-summary) resolve isso sem nova arquitetura de cache.

10 · Impacto imediato no Dashboard

Verificado: app/providers.tsx cria um único QueryClient no RootLayout — o mesmo cache é compartilhado entre Dashboard, Projetos e Workspace durante toda a sessão. Invalidar ["portfolio-summary"] funciona mesmo com o Dashboard desmontado — a próxima montagem já busca o dado atualizado.

05

Síntese

Os 10 passos são implementáveis com as capacidades atuais. Nenhum depende de backend futuro, agente novo, ou mudança de contrato/schema. O que falta é inteiramente do lado do cliente: seletor de agente, formulário de contexto, estados de loading/sucesso/erro, e invalidação de cache coordenada entre Workspace e Dashboard — mais fechar 1 gap real de leitura (summary/issues).

Ao final da Sprint, um Diretor de PMO abre o navegador, escolhe um projeto real, roda uma análise real, acompanha a execução, lê o resultado completo (riscos com matriz, recomendações, achados, decisões, ações), vê tudo refletido no Workspace e no Dashboard sem reload manual — e toma uma decisão real. Isso é demonstrável para um cliente real, não infraestrutura.
