ADR-V2-001

Evolução incremental por releases, não reescrita total

Contexto
O RC-1 carece de fundação Enterprise, mas está funcional, testado e sem defeito conhecido.
Problema
Como evoluir para Enterprise sem descartar 8 Capabilities funcionais.
Alternativas
(a) Reescrever a plataforma do zero; (b) Fracionar em releases incrementais preservando o RC-1.
Decisão
(b) — releases incrementais 0.1 a 0.5, cada uma verificável isoladamente.
Justificativa
Reescrita total descartaria valor testado sem necessidade técnica real.
Consequências
Entrega mais lenta que "big bang", mas cada incremento é demonstrável.
Impacto futuro
Toda proposta de nova release deve se encaixar nesse mesmo padrão de fracionamento.
ADR-V2-002

Projeto como entidade real desde o Release 0.1

Contexto
project_name é hoje um campo de texto livre em analysis_records.
Problema
Impossível construir hierarquia Enterprise sobre um campo de texto.
Alternativas
(a) Manter texto livre com metadado externo; (b) Promover Projeto a entidade relacional já no Release 0.1.
Decisão
(b).
Justificativa
É o pré-requisito de tudo mais na hierarquia -- adiar geraria retrabalho em toda release seguinte.
Consequências
Exige a estratégia de migração completa já no Release 0.1.
Impacto futuro
Toda nova entidade nativa (Portfólio, Programa) segue o mesmo padrão relacional desde a concepção.
ADR-V2-003

Um projeto por project_name distinto na migração, sem fusão automática

Contexto
Nomes de projeto no RC-1 podem ter variações textuais do mesmo projeto real.
Problema
Como migrar sem perder dado nem fundir projetos diferentes por engano.
Alternativas
(a) Fusão automática por similaridade textual; (b) Um projeto por nome distinto, reconciliação manual depois.
Decisão
(b) — decisão explícita do Founder.
Justificativa
Consistente com a rejeição de fusão por similaridade já estabelecida em Organizational Intelligence (RC-1).
Consequências
Pode gerar duplicatas pós-migração; mitigado por reconciliação administrativa futura.
Impacto futuro
Nenhuma migração futura de dado nativo pode usar fusão automática sem nova decisão explícita.
ADR-V2-004

4 papéis no Release 0.1, não os 14 do RBAC de referência

Contexto
O RBAC de referência tem 14 papéis; implementar todos de uma vez seria complexidade não validada.
Problema
Como validar o motor de permissões sem sobre-engenhar a primeira release.
Alternativas
(a) 14 papéis de uma vez; (b) 4 papéis essenciais primeiro.
Decisão
(b) — Organization Admin, PMO, Project Manager, Viewer.
Justificativa
Cobrem os 3 escopos de acesso já conceituais no RC-1 mais um nível read-only que valida negação de permissão.
Consequências
Atores externos e papéis granulares ficam para o Release 0.2.
Impacto futuro
Expansão de RBAC é sempre incremental, nunca "big bang" de todos os 14 papéis de uma vez.
ADR-V2-005

Document e Process Intelligence como camadas de referência, não GED/BPM nativos

Contexto
Empresas já usam SharePoint/Drive/GED e Bizagi/Camunda para essas funções.
Problema
Evitar duplicar investimento já feito pelo cliente e desviar o diferencial da STRATECH.
Alternativas
(a) Construir GED e motor BPMN próprios; (b) Referenciar e interpretar, nunca substituir.
Decisão
(b) — princípio de não competição.
Justificativa
(a) duplicaria investimento e desviaria o foco de inteligência/governança para execução genérica.
Consequências
Valor desses dois domínios depende da qualidade da integração com sistemas externos.
Impacto futuro
Toda proposta de "gestão documental completa" ou "modelador de processo próprio" deve ser rejeitada por padrão, salvo nova decisão explícita do Founder.
ADR-V2-006

Documentação da V2 na mesma branch, diretório separado, por ora

Contexto
Ambiente de execução restrito a uma branch salvo autorização explícita; Baseline Configuration do RC-1 exige que seus artefatos nunca sejam alterados diretamente.
Problema
Isolar a V2 do RC-1 sem mecanismo de branch ainda decidido.
Alternativas
(a) Nova branch já para documentação; (b) Mesma branch, diretório docs/product/stratech-v2/ isolado.
Decisão
(b) — aprovado explicitamente pelo Founder.
Justificativa
Documentação pura não altera nenhum arquivo do RC-1 -- risco mínimo.
Consequências
Estratégia de branch revisada junto com o Founder no início do Release 0.1 (código).
Impacto futuro
Nenhum arquivo de código da V2 deve ser commitado nesta branch sem essa revisão conjunta ocorrer primeiro.
ADR-V2-007

Toda ação crítica de IA exige validação humana por padrão

Contexto
RC-1 já proibiu interpretação especulativa de IA; V2 introduz orquestração que pode agir em sistemas externos.
Problema
Confiança reportada por um modelo não é garantia de correção; ação externa tem custo de reversão maior que uma recomendação exibida.
Alternativas
(a) Automação plena para aceleradores de alta confiança; (b) Validação humana obrigatória para toda ação crítica.
Decisão
(b).
Justificativa
Consistente com o princípio permanente Human Accountability (Blueprint Seção 3).
Consequências
Reduz autonomia percebida da IA no curto prazo -- aceito como princípio permanente, não limitação temporária.
Impacto futuro
Nenhum Acelerador futuro pode executar ação em sistema externo sem aprovação humana registrada, independentemente do nível de confiança reportado.
