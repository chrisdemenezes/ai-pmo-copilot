Product Vision

Qual problema o produto resolve

PMOs e gerentes de projeto gastam tempo desproporcional convertendo informação não-estruturada (transcrições de reunião, narrativas de risco, atualizações de status em prosa) em artefatos de governança estruturados — atas, registros de risco, relatórios executivos. O AI PMO Copilot automatiza essa conversão: recebe texto livre, devolve estrutura (decisões, ações, riscos classificados, status de saúde) através de agentes de IA especializados, já validados em produção.

O que o produto NÃO pretende resolver

  • Não é um sistema de registro de tarefas — não há quadro Kanban, sprint board, ou rastreamento de tempo. Ações extraídas de uma análise não viram itens acionáveis e editáveis (hoje ficam dentro do JSON da análise, imutável — ver Domain Model).
  • Não é um chatbot genérico — mesmo quando o Chat (Release 0.4) existir, será escopado a contexto de projeto, não conversa aberta.
  • Não substitui o julgamento do PM/PMO — produz insight estruturado a partir do que foi fornecido; não decide, não prioriza, não é fonte de verdade sobre o que realmente aconteceu no projeto.
  • Não é (ainda) um sistema de gestão documental — Document Intelligence (Release 0.5) está bloqueado, sem API de armazenamento hoje.

Público-alvo

Confirmado em docs/product/03-user-personas.md, sem alteração: Project Manager, PMO Analyst, IT Manager, Executive Sponsor. O RFC-001 já desenha o Dashboard (Release 0.2/0.3) primariamente para o Executive Sponsor e o PMO Analyst — as personas que precisam de agregação antes de detalhe.

Diferenciais competitivos

  • IA nativa sobre entrada não-estruturada — não é um formulário com um botão "resumir com IA" colado por cima de um sistema de registro tradicional.
  • Rollup de portfólio sem agregação manual — GET /api/portfolio/summary computa em tempo real a partir de todas as análises, não depende de alguém preencher uma planilha.
  • Histórico imutável e auditável por design — uma análise, uma vez criada, não pode ser silenciosamente editada (sem PUT/PATCH/DELETE no schema real).
  • Disciplina de engenharia verificável — 98%+ de cobertura de teste no backend, toda mudança rastreável a uma evidence entry com commit e execução de CI real, não alegação.

Posicionamento competitivo

ConcorrenteO que fazem bemOnde o AI PMO Copilot é diferente
Jira / ClickUp / MondaySistemas de registro — quadros, tarefas, prazos, workflow de execuçãoNão competem em execução; o Copilot é uma camada de inteligência sobre o que já aconteceu, não onde o trabalho é gerenciado dia a dia
Microsoft ProjectPlanejamento e cronograma (Gantt, dependências, alocação de recursos)Copilot não planeja cronograma — analisa o que foi relatado sobre um projeto que já existe em outro lugar

Implicação arquitetural direta: o Copilot deve permanecer uma camada complementar, com integrações para essas ferramentas (Release 0.6) em vez de tentar substituí-las — reforçado pela entidade Integration no Domain Model abaixo.

Event Model

Não existe barramento de eventos, fila ou pub/sub hoje. Confirmado: requirements.txt não tem Celery, Redis, Kafka ou qualquer dependência de mensageria. Todo "evento" abaixo é, na realidade atual, um efeito colateral síncrono dentro de uma única requisição HTTP request/response.

Fluxo real (síncrono) — o que de fato acontece hoje

sequenceDiagram
    participant U as Usuário
    participant BFF as Next.js BFF (planejado, Sprint 2)
    participant API as FastAPI (real)
    participant Agent as Agent (real)
    participant LLM as Anthropic (real)
    participant DB as Postgres/SQLite (real)

    U->>BFF: envia formulário de análise
    BFF->>API: POST /api/meetings/analyze (X-API-Key)
    API->>Agent: MeetingIntelligenceAgent.analyze()
    Agent->>LLM: generate(prompt)
    LLM-->>Agent: raw_output
    Agent->>Agent: parse_structured_output()
    API->>DB: save_analysis()
    DB-->>API: id
    API-->>BFF: 200 {agent, project_name, model_output}
    BFF-->>U: resultado renderizado
    Note over U,DB: Uma única requisição HTTP.
Nenhum passo é assíncrono ou desacoplado.

Todos os participantes marcados "real" existem em produção hoje; o BFF é o Sprint 2 aprovado no RFC-001, ainda não implementado.

Cadeia de eventos — visão de longo prazo (exemplo do usuário)

flowchart LR
    A[Meeting Uploaded] --> B[Meeting Analyzed]
    B --> C[Analysis Stored]
    C --> D[Project Summary Updated]
    D --> E[Portfolio Updated]
    E --> F[Notification Sent]
    F --> G[Dashboard Refreshed]

Esta cadeia é 100% aspiracional. Mapeamento contra a realidade:

Evento (visão)StatusRealidade hoje
Meeting UploadedBloqueadoNão existe upload de arquivo — só colagem de texto num formulário
Meeting AnalyzedAtualPOST /api/meetings/analyze, síncrono
Analysis StoredAtualAnalysisRepository.save_analysis(), mesma transação
Project Summary UpdatedNão é um eventoGET /api/projects/{name}/summary recomputa sob demanda a cada chamada — nada é "atualizado" e persistido, é sempre calculado na hora
Portfolio UpdatedNão é um eventoMesma situação — GET /api/portfolio/summary é computado, não materializado
Notification SentBloqueadoNenhuma infraestrutura de notificação existe
Dashboard RefreshedPlanejadoPolling client-side (RFC-001 §7), não um push de servidor
Implicação arquitetural: introduzir um barramento de eventos real só se justifica quando pelo menos um consumidor assíncrono genuíno existir (ex: Notification, ou um rollup materializado para performance em escala). Construir a infraestrutura antes disso repetiria o erro já evitado em SSE/WebSocket no RFC-001 (Decisão D3) — arquitetura paralela sem consumidor real.

Architecture Decision Records

Consolida as decisões arquiteturais já tomadas ao longo do backend e do RFC-001 no formato exigido, com o campo Rollback adicional.

ADR-001 — Persistência single-table sem normalização de domínio
Contexto
MVP precisava persistir resultados de análise rapidamente, sem um domain model maduro definido.
Problema
Como armazenar dado estruturado de IA sem travar o MVP num schema relacional prematuro?
Alternativas
Normalizar Risk/ActionItem/Decision como tabelas desde o início (mais trabalho, especulativo); usar um banco de documentos (NoSQL) (mudaria toda a stack já validada com SQLAlchemy/Alembic).
Decisão
Uma tabela (analysis_records) com payload JSON — o schema estruturado vive dentro do JSON, validado por Pydantic na borda da API, não pelo banco.
Consequências
Risk/ActionItem/Decision não são consultáveis individualmente hoje (Domain Model §2) — aceito conscientemente até uma feature real precisar disso.
Rollback
Migração de dados para extrair entidades do JSON para tabelas próprias — via nova migração Alembic, sem perda de dado (o JSON permanece como fonte).
ADR-002 — Backend-for-Frontend (BFF) para proteger API Key compartilhada
Contexto
Backend usa uma única API_KEY; não há conta de usuário nem múltiplas chaves.
Problema
Como o frontend chama a API sem expor a chave real a qualquer navegador?
Alternativas
Guardar a chave no navegador (inseguro); pedir chave individual por usuário (backend não suporta).
Decisão
Route Handlers do Next.js seguram a chave server-side, repassam ao FastAPI — ver RFC-001 Decisão D1.
Consequências
Um hop de latência extra; habilita a evolução de auth (ADR-007) sem quebra.
Rollback
Reverter exigiria expor a chave ao cliente — inaceitável; não há rollback real, só substituição por um modelo de auth mais maduro (RFC-001 §9).
ADR-003 — Processamento síncrono, sem mensageria
Contexto
As 3 análises de IA são requisições únicas, sem necessidade aparente de processamento em background no MVP.
Problema
Vale a pena introduzir fila/mensageria antes de haver um consumidor assíncrono real?
Alternativas
Celery+Redis desde o início (complexidade não paga pelo uso atual); SSE/WebSocket "para o futuro" (RFC-001 Decisão D3, rejeitado pela mesma razão).
Decisão
Manter síncrono até um caso de uso real exigir o contrário — ver Event Model §3.
Consequências
Notification, Job e parte de Workflow (Domain Model/State Machines) ficam bloqueados até essa decisão ser revisitada.
Rollback
Não é uma decisão reversível no sentido tradicional — é uma decisão de "ainda não", revisitável a qualquer momento sem custo de migração (nenhuma infraestrutura foi construída para desfazer).
ADR-004 — Radix UI vendorizado, sem CLI do shadcn/ui
Contexto
ui.shadcn.com bloqueado pela política de rede do ambiente de desenvolvimento.
Problema
Como obter primitivos acessíveis sem a ferramenta de CLI padrão do ecossistema?
Alternativas
MUI/Mantine/Chakra (linguagem visual própria); esperar liberação de rede (sem prazo).
Decisão
Instalar @radix-ui/react-* via npm, escrever wrappers à mão — RFC-001 Decisão D2.
Consequências
Atualizações de acessibilidade exigem merge manual.
Rollback
Migrar para a CLI do shadcn/ui a qualquer momento, se a rede for liberada — os componentes já seguem o mesmo padrão de código, troca de baixo risco.
ADR-005 — Rate limiting in-process, não distribuído
Contexto
US-SEC-03 precisava de rate limiting funcional rapidamente, com instância única em produção.
Problema
Implementar um limitador simples agora, ou já desenhar para múltiplas instâncias?
Alternativas
Redis desde o início (infraestrutura extra não paga pelo estágio atual).
Decisão
Janela deslizante em memória, por processo.
Consequências
Risco real classificado Alto (§12): se o backend rodar em múltiplas instâncias, o limite efetivo vira N× o configurado.
Rollback
Substituir o backend do RateLimiter por uma implementação Redis, mesma interface pública — mudança contida a um arquivo.
ADR-006 — Alembic como mecanismo de migração de schema
Contexto
Precisava de um caminho de evolução de schema para bancos reais (Postgres), além de create_all() usado em testes.
Problema
Como evoluir o schema em produção sem perder dado?
Alternativas
Só create_all() (não migra, recria); ferramenta proprietária de nuvem (acopla a um provedor).
Decisão
Alembic, validado contra Postgres real nesta sessão (Evidence Entry 012).
Consequências
Toda mudança de modelo exige gerar e revisar uma migração — disciplina já seguida desde a introdução.
Rollback
alembic downgrade nativo, já parte da ferramenta.
ADR-007 — Dois níveis de autenticação (sessão de workspace vs. credencial de backend)
Contexto
A Revisão 1 do RFC-001 tratava "sessão" e "chave de API" como um único conceito, gerando uma leitura incorreta do erro 401.
Problema
Como modelar autenticação sem confundir "quem pode usar a ferramenta" com "a chamada ao backend está autorizada"?
Alternativas
Manter um único conceito (o erro da Revisão 1).
Decisão
Nível 1 (navegador↔BFF, sessão de workspace) separado de Nível 2 (BFF↔FastAPI, API_KEY) — RFC-001 Decisão D4.
Consequências
Habilita a migração para JWT/OIDC (RFC-001 §9) como mudança contida ao BFF.
Rollback
N/A — é a correção de um erro de modelagem anterior, não uma decisão a desfazer.
ADR-008 — web/ como árvore de frontend paralela a src/
Contexto
frontend/ já existia no repositório, contendo só documentação de visão.
Problema
Onde colocar o código real do frontend sem misturar com docs aspiracionais?
Alternativas
Dentro de frontend/ (misturaria app real com 3 arquivos de visão).
Decisão
Novo diretório web/, frontend/ intocado — mesmo tratamento dado a backend/ quando seu Dockerfile quebrado foi removido.
Consequências
docs/development/01-project-structure.md atualizado para declarar as duas árvores oficiais.
Rollback
Mover web/ para dentro de frontend/ é uma operação de arquivo simples, se necessário — baixo custo.

Critérios para Início do Sprint 2

O projeto está pronto para iniciar a implementação do Frontend? SIM — com pendências não-bloqueantes registradas abaixo, nenhuma delas impede especificamente o Sprint 2 (BFF + Dashboard).

O raciocínio: nenhum dos dois riscos Críticos (backup, LGPD) ou Altos (rate limiter distribuído, versionamento de API, model versioning) bloqueia especificamente construir um Dashboard contra dado de desenvolvimento/mock — eles importam antes de dado real de produção existir, não antes da primeira tela. RFC-001 já está aprovado e detalhado em 3 revisões; este Product Blueprint fecha a lacuna de visão de produto multi-ano que faltava.

Pendências registradas — não bloqueiam o Sprint 2, mas precisam de dono e prazo

  • Confirmar que nenhum dado pessoal real (PII) será usado em qualquer piloto antes do gap de LGPD (§8/§12) ser formalmente endereçado.
  • Decidir Retry-After no 429 do backend (RFC-001 Pergunta em aberto).
  • Decidir integração do frontend ao docker-compose.yml (RFC-001 Pergunta em aberto).
  • Confirmar se o gate de contraste WCAG entra no CI do Sprint 2 ou fica para depois (RFC-001 Pergunta em aberto).
  • Agendar a introdução de versionamento de API (/v1/) antes que o frontend se torne um segundo consumidor real do contrato atual.
  • Agendar backup automatizado do banco antes de qualquer ambiente com dado real (não bloqueia dev/staging).

Nenhuma dessas pendências exige reabrir o RFC-001 ou este Blueprint — são itens de acompanhamento, não pré-condições.
