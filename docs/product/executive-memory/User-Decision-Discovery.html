User Decision Discovery — Executive Memory

Pergunta única autorizada pelo Founder para esta Discovery: "O que o executivo precisa lembrar para tomar melhores decisões?" Considerando explicitamente histórico de decisões, histórico de riscos, histórico de ações, evolução do contexto e aprendizado executivo — sem transformar a plataforma em um sistema de relatórios históricos. A Executive Memory deve existir para melhorar decisões futuras, não apenas para mostrar o passado. Nenhuma outra pergunta é respondida aqui. Nenhum código escrito.

01

System Integrity — novo critério permanente

Registrado oficialmente nesta etapa, por instrução do Founder: "Toda Capability deverá demonstrar explicitamente como melhora a plataforma como um sistema completo. Não apenas como um módulo isolado." Critério permanente, válido a partir desta fase para toda futura Capability -- não avaliado retroativamente nas 5 Capabilities já encerradas.

Aplicado a Executive Memory desde já: o achado da Seção 00 (já existe uma tela de navegação por histórico) torna a resposta natural, não forçada -- se Executive Memory fosse uma 6ª página isolada de "ver o passado", ela seria redundante com IntelligenceTimeline/AnalysisHistory e não melhoraria o sistema, apenas o duplicaria. A forma que melhora o sistema inteiro é diferente: um fato de memória (um delta real, computado e auditável) injetado como contexto adicional dentro das superfícies de decisão que já existem -- Executive Brief (Understanding), Executive Decision Card (Decision), Executive Portfolio Card (Prioritization) -- cada uma ficando um pouco mais profunda, nunca uma tela nova e desconectada. Essa é a diferença entre "acrescentar um módulo" e "melhorar o sistema completo": ver Seção 5 para o desenvolvimento completo desta hipótese.

02

Qual decisão o usuário deseja tomar?

A pergunta do Founder tem duas partes: o que lembrar (Seção 3 inventaria o que já existe disperso; Seção 4, o que falta) e como isso melhora decisões futuras, não apenas mostra o passado (Seção 5).

Isso desdobra a pergunta do Founder em sub-perguntas, todas verificáveis contra o código real:

  • O que já é lembrado hoje, mesmo que disperso -- que fatos históricos já existem em analysis_records, prontos para leitura, sem nenhuma leitura nova?
  • O que continua esquecido -- que comparação ou evolução o executivo precisaria hoje, mas que nenhuma leitura atual produz?
  • Onde a memória deve aparecer -- uma tela nova (arriscando duplicar IntelligenceTimeline/AnalysisHistory, achado da Seção 00) ou um enriquecimento das superfícies de decisão já existentes (System Integrity, Seção 1)?

O achado central: o dado bruto de risco e de ação já está, hoje, guardado para sempre -- mas nenhuma leitura hoje compara dois pontos no tempo. "Evolução do contexto" e "aprendizado executivo" são, literalmente, as únicas duas peças que não existem em nenhuma forma hoje.

04

Quais informações realmente faltam?

Honestidade sobre o que o schema e a arquitetura atuais não sustentam -- nenhuma lacuna abaixo deve ser preenchida por invenção da IA ou da interface.

  • Nenhuma leitura compara duas análises do mesmo tipo e projeto. Para dizer "este projeto piorou desde a última análise de Status", é preciso ler as 2 análises de status mais recentes (já possível via list_analyses(kind="status", limit=2)) e comparar health_status -- nenhum código faz isso hoje. É uma leitura nova, pequena, síncrona (coerente com ADR-003) -- não uma tabela nova.
  • Nenhum registro de que uma decisão foi seguida ou ignorada. Decision Center nunca escreve. "Aprendizado executivo" no sentido de "essa recomendação funcionou da última vez" exigiria um campo de outcome que não existe e que esta Discovery não propõe criar -- honestidade: sem essa escrita, "aprendizado executivo" só pode significar, por ora, "reconhecer um padrão real que já aconteceu" (ex.: "este tipo de risco já apareceu 3 vezes neste projeto"), nunca "esta decisão foi boa ou ruim".
  • Nenhum critério de relevância para "o que vale a pena lembrar". Nem todo delta histórico merece atenção -- mostrar toda mudança sem filtro contradiz o Princípio da Atenção. Que limiar torna uma mudança "digna de memória" é decisão da User Journey, não desta Discovery.
  • Nenhuma leitura hoje agrega padrões através do portfólio inteiro (ex.: "risco X já apareceu em 3 projetos diferentes este trimestre") -- isso se aproximaria de Organizational Intelligence (Capability futura, já citada em Portfolio Intelligence), não de Executive Memory, que é sobre a memória de um projeto/decisão por vez. Fronteira entre as duas, registrada aqui, a confirmar na User Journey.
05

Como a memória melhora decisões futuras (não é histórico)?

Resposta direta à instrução central do Founder: a diferença entre "mostrar o passado" e "melhorar a decisão de agora".

Um sistema de relatórios históricos responderia "o que aconteceu" -- exatamente o que IntelligenceTimeline/AnalysisHistory já fazem, sob demanda, quando o executivo escolhe olhar para trás.

Executive Memory, pela hipótese desta Discovery, nunca é uma tela para onde o executivo vai procurar o passado -- é um fato real, curto, que o passado entrega de volta ao executivo no momento em que ele já está decidindo algo, sem ele precisar pedir. Diferença concreta:

Superfície existenteContexto de memória que a aprofundaFonte real
Executive Brief (Status)"Este é o 3º status crítico consecutivo" / "Melhorou de Atenção para Saudável desde a última análise"Duas últimas análises kind="status" do projeto
Executive Decision Card (Decision Center)O mesmo delta, reaproveitado -- nunca recalculado por dois lugares diferentesA mesma leitura acima, consumida, nunca duplicada (Single Decision Source)
Executive Portfolio Card (camada de risco)"Este risco já foi identificado em N análises anteriores" -- contagem real de recorrêncialist_analyses(kind="risk") do projeto, contagem, não texto interpretado

Nenhuma dessas 3 linhas exige uma tela nova -- cada uma aprofunda uma superfície que já existe, coerente com Progressive Context (mais um nível de compreensão do mesmo objeto) e com System Integrity (o sistema inteiro fica mais informado, não apenas um módulo isolado). O fork sobre se, além disso, uma tela dedicada faz sentido fica registrado para a User Journey (Seção 8).

Nota de honestidade (mesmo padrão das Discoveries anteriores): Executive Memory não decide nem prevê nada -- ela relembra um fato real já registrado, no momento certo. O "Decidir" do Executive Loop continua do executivo; a memória só garante que ele decide sabendo o que já aconteceu antes, sem precisar ir buscar.

06

Como saberemos que a experiência ficou melhor?

Qualitativo, não uma métrica inventada: um executivo, ao ver uma decisão ou um risco, sabe se aquilo já aconteceu antes e como evoluiu, sem precisar abrir o Histórico completo para reconstruir isso manualmente. Validável por leitura direta do Founder e por observação qualitativa nas sessões de validação já em curso, mesma disciplina de TIP-005 a TIP-010.

Time To Decision / Time To Attention (princípios oficiais, adaptados): Executive Memory não cria um novo caminho de navegação -- ela reduz o tempo dentro de um caminho já existente (Executive Brief, Decision Center, Portfolio Intelligence), ao entregar de graça um fato que hoje exigiria abrir o Histórico completo e comparar manualmente.

System Integrity (Seção 1): validável objetivamente nas etapas seguintes por uma pergunta simples -- esta entrega tornou pelo menos uma Capability já existente mais profunda, sem introduzir uma tela isolada e desconectada? Se a resposta for não, a proposta não cumpriu o critério.
