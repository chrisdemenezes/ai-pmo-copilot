01

Contexto e Objetivo

Esta FS formaliza os 3 documentos já aprovados (Discovery, Journey, UX Flow). Executive Memory entrega, dentro das superfícies de decisão já existentes, o menor fato real necessário para o executivo lembrar de algo que já viveu antes -- nunca uma tela nova, nunca uma narrativa, nunca um recálculo do que outra Capability já produz.

Diferente de toda Capability anterior, Executive Memory não organiza um elo próprio do Executive Loop -- ela atravessa os elos já existentes (Compreensão, no Executive Brief; Decisão, no Executive Decision Card; Priorização, no Executive Portfolio Card), aprofundando cada um sem alterar sua natureza. É, por definição, transversal -- confirmado explicitamente na Seção 7.

02

Comportamento Executivo

Seção obrigatória em toda Feature Specification, por instrução do Founder: qual comportamento executivo esta Capability pretende incentivar?

CapabilityComportamento incentivado
Project Intelligence (Status)Acompanhar
Risk IntelligenceMitigar
Communication IntelligenceAlinhar
Action IntelligenceExecutar
Decision CenterDecidir
Portfolio IntelligencePriorizar
Executive MemoryLembrar

Natureza distinta: "Lembrar" não é um comportamento que o executivo pratica ao abrir uma tela -- é o que a própria plataforma faz por ele, silenciosamente, dentro dos comportamentos que já existem (Acompanhar, Decidir, Priorizar). Coerente com a Seção 1: Executive Memory nunca é o destino de uma jornada, só a torna mais completa.

03

Architecture Review (preliminar)

3.1 -- Workspace (Executive Brief): zero leitura nova

O Executive Brief já busca a análise de status mais recente do projeto (useWorkspaceLatestByKind(projectName, "status"), que internamente chama GET /api/bff/workspace/[projectName]/analyses?kind=status&limit=1). Calcular "Mudou" e "Persistiu" exige apenas aumentar o limit já existente (de 1 para um pequeno N, ex. 5) nessa mesma chamada -- mesma rota, mesmo parâmetro já suportado (AnalysisRepository.list_analyses já aceita limit), e comparar os health_status retornados no cliente. Zero rota nova, zero método novo de serviço -- dado real já existente, leitura estritamente necessária (Seção 7).

Da mesma forma, "Reapareceu" (risco) reaproveita a mesma rota genérica já usada pela AnalysisHistory/IntelligenceTimeline (GET /api/bff/workspace/[projectName]/analyses?kind=risk, sem a restrição de "só a mais recente" que list_latest_risks() aplica) -- os detalhes de cada análise (GET .../analyses/{"{id}"}) já existem, também sem mudança.

3.2 -- Achado explícito: Decision Center e Portfolio Intelligence operam em escala de portfólio

Para a Architecture Review resolver explicitamente, não decidido nesta FS: o Executive Decision Card e o Executive Portfolio Card iteram sobre todo o portfólio de uma vez -- diferente do Executive Brief, que já opera sobre um único projeto. Buscar 5 análises de status e todas as análises de risco de cada projeto do portfólio, individualmente, seria N+1 -- o mesmo problema que list_latest_risks() já resolveu uma vez para Decision Center (FS-008 §3.1). Resolver isso exigiria uma leitura agregada nova -- não estritamente necessária para o escopo desta entrega (Seção 7), portanto fora dela.

Duas saídas possíveis, nenhuma decidida aqui:
  1. Recomendada (escopo desta entrega): Executive Memory Insights aparecem primeiro só no Executive Brief (Workspace) -- zero leitura nova, entrega o objetivo central da Discovery/Journey ("o que o executivo precisa lembrar antes de decidir") no lugar onde a decisão de Status efetivamente acontece. Decision Center e Portfolio Intelligence ficam registrados como extensão futura, condicionada a uma leitura agregada nova.
  2. Escopo ampliado: introduzir a leitura agregada nova (ex.: list_recent_status_history(count=5), mesmo padrão de list_latest_risks()) já nesta entrega, para que Decision Center e Portfolio Intelligence também recebam Insights.
Recomendo a Opção 1, mesmo raciocínio já usado em toda Capability anterior desde Action Intelligence: entrega o valor central mais rápido, sem uma dependência técnica nova bloqueando a primeira entrega.

04

Silent Intelligence — diretriz derivada do Princípio da Atenção

Esclarecimento do Founder aplicado nesta etapa: Silent Intelligence não é um novo princípio permanente da plataforma -- é uma diretriz de UX derivada diretamente do Princípio da Atenção já consolidado (Decision Center, TIP-009 §08): "a ausência de informação também comunica."

Executive Memory é o primeiro caso concreto dessa aplicação, numa granularidade menor: o Princípio da Atenção já regia quando um card inteiro aparece ou não (Executive Decision Queue, Executive Portfolio View); Silent Intelligence é a mesma disciplina aplicada a um campo dentro de um card -- os 3 Memory Signals (UX Flow §2) só aparecem quando têm um fato genuíno a entregar (Seção 3.4); no silêncio, o campo simplesmente não existe, sem rastro, sem espaço reservado, sem estado vazio a preencher. Nenhum princípio novo é registrado por esta seção.

05

Executive Memory Hierarchy

Diretriz do Founder: avaliar a hierarquia conceitual de importância entre os 3 Memory Signals, para orientar evoluções futuras -- não altera a UX Flow já aprovada, os 3 continuam sendo exibidos exatamente como especificado quando aplicáveis, e esta avaliação não constitui um princípio ou uma regra de arquitetura nova.

1ºPersistiu
2ºReapareceu
3ºMudou

Justificativa da ordem, verificável contra o significado real de cada signal: "Persistiu" indica uma situação de atenção real que não se resolveu apesar do tempo passado -- o sinal mais grave, por natureza. "Reapareceu" indica um padrão que retorna, sugerindo que uma mitigação anterior pode não ter sido suficiente -- mais grave que uma mudança pontual, porque revela recorrência. "Mudou" é o mais neutro dos três -- pode ser uma piora ou uma melhora, e por si só não indica um problema não resolvido.

Por que isso importa agora, mesmo sem mudar UX: hoje, cada Insight aparece sobre um campo diferente do card (Mudou/Persistiu sobre Status; Reapareceu sobre Risco) -- nunca competem entre si por espaço. Esta hierarquia fica registrada para o dia em que essa suposição deixar de valer -- ex.: um limite de espaço exigir escolher só 1 Insight por card. Nenhuma dessas situações existe nesta entrega -- reafirma o Founder: avaliação, não implementação.

06

Executive Memory Insight — UI Pattern

A partir desta FS, "Executive Memory Insight" é tratado como um Executive UI Pattern -- um padrão de composição visual reutilizável (chip discreto, tom neutro, texto curto e verbatim), nunca um princípio ou um conceito de arquitetura. Mesma categoria de outros padrões visuais já emergentes na plataforma (ex.: a composição "Card > camada > sinal real > próximo movimento" do Executive Decision Card/Executive Portfolio Card) -- vocabulário visual comum, não regra de negócio nova. "Memory Signal" permanece como o termo técnico interno (nome de tipo, comentário de código); "Executive Memory Insight" é o nome do padrão de UI na documentação de produto e em qualquer texto voltado ao executivo.

12

User Stories

Como executivo, ao ler o Executive Brief de um projeto, sei imediatamente se aquele status já mudou, persiste, ou se um risco já reapareceu -- sem abrir o Histórico completo para descobrir.

Critério: o Insight aparece inline, ao lado do campo real, só quando há um fato genuíno a entregar.

Como executivo, quando não há nada digno de nota para lembrar, o Executive Brief se comporta exatamente como hoje -- nenhum espaço vazio, nenhum texto de preenchimento.

Critério: as 3 condições de silêncio (§3.4) suprimem o Insight sem deixar rastro visual.

Como executivo, nunca preciso clicar ou navegar para entender um Executive Memory Insight -- ele já chega pronto, junto com o resto do que eu já estava lendo.

Critério: nenhum Insight é interativo; nenhum gate de carregamento adicional (§3.4).

Critérios de Aceite

  • "Mudou" aparece quando e só quando o health_status muda entre as 2 análises de status mais recentes do projeto
  • "Persistiu" aparece quando e só quando o mesmo estado crítico/em atenção se repete por 2+ análises seguidas -- nunca para um estado saudável repetido
  • "Reapareceu" aparece quando e só quando um padrão de risco equivalente já foi identificado antes -- nunca na 1ª ocorrência
  • Nenhum dos 3 aparece na primeira análise daquele tipo para o projeto
  • Nenhum Insight é clicável, nem introduz um estado de carregamento próprio
  • Zero escrita -- nenhum botão de criar/editar/resolver relacionado a esta Feature
  • Testes: memory-insights.ts (unit), componente, integração no Executive Brief, E2E, backend como regressão (nenhuma mudança)
  • Verificação real contra Demo Mode, mesma disciplina das Features anteriores

Riscos Específicos

  • Definição de "risco equivalente" para "Reapareceu" -- comparar description verbatim seria frágil (mesma redação exata improvável entre 2 análises de IA); recomenda-se comparar por presença de risco na mesma categoria de probabilidade × impacto -- decisão final de implementação registrada no TIP, não nesta FS
  • Volume de análises por projeto ainda pequeno (dado fictício/piloto) -- mesmo gap já conhecido registrado em Capabilities anteriores, sem paginação necessária por ora
  • "Persistiu" e "Mudou" são mutuamente exclusivos por definição -- risco de bug se ambos aparecerem juntos; coberto explicitamente em teste unitário
