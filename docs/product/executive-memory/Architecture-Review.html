Executive Questions Framework aplicado — Executive Memory

Framework já registrado como princípio oficial na Architecture Review de Action Intelligence -- aplicado aqui, não redefinido.

1. Qual pergunta executiva ela responde?

"O que o executivo precisa lembrar antes de decidir?" (Discovery, meta-row) -- pergunta única, distinta de todas as outras 6 Capabilities já existentes (FS-010 §7/§19).

2. Qual decisão ela permite tomar?

Nenhuma decisão nova -- ela enriquece a decisão que o Executive Brief já permite tomar (Status), com um fato real que o executivo teria de reconstruir manualmente no Histórico completo.

3. Qual ação ela desencadeia?

Nenhuma -- Executive Memory nunca é interativa (UX Flow §5, FS-010 §3.4). A única ação possível continua sendo a que o card já desencadeava antes desta Feature existir.

4. Com que frequência um executivo utilizará este módulo?

Não aplicável como módulo -- Executive Memory não tem frequência de uso própria; ela aparece (ou não) toda vez que o Executive Brief já é lido, na mesma frequência dele.

5. Como ela conversa com os demais módulos da plataforma?

Lê dado já produzido por Project Intelligence (Status) e Risk Intelligence (Risco), nunca recalculando o que essas Capabilities já produzem -- ver Seção 4 sobre a fronteira de escala com Decision Center e Portfolio Intelligence.

02

O Executive Loop

Pergunta→ Compreensão→ Decisão→ Ação→ Retorno

Executive Memory não organiza nenhum elo próprio -- ela se anexa ao elo "Compreensão" (Executive Brief), tornando-o mais completo sem introduzir uma pausa, uma confirmação ou um passo novo. Se um Insight estiver em silêncio (Silent Intelligence, FS-010 §4), o Loop se comporta exatamente como se Executive Memory não existisse -- confirmado formalmente na Seção 6, pergunta 4.

03

Sobrecarga Cognitiva e a One Memory Insight Rule

Avaliação explícita exigida pelo Founder, respondendo as 3 perguntas diretamente.

3.1 -- Pode existir mais de um Executive Memory Insight simultaneamente no mesmo contexto?

Verificado contra a composição real (FS-010 §3.3-3.4): "Mudou" e "Persistiu" são mutuamente exclusivos por construção -- ambos derivam da mesma comparação de health_status entre as 2 análises mais recentes, e essa comparação só pode produzir um resultado por vez (mudou OU se manteve). "Reapareceu" deriva de um dado completamente diferente (histórico de risco), e nesta entrega (Executive Brief) aparece num campo distinto do de Status.

Resposta honesta: no escopo real desta entrega, não existe hoje nenhum contexto onde 2 candidatos a Insight competiriam pelo mesmo espaço -- a pergunta do Founder antecipa corretamente um risco que a arquitetura atual ainda não materializa, mas que se tornaria real no momento em que uma única superfície (ex.: um futuro card combinado Status+Risco) precisasse mostrar os dois ao mesmo tempo. A One Memory Insight Rule (Seção 3.3) é adotada como salvaguarda permanente para esse cenário, não como correção de um problema já existente.

3.2 -- Isso melhora ou reduz a clareza da decisão?

Mostrar mais de um Insight ao mesmo tempo reduziria a clareza -- fragmentaria a atenção do executivo entre 2+ fatos de memória, exatamente o oposto do objetivo desta Capability (UX Flow §1: lembranças, não relatórios). Um único Insight, selecionado pela gravidade real que ele representa (Seção 3.3), preserva o mesmo princípio já usado em toda a plataforma: mostrar só o que realmente merece atenção, nunca tudo de uma vez (Princípio da Atenção, aplicado aqui à granularidade de um único campo, mesma disciplina de Silent Intelligence, FS-010 §4).

3.3 -- Qual regra preserva melhor o menor contexto necessário? A One Memory Insight Rule, adotada

Adotada nesta Architecture Review, por instrução do Founder: cada superfície executiva apresenta, no máximo, um Executive Memory Insight por contexto analisado. Quando mais de um candidato for aplicável, a hierarquia de seleção (já registrada como avaliação conceitual em FS-010 §5, agora usada como regra de seleção concreta) decide:
1ºPersistiu
2ºReapareceu
3ºMudou
export function selectPrimaryInsight(
  candidates: Array<ExecutiveMemoryInsight | null>,
): ExecutiveMemoryInsight | null {
  const real = candidates.filter((c): c is ExecutiveMemoryInsight => c !== null);
  if (real.length === 0) return null; // Silent Intelligence -- nenhum candidato real
  const rank: Record<MemorySignalKind, number> = { persistiu: 0, reapareceu: 1, mudou: 2 };
  return real.slice().sort((a, b) => rank[a.kind] - rank[b.kind])[0];
}

Esta função é a mesma disciplina de buildExecutiveDecisionQueue()/ buildExecutivePortfolioView() -- uma única implementação da regra de seleção, nunca duplicada por superfície. Preserva Decision Context Preservation por construção: o executivo nunca recebe mais que um fato, mesmo que mais de um fosse tecnicamente verdadeiro.

04

Resolução do achado FS-010 §3.2 (escala de portfólio)

FS-010 registrou um fork explícito, não decidido, entre enriquecer só o Executive Brief (Opção 1) ou já estender a Decision Center/Portfolio Intelligence com uma leitura agregada nova (Opção 2). Esta Architecture Review resolve o fork.

Decisão desta Review: Opção 1 -- Executive Memory Insights entram nesta entrega exclusivamente no Executive Brief (Workspace). Justificativa, à luz da Seção 6 (Platform Integrity Assessment, pergunta 6): a leitura agregada nova que Decision Center/Portfolio Intelligence exigiriam é uma complexidade real, e nada nesta Discovery/Journey/UX Flow demonstrou que o valor entregue por ela seria proporcional ainda -- especialmente com a One Memory Insight Rule recém-adotada, que só faz sentido avaliar em produção real antes de multiplicar sua aplicação para mais 2 superfícies. Extensão futura permanece registrada, condicionada a essa leitura nova, não descartada.

06

Platform Integrity Assessment

Nova seção permanente, por instrução do Founder, a partir desta fase -- respondendo as 6 perguntas obrigatórias para Executive Memory.

1. Esta Capability preserva a identidade da STRATECH?

SIM

Nenhuma decisão automática, nenhuma alegação de fonte de verdade sobre o que aconteceu no projeto (guardrail do Product Vision, Discovery §00) -- Executive Memory só relembra um fato já registrado, nunca decide ou interpreta em nome do executivo.

2. Ela responde uma pergunta executiva única?

SIM

"O que preciso lembrar antes de decidir?" -- verificado contra as 6 Capabilities já existentes (Seção 1, pergunta 1); nenhuma sobreposição de pergunta.

3. Alguma responsabilidade passou a competir com outra Capability?

NÃO

Executive Memory não decide (Decision Center), não prioriza (Portfolio Intelligence), não organiza compromissos (Action Intelligence) -- ela só anota um fato de apoio sobre dado que essas Capabilities já leem. Escopo desta entrega limitado ao Executive Brief (Seção 4) elimina, por ora, qualquer sobreposição residual com as duas Capabilities de escala de portfólio.

4. O Executive Loop permanece íntegro?

SIM

Confirmado na Seção 2 -- nenhum elo novo, nenhuma interrupção, comportamento idêntico ao de hoje quando o Insight está em silêncio.

5. O TTD tende a diminuir ou permanecer igual?

DIMINUIR (marginal, qualitativo)

Nos casos em que um Insight real existe, elimina o custo de abrir o Histórico completo e comparar manualmente (Journey §8). Nos casos de silêncio (a maioria, por desenho), o TTD permanece exatamente igual -- nunca aumenta, porque a leitura já acontece dentro da mesma chamada que o Executive Brief já fazia (Seção 5).

6. Existe algum aumento de complexidade que não gere valor proporcional?

NÃO, NO ESCOPO DECIDIDO (Seção 4)

A complexidade desta entrega é mínima e proporcional: 2 funções puras, 1 função de seleção, 1 componente de chip, 1 parâmetro de limit ajustado -- nenhuma rota, nenhum serviço novo. A complexidade que seria desproporcional agora -- estender Decision Center/Portfolio Intelligence com uma leitura agregada nova -- foi identificada (FS-010 §3.2) e conscientemente adiada (Seção 4), exatamente para não introduzir esse aumento antes de haver evidência real de que o valor compensa.
