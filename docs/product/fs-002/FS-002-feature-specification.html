01

Contexto e Objetivo

O produto hoje tem exatamente uma rota autenticada (/dashboard) sem nenhuma moldura de navegação ao redor — o usuário chega direto no conteúdo, sem contexto de marca ou estrutura de produto. Isso deixou de refletir a maturidade real do produto (Release 0.2 concluída, Demo Mode operacional, Platform Blueprint aprovado). O objetivo desta feature é dar ao Dashboard uma moldura de produto Enterprise — sem adicionar nenhuma tela, dado ou capacidade nova.

Achado colateral: web/app/page.tsx (rota "/") ainda exibe um placeholder de "Sprint 1 de 6" dizendo "o Dashboard... ainda não existe" — falso desde a Release 0.2. Corrigir isso está dentro do escopo desta feature ("Integração com o Dashboard existente", Seção 4).
02

Architecture Review

MecanismoNext.js nested layout — novo web/app/dashboard/layout.tsx envolve web/app/dashboard/page.tsx sem mover nenhum arquivo existente. /entrar não ganha layout próprio, permanece sem shell.
Gate de sessãoweb/proxy.ts — nenhuma alteração. O matcher já cobre /dashboard; o shell só existe depois que o gate já autenticou.
Componentes novoscomponents/shell/sidebar.tsx, components/shell/app-shell.tsx — seguem exatamente o padrão de class-variance-authority + tokens já usado em badge.tsx/button.tsx. Nenhuma biblioteca nova.
HeaderA linha de título + botão "Atualizar" que já existe dentro de dashboard/page.tsx (ligada a usePortfolioSummary().refetch) não é movida para o layout — mover exigiria levantar estado de query para fora da página, risco desnecessário para uma feature "sem funcionalidade nova". O Header do shell é a moldura em volta; o conteúdo do título continua sendo responsabilidade da página.
Rota "/"app/page.tsx passa a redirecionar para /dashboard via redirect() do Next.js — reaproveita o gate de sessão já existente (usuário não autenticado cai em /entrar normalmente, pelo mecanismo já real do proxy.ts).
Contrato de APIZero mudança. Nenhuma nova chamada de rede introduzida pelo shell.
03

UX Review

Reaproveita a Decisão D6 já registrada em RFC-001 (comportamento responsivo de sidebar) — não inventa um novo padrão.

BreakpointComportamento (D6, RFC-001)
<768px (mobile)Sidebar vira barra fixa inferior — adaptada para 1 destino real (Dashboard), não os 3 originalmente especulados em D6
768–1023px (md)Sidebar colapsa para rail somente-ícone (56px)
≥1024px (lg)Sidebar completa (220px, ícone+label)
Decisão deliberada: a sidebar mostra apenas o item "Dashboard" — nenhum item cinza/desabilitado para "Projetos", "Central de Riscos" etc., mesmo esses estando na proposta original do Visual Gap Analysis. Motivo: as sessões do Protocolo de Validação estão rodando em paralelo a esta mesma release (ADR-012) — um item de navegação visível, mesmo desabilitado, é exatamente o tipo de afordance que geraria uma pergunta "isso existe?" contaminando a métrica de "informação procurada e não encontrada" que a Validation Sprint está tentando medir honestamente. Menos é mais seguro aqui.

Tokens visuais: raio 8px, sombra suave (--shadow já usado em cards), paleta reaproveitada integralmente de web/app/globals.css — nenhuma cor nova introduzida nesta feature (refinamento de token, se necessário, é o Épico 4 do Release Plan, tratado à parte).

04

Escopo

  • Sidebar com logo + nome do produto + 1 item de navegação real ("Dashboard", estado ativo)
  • Comportamento responsivo por breakpoint conforme D6 (Seção 3)
  • Layout compartilhado (app/dashboard/layout.tsx) envolvendo a página existente, sem mover ou reescrever seu conteúdo
  • Correção do placeholder obsoleto em app/page.tsx (redirect para /dashboard)
  • Verificação de contraste WCAG AA da sidebar nos dois temas (claro/escuro)
05

Fora de Escopo (explícito)

  • Qualquer item de navegação além de "Dashboard" — mesmo desabilitado (Seção 3)
  • Busca global, notificações, botão de chat no header
  • Identidade de usuário nomeada ou avatar — sessão continua Nível 1 (senha compartilhada)
  • Mover o botão "Atualizar" ou qualquer lógica de usePortfolioSummary para o shell
  • Qualquer mudança em proxy.ts, contratos de API, ou nos 3 agentes de IA
  • Reorganização visual do conteúdo do Dashboard (KPIs, tabela, Risk Center) — isso é a próxima feature (Dashboard Enterprise), não esta
06

User Stories

Como usuário autenticado, vejo uma sidebar com a identidade do produto ao acessar o Dashboard.

Critério: logo + nome do produto visíveis em lg/md; sidebar não sobrepõe conteúdo em nenhum breakpoint.

Como usuário em mobile, a navegação não consome espaço vertical do conteúdo.

Critério: sidebar vira barra inferior fixa <768px, conteúdo tem padding-bottom reservado (não sobreposto).

Como usuário acessando "/", sou levado ao Dashboard (ou ao login, se não autenticado) em vez de ver um placeholder desatualizado.

Critério: / redireciona para /dashboard; usuário não autenticado cai em /entrar pelo gate já existente, sem nenhuma lógica nova de auth.

07

NFRs Aplicáveis

  • Contraste WCAG AA em ambos os temas (claro/escuro) para todo texto e ícone da sidebar
  • Responsividade nos 3 breakpoints do RFC-001 (mobile/md/lg), testada nos mesmos projetos do Playwright já configurados (web/playwright.config.ts)
  • Nenhuma regressão de performance perceptível — o shell é CSS/layout puro, sem nova chamada de rede
  • Nenhuma regressão nos 87 testes de backend nem nos 53 testes de frontend já existentes
08

Critérios de Aceite

  • Sidebar renderiza nos 3 breakpoints exatamente como a Seção 3 descreve
  • Apenas 1 item de navegação existe, e está sempre ativo (não há outra rota para desativá-lo)
  • /entrar não é afetado — continua sem shell, sem regressão nos testes de sessão (proxy.test.ts)
  • / redireciona corretamente para /dashboard, respeitando o gate de sessão sem lógica nova
  • Suíte completa (87 backend + 53 frontend) passa sem alteração de contrato
09

Riscos Específicos

  • Baixo risco geral — feature puramente de apresentação, zero mudança de contrato ou dado.
  • Redirect de "/" é a única mudança de comportamento de rota — precisa de teste explícito confirmando que não introduz um loop de redirecionamento com proxy.ts.
  • Contraste em tema escuro — já sinalizado como risco no Visual Gap Analysis; precisa de verificação real, não suposição.
