01

Executive Questions Framework (novo princípio oficial)

A partir desta Architecture Review, toda Capability -- esta e as futuras -- deve responder explicitamente 5 perguntas antes de ser considerada arquiteturalmente pronta. O framework não substitui a Architecture Review técnica (decisões de código, §05) -- ele vem antes, garantindo que a Capability tenha uma razão executiva clara para existir, não só uma implementação correta.

  1. Qual pergunta executiva ela responde?
  2. Qual decisão ela permite tomar?
  3. Qual ação ela desencadeia?
  4. Com que frequência um executivo utilizará este módulo?
  5. Como ela conversa com os demais módulos da plataforma?

Registrado aqui como referência para toda Architecture Review futura -- não repetido por extenso em cada uma; cada Capability nova aplica o mesmo framework na sua própria Seção correspondente.

02

Framework aplicado — Action Intelligence

1. Qual pergunta executiva ela responde?

"O que precisa ser feito agora?" -- desdobrada nas 3 perguntas fixas já formalizadas na FS-007 §01: o que está atrasado, o que vence em seguida, o que exige atenção hoje.

2. Qual decisão ela permite tomar?

Por onde começar o dia -- qual projeto ou compromisso revisitar primeiro, com prioridade já calculada, sem o executivo precisar montar essa priorização sozinho a partir de reuniões espalhadas.

3. Qual ação ela desencadeia?

Nesta fase (Discovery §04, Opção 1): navegar até a análise de origem e, a partir dali, para o Workspace do projeto -- nunca uma ação executada dentro do próprio módulo (Diretriz "não é gerenciador de tarefas"). A ação real acontece fora da plataforma (uma ligação, uma decisão, uma cobrança) ou dentro de outra Capability já existente (rodar Status/Risco de novo) -- nunca dentro de Action Intelligence em si.

4. Com que frequência um executivo utilizará este módulo?

Diária, por desenho -- é o único módulo da plataforma pensado para abertura no início do dia (UX Flow §02), diferente das 3 Capabilities de Compreender, tipicamente abertas por projeto, sob demanda, quando algo muda.

5. Como ela conversa com os demais módulos da plataforma?

Nos dois sentidos -- FS-007 §2.7: recebe contexto do Dashboard (KPI vira link) e dos 3 Executive Briefs (linha de contagem); devolve contexto ao apontar cada item de volta à sua análise de origem no Workspace. Nenhuma dessas conversas exige uma tela nova além das já desenhadas.

03

O ciclo executivo e o retorno natural

Compreender→ Decidir→ Agir→ Acompanhar→ Volta à plataforma

Avaliação honesta, item por item, contra o desenho já aprovado na FS-007 -- não uma reafirmação genérica:

Passo do cicloOnde vive hojeFecha o círculo?
Compreender3 Capabilities de Intelligence (Status/Risco/Comunicação)-- (ponto de entrada do ciclo)
DecidirDecision Momentum nos 3 Briefs-- (já fecha para Compreender por natureza)
AgirAction Intelligence -- responde as 3 perguntas, prioriza (§02)Sim -- cada item aponta para source_analysis_id (FS-007 §2.1)
AcompanharAgrupamento por urgência + attentionHeadline() (FS-007 §2.4)Sim -- visibilidade contínua, mesmo sem estado persistido ainda (Journey §05)
Volta à plataformaClique no item → análise de origem no Workspace (Journey passo 5)Sim, por construção -- o Workspace é exatamente o ponto de entrada de Compreender; o ciclo fecha sem nenhuma tela nova além das já desenhadas
Confirmado, não assumido: o retorno natural não foi adicionado agora -- já estava desenhado desde a User Journey (Etapa 2, passo 5: "Clicar num item para ver a reunião de origem"). A Architecture Review confirma que essa decisão, tomada por outro motivo na época (rastreabilidade), é também o que fecha o ciclo exigido por esta diretriz -- nenhuma mudança de design necessária.

Risco identificado, não bloqueador: o "Próximo passo" de suggestedNextAction() (FS-007 §2.4) aponta para o item mais urgente, não explicitamente para uma ação de outra Capability (ex.: "rodar Status Executivo de novo"). Isso é consistente com a Diretriz "não é gerenciador de tarefas" -- a plataforma não pode sugerir rodar uma análise só porque um item está atrasado, seria uma sugestão fabricada sem base real. O retorno ao ciclo acontece pela navegação (Workspace), não por uma sugestão de próxima Capability -- suficiente para cumprir a diretriz, sem inventar uma recomendação que o dado não sustenta.

04

Coerência da convenção de rotas de leitura

Validação, não implementação -- por instrução explícita do Founder.

AspectoAvaliação
Nome do recurso (/api/action-items vs. /api/analyses)COERENTE -- já existe precedente de um recurso agregado com nome próprio, distinto de "análises": /api/portfolio/summary. action-items segue a mesma lógica -- uma visão derivada, não um CRUD novo.
Padrão de serviço (ProjectSummaryService.list_action_items)COERENTE -- mesma classe, mesmo prefixo list_ já usado por AnalysisRepository.list_analyses, sem serviço paralelo.
Autenticação e rate limitingCOERENTE -- mesmo router, herda tudo automaticamente, nenhuma exceção criada.
Ausência de versionamento (/api/, sem /v1/)HERDA UM GAP JÁ CONHECIDO -- mesmo risco Alto já registrado no Product Blueprint §12 e no STRATECH V1 Completion Board (Enterprise Essentials). Esta rota não piora o gap, também não o resolve -- consistente com adiar o versionamento como trilha própria, não escondida dentro desta Feature.
Escala -- endpoints agregados crescendo um a um (portfolio/summary, agora action-items)OBSERVAÇÃO PARA O FUTURO, NÃO AÇÃO AGORA -- se uma 3ª ou 4ª visão agregada surgir, pode valer a pena um padrão mais genérico (ex.: um único endpoint parametrizado por tipo de rollup). Duas instâncias não justificam essa generalização ainda (mesmo princípio de ADR-001: normalizar só quando uma necessidade real, repetida, aparecer) -- registrado como ponto de atenção, não como recomendação de mudança.

Conclusão da validação: a convenção proposta é coerente com a arquitetura global hoje. Nenhuma mudança de implementação recomendada.
