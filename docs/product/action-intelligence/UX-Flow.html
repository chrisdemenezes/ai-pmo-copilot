01

Princípio

A plataforma organiza o trabalho executivo. O usuário não organiza a plataforma. Mesma família de princípio da Rev. 3 de Communication Intelligence ("o usuário nunca escolhe agentes") -- aqui aplicado à forma, não ao conteúdo. Corolários concretos, que valem para toda decisão de design desta UX Flow:

  • Nenhum filtro customizável, nenhuma tag, nenhuma view salva pelo usuário.
  • Nenhuma reordenação manual -- a ordem (por urgência) é sempre a mesma regra fixa, nunca uma preferência configurável.
  • Nenhuma criação de item, nenhuma edição de campo -- reafirma a Opção 1 (Discovery §04) por um ângulo de UX, não só de arquitetura.
  • O usuário lê e decide o que fazer fora da tela; a tela nunca pede para ele arrumar a própria tela.
02

Como o usuário organiza seu dia utilizando este módulo?

Ele não organiza -- a tela já chega organizada. O padrão de uso esperado: "Ações" é o primeiro lugar que um PM/PMO abre, antes de entrar em qualquer projeto específico -- o mesmo papel que o Dashboard já cumpre para "como estão meus projetos", agora para "o que eu preciso fazer com eles". A tela mostra, sem exigir nenhuma configuração:

  1. O que está atrasado, primeiro, sempre.
  2. O que vence em breve, em seguida.
  3. O resto, visível mas deliberadamente menos proeminente -- mesmo princípio de "Também identificado" já usado em Riscos, nunca escondido, nunca com o mesmo peso do que exige atenção.

Terminada a leitura de "Ações", o usuário navega para o projeto específico que precisa de atenção -- o módulo é um ponto de partida do dia, não um lugar onde ele fica trabalhando dentro da plataforma (reforça: não é um gerenciador de tarefas, Seção 6).

04

Como o usuário percebe prioridade, urgência e impacto sem aumentar a complexidade visual?

  • Zero vocabulário visual novo. Mesmo Badge, mesmo Card, mesma lógica de "zona de atenção primeiro, resto visível mas discreto" já usada em Riscos -- nenhum semáforo novo, nenhuma escala de cor inventada.
  • Urgência é a única dimensão exposta. Não existe pontuação de "impacto" nem "prioridade" como número -- seria inventar um dado que o schema não sustenta (mesma disciplina de honestidade da Discovery §03/§05 de Communication Intelligence, "Impacto" nunca é uma nota inventada). Impacto é comunicado pela contagem e pelo agrupamento (quantos atrasados, em quantos projetos), nunca por um score.
  • Densidade igual às Briefs já existentes. Lista curta, não uma tabela densa -- mesmo padrão de "Decisão sugerida" + "Próximo passo" ancorando a leitura no que importa primeiro.
05

Como a experiência preparada hoje evoluirá naturalmente para leitura e escrita?

Detalhado arquiteturalmente na User Journey (Etapa 2, Seção 5) -- aqui, o mesmo compromisso pelo ângulo de percepção do usuário: o indicador de urgência que ele já vê hoje (Atrasado / Vence em breve / No prazo) é visualmente o mesmo espaço onde, no futuro, "Concluído" vai aparecer. O usuário de hoje já está, sem saber, olhando para o layout final -- só um dos estados possíveis daquele indicador ainda não existe. Nenhuma segunda versão da tela será necessária; a mesma tela ganha um comportamento novo (marcar como feito) no mesmo lugar onde hoje só se lê o estado.

06

Como evitar que este módulo seja percebido como um gerenciador de tarefas genérico?

  • Vocabulário: nunca "tarefa" -- sempre "ação" ou "compromisso", palavras que carregam a origem real (algo que alguém disse numa reunião real), não um item que o usuário criou do nada.
  • Proveniência sempre visível: todo item aponta para a reunião de origem (Journey, passo 5) -- diferente de um gerenciador de tarefas genérico, onde o item não tem "de onde veio".
  • Nenhum botão de criar. A ausência mais importante desta tela -- reforça que ela reflete compromissos já assumidos, não um lugar para planejar novos.
  • Nenhuma atribuição pela interface. owner vem só do que a IA já extraiu de uma transcrição real -- o usuário nunca atribui algo a alguém pela tela.
  • Moldura da pergunta: "o que já foi comprometido e está em aberto", não "o que eu ainda preciso planejar" -- retrospectivo e factual, não prospectivo e criativo.
07

Integração contextual: Executive Brief → Action Intelligence (avaliação)

Por instrução explícita do Founder: avaliação apenas, nenhuma implementação nesta etapa.

Viável, e barato. ExecutiveBrief (Status) já busca useWorkspaceSummary, que já retorna pending_action_items -- uma linha como "N ações relacionadas exigem atenção", com link para a seção "Ações" do Workspace, é reaproveitamento puro do dado já buscado, zero chamada nova. RisksPanel e CommunicationBrief hoje não chamam useWorkspaceSummary -- adicionar essa chamada é o mesmo hook já usado em outro lugar, não um endpoint novo.

Recomendação para a Feature Specification avaliar formalmente: aplicar a mesma linha de contexto nos 3 Briefs, não só em Comunicação -- o usuário pode estar olhando Status ou Risco de um projeto que também tem ações em aberto vindas de uma reunião; o princípio "a plataforma organiza" (Seção 1) sugere avisar independente de qual Brief ele está lendo, não só naquele que originou o dado. Risco identificado: se a contagem for zero, a linha não deve aparecer -- reforça "nunca listar tudo com o mesmo peso", mesmo quando o "tudo" é zero.
