01

Contexto e Objetivo

Esta FS formaliza os 3 documentos já aprovados (Discovery, Journey, UX Flow) sob a diretriz estratégica do Founder: Action Intelligence não é um gerenciador de tarefas -- é um módulo de apoio à decisão e execução executiva (Diretriz 1). A diferença não é de conteúdo (os dados são os mesmos action_items[] já reais), é de enquadramento: a tela nunca pede ao usuário para organizar nada -- ela responde três perguntas executivas fixas, sempre na mesma ordem, por hierarquia visual, nunca por configuração manual (Diretrizes 2 e 3):

"O que está atrasado?"→ "O que vence em seguida?"→ "O que exige minha atenção hoje?"

Fecha o ciclo executivo Compreender → Decidir → Agir → Acompanhar (User Decision Discovery §01), e passa a ser referência funcional integrada de Dashboard, Workspace e os 3 Executive Briefs (Diretriz 4) -- não mais um módulo isolado.

02

Architecture Review

Achado real, revisa um pressuposto da Journey/UX Flow: a Journey (§Reuso) e a UX Flow (§3) assumiram "zero endpoint novo" com base em AnalysisRepository.list_analyses já existir. Verificado contra o código real: GET /api/analyses devolve AnalysisSummary (id, kind, project_name, created_at) -- sem payload; só GET /api/analyses/{"{id}"} devolve o payload completo, um registro por vez. Buscar action_items[] de todas as reuniões de um projeto (ou do portfólio) por essa rota exigiria N+1 chamadas -- inaceitável, mesmo princípio já evitado em ProjectSummaryService.summarize_portfolio() ("uma busca, agrupado em memória, não uma consulta por projeto"). A camada de repositório já tem o dado (o mesmo list_analyses() que ProjectSummaryService usa para contar pending_action_items devolve AnalysisRecord completo, com payload) -- só a API pública não expõe a lista de itens, só a contagem. Correção: uma rota de leitura nova é necessária, descrita em §2.2. Isso não contradiz "zero persistência nova" (Discovery §04, Opção 1) -- é leitura pura, mesmo padrão de GET /api/portfolio/summary, sem tabela nova, sem migração.

2.1 -- Serviço: extensão de ProjectSummaryService

Reaproveita a mesma classe, mesmo padrão de _aggregate -- não um serviço paralelo:

class ProjectSummaryService:
    def list_action_items(self, project_name: str | None = None) -> list[dict]:
        # Mesma chamada já usada por summarize()/summarize_portfolio() --
        # zero query nova, zero tabela nova.
        records = self._repository.list_analyses(
            project_name=project_name, kind="meeting", limit=None,
        )
        items = []
        for record in records:
            model_output = (record.payload or {}).get("model_output")
            if not isinstance(model_output, dict) or not model_output.get("structured"):
                continue
            for item in model_output.get("action_items") or []:
                items.append({
                    "project_name": record.project_name,
                    "description": item.get("description"),
                    "owner": item.get("owner"),
                    "due_date": item.get("due_date"),
                    "source_analysis_id": record.id,
                    "source_created_at": record.created_at,
                })
        return items

Mesma disciplina de robustez de schema já estabelecida (structured e formato verificados antes de ler o array) -- uma resposta mal-formada de uma reunião específica não derruba o rollup inteiro, só é excluída dele.

2.2 -- Nova rota: GET /api/action-items

Mesmo router (src/api/routes/intelligence.py), mesmo padrão de DI (Depends(build_repository)), project_name opcional (presente = visão de Workspace; ausente = visão de portfólio) -- idêntico ao parâmetro já usado em GET /api/analyses. Mesma autenticação (X-API-Key), mesmo rate limiting, nenhum middleware novo.

2.3 -- BFF e Hook

Mesmo padrão de sempre: web/app/api/bff/action-items/route.ts (portfólio) reaproveitando o mesmo proxy fino já usado por /api/bff/dashboard; a variante de Workspace usa project_name como query param, mesmo mecanismo. Novo hook useActionItems(projectName?: string), TanStack Query, mesmo template dos hooks de leitura já existentes (useWorkspaceSummary, usePortfolioSummary) -- nunca useMutation, porque não há escrita.

2.4 -- action-momentum.ts (novo, mesmo princípio de meeting-momentum.ts)

export type UrgencyBucket = "atrasado" | "vence_em_breve" | "no_prazo" | "sem_prazo";

export function bucketByUrgency(dueDate: string | null, today: Date): UrgencyBucket {
  if (!dueDate) return "sem_prazo";
  const due = new Date(dueDate);
  if (due < today) return "atrasado";
  if (due <= addDays(today, 7)) return "vence_em_breve";
  return "no_prazo";
}

// "O que exige minha atenção hoje?" -- nunca uma nota de severidade inventada,
// mesma honestidade de impactHeadline() em meeting-momentum.ts.
export function attentionHeadline(atrasadoCount: number, venceEmBreveCount: number): string {
  return `${atrasadoCount} atrasada(s) · ${venceEmBreveCount} vence(m) em breve`;
}

2.5 -- Reuso de Executive Brief e Decision Momentum como componentes nativos (Diretriz 7)

Avaliado, conforme pedido. Os 3 Briefs já reais (executive-brief.tsx, risks-panel.tsx, communication-brief.tsx) compartilham a mesma composição -- Card > estados independentes (loading/erro/vazio/fallback/sucesso) > corpo com "Decisão sugerida" + "Próximo passo". Existe, sim, oportunidade real de extração -- as 4 instâncias (as 3 existentes + esta) seriam candidatas naturais a um componente compartilhado. Recomendação: não extrair agora. Essa extração é exatamente o escopo já registrado como "Executive Brief Contract" (FS-006 §15) -- e o STRATECH V1 Completion Board Rev. 2 já registrou, por instrução do Founder, que esse item fica deliberadamente parado até o Action Intelligence ser concluído. Extrair uma abstração parcial agora, só para este 4º caso, seria fazer o trabalho do Executive Brief Contract por partes, fora de ordem. Esta Feature segue o mesmo padrão por convenção (copiando a forma comprovada, como as 3 anteriores fizeram entre si) -- quando o Executive Brief Contract for priorizado, as 4 instâncias são candidatas igualmente maduras à extração real, com action já disponível como um quarto exemplo concreto para validar a abstração contra a necessidade real, não contra a hipótese.

2.6 -- Preparação para estados persistidos futuros (Diretriz 5)

Registro explícito de como a arquitetura de hoje já reserva o lugar para Concluído/Em andamento/Delegado, sem mexer em navegação, componentes ou estrutura visual quando a escrita chegar:

// Hoje: união de 2 conjuntos de valores no mesmo campo visual "estado do item".
// UrgencyBucket é sempre calculado (nunca persistido).
// Quando a escrita existir, ActionStatus passa a coexistir no mesmo lugar --
// o componente de card já recebe um "estado" único, não dois campos.
type UrgencyBucket = "atrasado" | "vence_em_breve" | "no_prazo" | "sem_prazo"; // hoje, sempre derivado
type ActionStatus = "aberto" | "em_andamento" | "concluido" | "delegado";      // futuro, quando persistido

// O componente de item já é desenhado para receber um "estado" (união dos dois),
// não uma prop separada por conceito -- é o mesmo ponto de extensão descrito
// na User Journey §05, agora expresso como forma de tipo, não só como intenção.

Nenhum desses tipos futuros é implementado nesta Feature -- ficam registrados como o contrato que a Feature de escrita, quando vier, precisa honrar, não como código morto nesta entrega.

Query Keys e Cache

Leitura pura -- não há invalidação de mutação (não existe useMutation nesta Feature). Chave de cache: ["action-items", projectName ?? "portfolio"], mesmo staleTime/refetchInterval já usado por usePortfolioSummary. Os 3 Briefs, ao passar a chamar useActionItems, compartilham a mesma entrada de cache por projeto -- nenhuma chamada duplicada entre a seção "Ações" e a linha de contexto de cada Brief quando ambas estão montadas na mesma tela de Workspace.

07

User Stories

Como PM/PMO, abro "Ações" e sei em segundos o que está atrasado em todo o portfólio, sem escolher um projeto antes.

Critério: a tela abre já agrupada por urgência, atrasado primeiro, sem nenhuma configuração prévia.

Como PM/PMO, ao ler o Brief de Status ou Risco de um projeto, sei se há ações em aberto sem precisar visitar outra tela.

Critério: a linha de contexto aparece nos 3 Briefs quando a contagem de atenção é maior que zero, some quando é zero.

Como PM/PMO, nunca vejo um botão de criar ou editar ação -- só o que já foi realmente comprometido em reuniões reais.

Critério: nenhum formulário de criação/edição existe em nenhuma das duas telas novas.

Riscos Específicos

  • due_date em formato inconsistente entre análises (texto livre extraído pela IA) -- bucketByUrgency precisa tratar data inválida como "sem_prazo", nunca quebrar a tela
  • Volume de itens crescendo sem paginação -- aceitável no volume atual de dado fictício/piloto; registrar como item de acompanhamento, não implementar paginação nesta Feature (fora de escopo, mesmo gap já conhecido de GET /api/portfolio/summary)
  • Linha de contexto nos 3 Briefs aumentar a carga de leitura de cada tela -- mitigado por só aparecer quando a contagem é maior que zero (§2.7)

Checklist de Segurança (ADR-011)

Mesmo padrão já auditado: API_KEY só no servidor, nova rota sob o mesmo router autenticado, nenhuma rota de sessão nova. Dado exposto (description, owner, due_date) já trafega hoje dentro do payload de reunião via API -- esta Feature só reorganiza a leitura, não introduz uma categoria de dado sensível nova.
