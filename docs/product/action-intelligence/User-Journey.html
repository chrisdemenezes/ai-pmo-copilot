01

Persona e Pontos de Entrada

Persona: PM/PMO, a mesma de toda a plataforma. Diferente das 3 Capabilities de Compreender -- todas escopadas a um projeto por vez, dentro do Workspace -- "o que precisa ser feito agora" é, por natureza, uma pergunta que atravessa projetos. Por isso esta jornada tem dois pontos de entrada legítimos, a mesma lista de dados por trás dos dois:

  • Portfólio (novo item de navegação "Ações"). Ponto de entrada primário -- resposta direta à pergunta "o que precisa ser feito agora" em todos os projetos, sem escolher um antes. Segue a Regra de Entrada na Runtime Navigation já formalizada (docs/product/platform-architecture/Platform-Navigation-Architecture.html): só entra em navigation.ts quando tiver rota real, dado real, estados completos e testes -- exatamente o que esta Feature entrega.
  • Workspace (seção "Ações" dentro do projeto). O mesmo rollup, filtrado a um projeto -- mesma posição das seções já existentes (Riscos, Comunicação), mesmo componente reaproveitado com um filtro diferente.

Nenhuma tela nova de "minhas tarefas" por usuário individual -- não existe login individual ainda (Discovery §03, Enterprise Expansion/V2).

04

Pontos de Decisão Explícitos

  • Passo 2: o PM/PMO decide, ao ver o que está atrasado, o que resolver primeiro -- a plataforma prioriza a lista, mas não decide por ele.
  • Passo 4: o único ponto de decisão que a plataforma ativamente propõe -- qual item olhar primeiro. Nunca uma ação silenciosa, nunca uma marcação automática de urgência além do cálculo honesto de data.
  • Passo 5: decidir se vale revisitar a reunião de origem antes de agir -- caminho opcional, nunca obrigatório para entender o item.
05

Preparação para Escrita Futura

Condição arquitetural obrigatória do Founder: a experiência de leitura de hoje precisa ser exatamente onde a escrita futura vai encaixar, sem mover nada. Esta Journey descreve como isso fica garantido nesta fase -- o desenho concreto do componente é assunto da Feature Specification, não desta Journey.

  • Mesmo cartão, mesma posição de indicador. Cada item já nasce com um indicador visual de estado (hoje: Atrasado / Vence em breve / No prazo / Sem prazo, todos calculados). O dia em que "Concluído" existir, ele ocupa o mesmo espaço visual -- não é uma coluna nova, é um novo valor possível do mesmo indicador.
  • Mesma lista, mesmo agrupamento. Agrupar por urgência hoje é o mesmo mecanismo que agrupará "Concluído" para o fim da lista amanhã -- sem trocar de componente.
  • Mesma navegação, mesmas duas entradas. Portfólio ("Ações") e Workspace ("Ações" do projeto) continuam sendo os dois lugares certos quando a escrita chegar -- nenhuma tela nova, nenhum novo ponto de entrada a inventar depois.
  • Mesma fonte de dado, evolução aditiva. A leitura de hoje já busca action_items[] agregados via AnalysisRepository. Se/quando a extração para uma tabela ActionItem própria acontecer (ADR-001, Discovery §00), a mesma consulta lógica troca de fonte por trás -- o componente de tela não precisa saber a diferença.
Nenhum desses quatro pontos exige decidir agora como a escrita vai funcionar -- só exige que a leitura de hoje não feche portas. A Feature Specification deve tratar isso como um requisito de design de componente (onde o indicador de estado vive, como a lista é montada), não como uma antecipação de schema de banco.

Fora de Escopo (confirmado)

  • Nenhuma criação de item novo pela interface -- só os já extraídos de análises de reunião reais.
  • Nenhuma edição de descrição/responsável/prazo.
  • Nenhum campo de status persistido, nenhuma marcação de "concluído" real nesta fase (Discovery §04, Opção 1).
  • Nenhuma migração de banco, nenhuma alteração de ActionItem como entidade -- ADR-001 permanece intocada.
  • Nenhuma tela de "minhas tarefas" por usuário individual -- sem login individual (V2).
  • Nenhum quadro Kanban, sprint board ou rastreamento de tempo -- fronteira já registrada no Product Blueprint, reafirmada aqui.
