Workspace Discovery — Project Intelligence Workspace

Mapeia exclusivamente o que já existe hoje no backend real e o que pode ser derivado dele, para cada seção candidata de um Workspace de projeto individual. Nenhum dado, endpoint ou integração novos são propostos aqui — apenas classificação do estado real.

Fonte src/database/repository.py, src/api/routes/intelligence.py, src/agents/*/prompts/analysis.md Data 2026-07-12 Status Discovery — antecede a FS-004

00 · Método

Cada seção candidata do Workspace foi verificada diretamente contra o schema real do banco, os endpoints reais e os prompts reais dos 3 agentes — não contra o Figma, não contra suposição. Classificação:

JÁ EXISTE dado real, alcançável hoje via API PODE SER DERIVADA calculável a partir de dados já reais, sem novo dado/endpoint DEPENDE DE BACKEND FUTURO exige coluna/tabela/endpoint que não existe DEPENDE DE IA FUTURA exige capacidade de agente que não existe FORA DE ESCOPO R0.3 exige sistema inteiro ausente (identidade, storage, integrações)
Fato estrutural que condiciona tudo abaixo: não existe uma tabela Project. Todo o produto hoje é uma tabela única, analysis_records (id, kind, project_name, payload JSON, created_at) — project_name é uma string livre, não uma chave estrangeira. Um "Project Intelligence Workspace" real, portanto, é uma agregação de análises filtradas por project_name, não a página de uma entidade própria com seu próprio ID.

A · Cabeçalho do Projeto

Nome do projeto, saúde atual, contagem de análises/riscos/ações.

JÁ EXISTE

Idêntico ao já usado no Dashboard: GET /api/projects/{project_name}/summary retorna project_name, total_analyses, open_risks, pending_action_items, latest_health_status. Já real, já reachable — só não está renderizado em nenhuma tela própria de projeto hoje.

B · Histórico de Análises

Lista cronológica das execuções (reunião, risco, status) de um projeto.

JÁ EXISTE

GET /api/analyses retorna id, kind, project_name, created_at por execução (sem o payload pesado). Filtrável por projeto e por tipo/período (já entregue em releases anteriores). Isso é literalmente a lista de "execuções" que um histórico de Workspace precisa.

C · Riscos Identificados

Lista estruturada de riscos (descrição, probabilidade, impacto, mitigação) da análise de risco mais recente do projeto.

JÁ EXISTE

Dentro do payload.model_output.risks[] da análise mais recente do tipo "risk", acessível via GET /api/analyses/{id}. Cada item já tem description, probability (low/medium/high), impact (low/medium/high), mitigation. escalation_recommendation também já vem no mesmo payload.

Limite real a documentar, não a resolver aqui: não existe campo de status por risco (aberto/mitigado/fechado) — todo risco presente no payload mais recente é contado como "aberto" implicitamente. Rastrear o ciclo de vida de um risco individual ao longo de múltiplas análises é DEPENDE DE BACKEND FUTURO (exigiria uma tabela própria de riscos com estado, não um blob JSON por execução).

D · Ações Pendentes

Lista de action items (descrição, responsável, prazo) da análise de reunião mais recente.

JÁ EXISTE

payload.model_output.action_items[] de uma análise "meeting" já traz description, owner (string|null), due_date (string|null).

Limite real: owner é texto livre, não vinculado a nenhuma identidade de usuário — e não há campo de conclusão (feito/pendente) rastreado por item. Marcar uma ação como concluída é DEPENDE DE BACKEND FUTURO.

E · Decisões & Dependências

Decisões registradas e dependências identificadas na reunião mais recente.

JÁ EXISTE

payload.model_output.decisions[] e payload.model_output.dependencies[] — ambos arrays de string, já presentes no payload de toda análise "meeting".

F · Achados-chave & Recomendações

Principais achados e recomendações da análise de status mais recente.

JÁ EXISTE

payload.model_output.key_findings[] e payload.model_output.recommendations[] de uma análise "status" — já reais.

G · Linha do Tempo de Saúde

Evolução do health_status do projeto ao longo do tempo (série histórica, não um único valor atual).

PODE SER DERIVADA

Diferente do sparkline de KPI do portfólio (recusado na FS-003 por falta de série histórica agregada), aqui cada análise individual do tipo "status" já carrega seu próprio created_at e payload.model_output.health_status. Buscar todas as análises "status" de um projeto via GET /api/analyses e montar a série ordenada por data é agregação sobre dados já reais — nenhum novo dado, nenhuma nova coluna, nenhum novo endpoint.

H · Detalhe de uma Análise

Abrir uma execução específica do histórico (Seção B) e ver seu payload completo.

JÁ EXISTE

GET /api/analyses/{id} retorna o registro completo, incluindo payload — exatamente o drill-down que esta seção precisa.

Limite real: não há transcript bruto persistido como texto pesquisável — só existe o que o LLM ecoou de volta dentro de payload.model_output, e os prompts atuais não pedem isso. Exibir a transcrição original de uma reunião é DEPENDE DE BACKEND FUTURO (exigiria uma coluna dedicada para persistir o texto de entrada).

I · Matriz de Riscos (Probabilidade × Impacto)

Agrupamento visual dos riscos por severidade.

PODE SER DERIVADA

probability e impact já existem por risco individual (Seção C) — agrupar/contar riscos em uma grade 3×3 é agregação client-side sobre dado já real, sem novo campo.

J · Responsável / Time do Projeto

Um PM ou dono nomeado para o projeto como um todo (não por action item).

FORA DE ESCOPO R0.3

Não existe identidade de usuário no produto — sessão é Nível 1 (senha única compartilhada, sem login individual). owner em action items é texto livre por item, não um responsável do projeto. Introduzir isso exige um sistema de identidade que a Release 0.3 não tem.

K · Documentos & Anexos

FORA DE ESCOPO R0.3

Não há storage de arquivos no produto.

L · Comentários & Colaboração

FORA DE ESCOPO R0.3

Não há tabela de comentários/threads. Exigiria também identidade de usuário (Seção J).

M · Notificações & Alertas

FORA DE ESCOPO R0.3

Não existe sistema de notificação no produto.

N · Integrações Externas (Jira, Azure DevOps, SAP)

FORA DE ESCOPO R0.3

Não existem — conteúdo exclusivamente fictício do Figma de referência, já sinalizado como tal na Visual Fidelity Sprint.

O · Progresso, Cronograma e Orçamento

FORA DE ESCOPO R0.3

Nenhum desses campos existe no schema — nenhum agente produz progresso percentual, data de início/fim ou valores de orçamento.

P · Síntese de IA entre Análises

Um resumo consolidado que combine achados da reunião + risco + status mais recentes em uma única narrativa.

DEPENDE DE IA FUTURA

Os 3 agentes operam de forma isolada — não existe um agente ou prompt que leia múltiplas análises e produza uma síntese entre elas. Diferente das Seções C–F (que já são o output direto de um agente existente), isto exigiria uma nova capacidade de IA, não apenas reorganizar dado já real.

01 · Resumo — Escopo Habilitado para FS-004

Por decisão do Product Owner, apenas JÁ EXISTE e PODE SER DERIVADA podem nascer na FS-004. As demais permanecem de fora, sem exceção.

8 seções habilitadas para a FS-004: A, B, C, D, E, F, G, H, I — cabeçalho do projeto, histórico de análises, riscos, ações pendentes, decisões/dependências, achados-chave/recomendações, linha do tempo de saúde derivada, detalhe de análise individual, e matriz de riscos derivada. Juntas, elas descrevem um Workspace real: drill-down de um projeto a partir dos dados que os 3 agentes já produzem hoje, sem inventar nenhum dado, endpoint ou integração.
