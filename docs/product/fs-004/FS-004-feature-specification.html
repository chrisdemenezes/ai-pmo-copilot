Feature Specification · Release 0.3

FS-004 — Project Intelligence Workspace

Tela de drill-down por projeto, construída exclusivamente por agregação e derivação das análises já existentes em analysis_records. Nenhuma entidade, tabela, repositório ou endpoint de "Project" é criado.

Release 0.3 Origem Workspace Discovery (aprovado) Data 2026-07-12 Status Aguardando aprovação — TIP-004 ainda não iniciado
01

Contexto e Objetivo

O Workspace Discovery mapeou 16 seções candidatas contra o backend real e aprovou 8 para esta feature — todas classificadas já existe ou pode ser derivada. O Chief Product Architect formalizou o princípio arquitetural que governa toda esta FS:

"O Project Workspace não representa uma entidade Project persistida. Ele representa uma visão agregada das análises existentes para um mesmo project_name."

Isso significa que o Workspace é uma tela nova, não um domínio novo. A fonte de verdade continua sendo a mesma tabela única analysis_records já usada pelo Dashboard — o Workspace apenas recorta, agrega e apresenta essas mesmas linhas filtradas por um project_name.

02

Architecture Review

Consequências do princípioNenhuma entidade Project, nenhuma tabela projects, nenhum ProjectRepository, nenhum endpoint de backend novo. Os 3 endpoints já existentes (GET /api/projects/{"{"}project_name{"}"}/summary, GET /api/analyses, GET /api/analyses/{"{"}id{"}"}) são os únicos consumidos. Zero linha nova em src/database/, src/api/ ou src/services/.
Rota novaweb/app/workspace/[projectName]/page.tsx — rota dinâmica. web/app/workspace/[projectName]/layout.tsx reaproveita o <AppShell> já existente (mesmo componente do dashboard/layout.tsx), sem duplicar sidebar/header.
Gate de sessão — única mudança em proxy.tsweb/proxy.ts precisa estender o matcher de ["/dashboard", "/dashboard/:path*", "/api/bff/:path*"] para incluir "/workspace", "/workspace/:path*". É a única mudança fora de web/app/workspace/ e web/lib/workspace/ nesta feature — sem ela, o Workspace ficaria acessível sem autenticação.
BFF — 3 rotas finas, proxy puroweb/app/api/bff/workspace/[projectName]/summary/route.ts (proxia /api/projects/{"{"}project_name{"}"}/summary), .../analyses/route.ts (proxia GET /api/analyses?project_name=..., reaproveitando os filtros já existentes de tipo/período/paginação), .../analyses/[analysisId]/route.ts (proxia GET /api/analyses/{"{"}id{"}"}). Seguem exatamente o padrão de api/bff/dashboard/route.ts: timeout de 8s, X-API-Key nunca exposto ao navegador, cache: "no-store". Nenhuma agregação de negócio acontece no BFF — ele só repassa.
Agregação/derivação — no frontend, não no backendweb/lib/workspace/aggregate.ts (novo, mesma pasta-padrão de lib/dashboard/aggregate.ts) computa a Intelligence Timeline (ordena análises por created_at, anota health_status nas de tipo "status") e a Matriz de Riscos (agrupa probability×impact dos riscos já carregados) — puramente client-side, sobre payloads já buscados, sem persistir nada novo. Isso satisfaz literalmente "não duplicar dados já existentes em analysis_records": nada é copiado para um novo lugar, só recalculado a cada carregamento.
Tipos — sem entidade Projectweb/lib/workspace/types.ts define WorkspaceSummary, AnalysisListItem, AnalysisDetail — todos espelhando 1:1 os schemas reais do backend (ProjectSummaryResponse, AnalysisSummary, AnalysisDetail). Nenhum tipo Project é declarado.
Achado crítico — codificação de project_nameO portfólio real já contém um projeto chamado "Implantacao SAP S/4HANA" — o caractere / no nome já quebrou uma chamada de API real nesta mesma Release (barra não codificada em curl .../{"{"}project_name{"}"}/summary virou 3 segmentos de rota em vez de 2). O link do Dashboard para o Workspace, a rota dinâmica [projectName], e as 3 chamadas do BFF ao backend devem usar encodeURIComponent(project_name) de forma consistente em toda a cadeia — coberto por teste explícito (Seção 9).
Ponto de entradacomponents/dashboard/project-health-grid.tsx — cada linha da tabela (md/lg) e cada card (mobile) passa a ser um Link para /workspace/{"{"}encodeURIComponent(project_name){"}"}. Único touch no Dashboard já existente; nenhuma coluna ou dado novo é adicionado a essas visualizações.
Navegação (sidebar)NAV_ITEMS permanece com exatamente 1 entrada ("Dashboard"). O Workspace é uma tela de drill-down, não um destino de navegação persistente — mesma decisão deliberada já registrada na FS-002 (menos itens visíveis = menos afordance especulativa).
03

UX Review

Sequência recomendada pelo Chief Product Architect, mapeada 1:1 às seções aprovadas no Workspace Discovery. "Decisões" (6) absorve também "Dependências" — o Discovery já classificou os dois arrays (decisions[], dependencies[]) como vindos do mesmo payload de reunião; separar em duas seções distintas duplicaria cabeçalho sem separar dado real.

1 · Cabeçalho→ 2 · Executive Summary→ 3 · Intelligence Timeline→ 4 · Riscos→ 5 · Ações→ 6 · Decisões (+ Dependências)→ 7 · Recomendações→ 8 · Histórico completo
SeçãoFonte real
1. Cabeçalho do ProjetoGET .../summary: nome, saúde atual, contagens
2. Executive SummaryMesma resposta de .../summary + key_findings/health_status da análise "status" mais recente — apresentação de dado já real, não uma nova síntese de IA (essa é a distinção explícita com a Seção P do Discovery, "Síntese de IA entre Análises", que continua fora de escopo)
3. Intelligence TimelineGET .../analyses + enriquecimento client-side com health_status das entradas "status" — Seção G do Discovery (derivada)
4. Riscospayload.model_output.risks[] da análise "risk" mais recente, com a Matriz (Seção I do Discovery, derivada) como sub-visualização
5. Açõespayload.model_output.action_items[] da análise "meeting" mais recente
6. Decisões (+ Dependências)decisions[] + dependencies[] da mesma análise "meeting"
7. Recomendaçõesrecommendations[] da análise "status" mais recente
8. Histórico completoGET .../analyses paginado (reaproveita limit/offset já existentes) + drill-down para GET .../analyses/{"{"}id{"}"}

Estados de carregamento, vazio e erro reaproveitam integralmente o padrão já estabelecido no TIP-003: Card para vazio/erro, Skeleton proporcional à forma real de cada seção. Um projeto sem nenhuma análise de um tipo (ex.: nunca teve reunião analisada) mostra um estado vazio por seção, não a tela inteira — cada uma das 8 seções degrada independentemente.

Decisão deliberada: se um projeto não tem nenhuma análise "status", a seção 2 (Executive Summary) mostra um estado vazio explícito — nunca um resumo inventado a partir de dados de outro tipo de análise. O mesmo vale para as seções 4-7 quando o tipo de análise correspondente nunca rodou para aquele projeto.
04

Escopo

  • Rota /workspace/[projectName], protegida pelo mesmo gate de sessão do Dashboard
  • 8 seções na sequência definida (Seção 3), cada uma alimentada por dado já real ou derivado
  • 3 rotas BFF finas (proxy puro, sem lógica de negócio) para os 3 endpoints já existentes
  • lib/workspace/aggregate.ts para Intelligence Timeline e Matriz de Riscos (derivação client-side)
  • Link de cada linha/card do Dashboard para o Workspace correspondente, com project_name corretamente codificado
  • Estados de vazio/erro/carregamento por seção, reaproveitando Card/Skeleton já existentes
  • Extensão do matcher de proxy.ts para cobrir /workspace
05

Fora de Escopo (explícito)

  • Qualquer entidade, tabela, repositório ou endpoint de "Project" no backend — proibido pelo princípio arquitetural desta FS
  • Responsável/Time do projeto, Documentos/Anexos, Comentários/Colaboração, Notificações/Alertas, Integrações externas, Progresso/Cronograma/Orçamento — todas classificadas "fora de escopo R0.3" pelo Workspace Discovery
  • Síntese de IA entre análises (um agente que combine reunião+risco+status em uma narrativa nova) — classificada "depende de IA futura"; a seção "Executive Summary" desta FS usa apenas dado já existente, nunca uma síntese nova
  • Status individual de risco ou ação (aberto/mitigado/concluído) rastreado ao longo do tempo — exigiria schema novo, já sinalizado como limite real no Discovery
  • Transcript bruto da reunião — não persistido hoje
  • Qualquer alteração em src/database/, src/api/ ou src/services/ além do consumo já existente dos 3 endpoints
  • Qualquer alteração nos 3 agentes de IA ou no Demo Mode
06

User Stories

Como PMO executivo, clico em um projeto no Dashboard e vejo um panorama completo dele em uma tela dedicada.

Critério: clicar em qualquer linha/card de ProjectHealthGrid navega para /workspace/{"{"}project_name{"}"}, inclusive para projetos com / no nome.

Como PMO executivo, vejo a saúde do projeto ao longo do tempo, não só o valor atual.

Critério: a Intelligence Timeline lista as análises em ordem cronológica com o health_status de cada análise "status" visível.

Como PMO executivo, acesso os riscos, ações, decisões e recomendações mais recentes sem precisar abrir cada análise manualmente.

Critério: as seções 4-7 mostram o conteúdo da análise mais recente de cada tipo, já extraído do payload.

Como PMO executivo, consigo abrir qualquer análise antiga para ver o detalhe completo daquele momento.

Critério: a seção 8 lista todo o histórico paginado, e cada item abre o payload completo via GET .../analyses/{"{"}id{"}"}.

NFRs Aplicáveis

  • Nenhuma regressão nos 87 testes de backend — nenhum arquivo de backend é tocado
  • Nenhuma regressão nos testes de frontend já existentes
  • Contraste WCAG AA em ambos os temas nas 8 novas seções
  • Responsividade nos 3 breakpoints já estabelecidos (RFC-001 D6)
  • O Workspace não é otimizado para portfólios com centenas de análises por projeto nesta primeira versão — o volume real atual do produto (dezenas de análises por projeto) não justifica paginação agressiva na Intelligence Timeline; otimização futura (se necessária) é decisão de uma próxima feature, não desta
  • project_name com caracteres especiais (/, espaços, acentos) deve navegar e carregar corretamente de ponta a ponta — teste explícito com "Implantacao SAP S/4HANA", o caso real já conhecido
08

Critérios de Aceite

  • As 8 seções renderizam na ordem definida, cada uma com dado real ou estado vazio explícito
  • Nenhuma seção 9-16 do Workspace Discovery (fora de escopo / depende de IA futura) aparece na tela
  • /workspace/[projectName] não é acessível sem sessão válida — mesmo comportamento de redirect para /entrar já testado em /dashboard
  • Navegar para um projeto com / no nome (ex.: "Implantacao SAP S/4HANA") funciona sem erro 404 em nenhuma das 3 chamadas de BFF
  • Zero linha nova em src/database/, src/api/, src/services/
  • Suíte completa (backend + frontend + E2E) passa sem alteração de contrato de backend
09

Riscos Específicos

  • Codificação de project_name (Seção 2) — risco já materializado uma vez nesta Release; precisa de teste E2E dedicado, não apenas revisão de código
  • N+1 requisições para a Intelligence Timeline — buscar o payload de cada análise "status" individualmente para montar a série histórica gera múltiplas chamadas ao backend por carregamento de tela. Aceitável no volume atual (Demo Mode, portfólio real pequeno); risco de degradação se o volume crescer significativamente — não resolvido nesta FS, por decisão explícita (NFR, Seção 7)
  • Extensão do matcher de proxy.ts — única mudança fora da árvore nova; erro de configuração deixaria /workspace acessível sem autenticação. Precisa de teste E2E explícito de redirect não-autenticado, espelhando o já existente para /dashboard
  • Seções vazias mal distinguidas de erro — um projeto real sem análise de um tipo específico não é um erro; a UI precisa deixar essa diferença clara (Seção 3, UX Review)
10

Checklist de Segurança (ADR-011)

Diferente da FS-002/FS-003, esta feature toca o gate de sessão — checklist aplicado de fato, não N/A:

  • Verificado As 3 novas rotas BFF seguem exatamente o padrão de api/bff/dashboard/route.ts: API_KEY só existe no servidor, nunca é serializado na resposta ao navegador
  • Verificado matcher de proxy.ts estendido, não substituído — /dashboard e /api/bff/:path* continuam cobertos sem alteração de comportamento
  • A verificar no TIP-004 teste E2E confirmando que acesso não-autenticado a /workspace/qualquer-coisa redireciona para /entrar, e que o cookie de sessão continua httpOnly
  • A verificar no TIP-004 project_name vindo da URL nunca é interpolado diretamente em HTML sem sanitização padrão do React (proteção XSS já garantida pelo framework, mas validar que nenhum dangerouslySetInnerHTML é introduzido)
11

Rollback

Reverter o commit desta feature remove a rota /workspace, as 3 rotas BFF, e reverte o matcher de proxy.ts ao estado atual. O link adicionado em project-health-grid.tsx também é revertido — o Dashboard volta a não ter nenhum link de drill-down. Nenhuma migração de dado, nenhuma mudança de schema — rollback é reversão de código pura.
