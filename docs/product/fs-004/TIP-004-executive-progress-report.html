Achado real durante a verificação — fora do escopo do TIP-004

O endpoint GET /api/projects/{"{"}project_name{"}"}/summary tem uma limitação pré-existente: não consegue servir nenhum projeto cujo nome contenha "/", mesmo com o nome corretamente codificado. Confirmado direto contra o backend real: 
curl -H "X-API-Key: ..." ".../api/projects/Implantacao%2520SAP%2520S%252F4HANA/summary"
→ {"detail":"Not Found"}

curl -G -H "X-API-Key: ..." ".../api/analyses" --data-urlencode "project_name=Implantacao SAP S/4HANA"
→ 200 OK, resultado correto
A causa é estrutural: {"{"}project_name{"}"} é um segmento de path único (conversor padrão do Starlette), que não pode capturar uma barra literal — nenhuma codificação do lado do cliente resolve isso, porque o roteamento acontece depois da decodificação percent-encoding pelo servidor ASGI. O ajuste da Seção 2 do FS-003 (codificar a barra em chamadas curl) evitou o sintoma imediato daquele incidente, mas não corrigia a causa raiz — esta é a primeira vez que o endpoint é exercitado ponta a ponta o suficiente para expor isso.

Por que isso não bloqueou o TIP-004: é exatamente o cenário que a diretriz de painéis independentes foi desenhada para tolerar. O Painel A (contagens/cabeçalho, que usa esse endpoint) degrada para um estado de erro explícito — e os Painéis B e C (que usam GET /api/analyses, com project_name como query param, imune a esse bug) continuam funcionando normalmente. As capturas de tela abaixo mostram exatamente esse comportamento acontecendo ao vivo contra o backend real.

Não corrigido nesta feature, por estar fora do escopo autorizado (nenhuma alteração de backend/API). Candidato real para uma correção futura — ex.: mover project_name para query param nesse endpoint também, espelhando o padrão já usado em /api/analyses.

Bug real corrigido antes do commit: esta versão do Next.js não decodifica automaticamente segmentos de rota dinâmica — tanto page.tsx quanto as 3 rotas BFF estavam recebendo project_name já codificado e o codificando de novo (dupla codificação), quebrando exatamente o caso "Implantacao SAP S/4HANA". Corrigido decodificando uma única vez em cada ponto de entrada; os testes unitários das rotas foram atualizados para simular o parâmetro bruto que o Next.js realmente entrega, fechando a lacuna que permitia esse tipo de bug passar despercebido sem depender de E2E.

Limite honesto da suíte E2E: o mock de backend usado pelo Playwright (e2e/mock-backend.mjs) não reproduz o comportamento real de roteamento do Starlette/FastAPI para barras codificadas — por isso os 64 testes E2E passam mesmo com o achado acima presente no backend real. A verificação manual contra o backend real (demo local) foi o que revelou o problema; os testes automatizados sozinhos não teriam capturado.
